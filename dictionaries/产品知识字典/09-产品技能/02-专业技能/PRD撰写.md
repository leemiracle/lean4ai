# PRD撰写

## 概述

PRD（Product Requirement Document，产品需求文档）是产品经理最重要的工作产出之一，是将需求转化为可执行方案的核心文档。

## PRD价值

### 对产品经理
- 系统梳理需求
- 深入思考细节
- 逻辑验证工具
- 工作成果沉淀

### 对研发团队
- 明确开发内容
- 评估工作量
- 技术方案设计依据
- 开发参考文档

### 对设计团队
- 了解功能需求
- 设计界面依据
- 交互流程参考
- 体验优化方向

### 对测试团队
- 测试用例编写依据
- 验收标准参考
- 功能完整性检查
- Bug修复验证

## PRD核心要素

### 1. 文档基础信息
```markdown
项目名称：[项目名]
文档版本：v1.0
创建日期：2024-01-01
最后更新：2024-01-15
产品经理：[姓名]
相关文档：[链接]
```

### 2. 项目背景
- 业务背景
- 用户痛点
- 市场机会
- 项目目标

### 3. 用户范围
- 目标用户画像
- 使用场景
- 用户价值

### 4. 功能需求
- 功能清单
- 详细功能说明
- 优先级排序

### 5. 非功能需求
- 性能要求
- 安全要求
- 兼容性要求
- 数据要求

### 6. 其他内容
- 数据埋点
- 风险评估
- 时间计划
- 附录材料

## 写作技巧

### 1. 结构清晰
```
✓ 层次分明
✓ 逻辑清晰
✓ 重点突出
✓ 易于查找

✗ 结构混乱
✗ 逻辑跳跃
✗ 重点分散
✗ 难以定位
```

### 2. 表达准确
```markdown
✓ 使用精确的词汇
✓ 避免模糊表述
✓ 定义专业术语
✓ 提供示例说明

示例：
✗ "用户可以查看信息"
✓ "用户可以查看订单的以下信息：订单号、商品名称、价格、状态、下单时间"
```

### 3. 图文并茂
- 流程图：展示业务流程
- 时序图：说明系统交互
- 状态图：描述状态变化
- 界面图：呈现UI设计

### 4. 版本管理
```markdown
版本命名规范：
- v1.0：初始版本
- v1.1：小修改
- v2.0：大改动

变更记录：
- 变更内容
- 变更原因
- 影响范围
- 通知人员
```

## 案例分析

### 案例1：用户注册功能PRD

```markdown
# 用户注册功能 PRD

## 1. 文档信息

| 项目 | 内容 |
|------|------|
| 项目名称 | 用户注册功能 |
| 文档版本 | v1.0 |
| 创建日期 | 2024-01-15 |
| 产品经理 | 张三 |
| 状态 | 已确认 |

## 2. 项目背景

### 2.1 业务背景
当前产品注册转化率仅为30%，低于行业平均水平45%，需要优化注册流程提升转化率。

### 2.2 用户痛点
- 注册流程繁琐，步骤多
- 验证码接收慢
- 密码规则复杂，容易出错
- 第三方登录不稳定

### 2.3 项目目标
- 注册转化率提升至40%以上
- 注册时间缩短至30秒内
- 注册成功率提升至95%

## 3. 功能需求

### 3.1 功能清单

| 功能模块 | 功能点 | 优先级 | 工作量 |
|---------|-------|-------|--------|
| 手机号注册 | 手机号输入 | P0 | 1人日 |
| | 短信验证码 | P0 | 2人日 |
| | 密码设置 | P0 | 1人日 |
| 第三方登录 | 微信登录 | P0 | 2人日 |
| | QQ登录 | P1 | 1人日 |
| | 微博登录 | P2 | 1人日 |
| 注册引导 | 隐私协议 | P0 | 0.5人日 |
| | 注册引导页 | P1 | 1人日 |

### 3.2 功能详细说明

#### 3.2.1 手机号注册

**功能描述**
用户使用手机号完成注册，包括手机号输入、验证码验证、密码设置三个步骤。

**用户场景**
新用户首次使用产品，选择手机号注册，输入手机号后接收验证码，设置密码完成注册。

**业务规则**

1. 手机号验证
   - 支持中国大陆11位手机号
   - 格式：1[3-9]\d{9}
   - 实时格式验证
   - 检查是否已注册

2. 验证码
   - 6位数字
   - 有效期5分钟
   - 单手机号每天最多发送10次
   - 发送间隔60秒

3. 密码规则
   - 8-20位字符
   - 必须包含数字和字母
   - 支持特殊字符
   - 不能与手机号相同

**界面元素**

| 元素名称 | 类型 | 说明 | 验证规则 |
|---------|------|------|---------|
| 手机号输入框 | Input | placeholder:请输入手机号 | 11位数字 |
| 获取验证码 | Button | 点击发送验证码 | 60秒倒计时 |
| 验证码输入框 | Input | placeholder:请输入验证码 | 6位数字 |
| 密码输入框 | Password | placeholder:请设置密码 | 8-20位 |
| 确认密码框 | Password | placeholder:请确认密码 | 与密码一致 |
| 注册按钮 | Button | 提交注册 | 所有字段必填 |

**交互说明**

1. 用户输入手机号
2. 点击"获取验证码"按钮
3. 系统发送短信验证码
4. 用户输入验证码
5. 用户设置密码
6. 勾选隐私协议
7. 点击"注册"按钮
8. 注册成功，跳转首页

**异常处理**

| 异常场景 | 处理方式 |
|---------|---------|
| 手机号格式错误 | 输入框下方提示"请输入正确的手机号" |
| 手机号已注册 | 提示"该手机号已注册，请直接登录" |
| 验证码错误 | 提示"验证码错误，请重新输入" |
| 验证码过期 | 提示"验证码已过期，请重新获取" |
| 密码不符合规则 | 实时提示密码规则要求 |
| 两次密码不一致 | 提示"两次密码输入不一致" |
| 网络错误 | 提示"网络异常，请稍后重试" |

#### 3.2.2 第三方登录

**微信登录**

**功能描述**
用户使用微信账号快速登录，无需填写注册信息。

**接入流程**
1. 用户点击"微信登录"
2. 跳转微信授权页
3. 用户确认授权
4. 获取用户信息
5. 完成登录/注册

**数据获取**
- openid（必须）
- nickname（必须）
- headimgurl（必须）
- unionid（可选）

**注意**
- 首次授权创建新账号
- 已授权用户直接登录
- 需要绑定手机号

## 4. 非功能需求

### 4.1 性能要求
- 短信发送成功率 > 99%
- 验证码到达时间 < 10秒
- 注册接口响应时间 < 500ms

### 4.2 安全要求
- 密码加密存储
- 防刷机制
- 接口限流
- HTTPS传输

### 4.3 兼容性要求
- iOS 11.0+
- Android 6.0+
- 微信版本 7.0+

## 5. 数据需求

### 5.1 埋点事件

| 事件名称 | 触发时机 | 属性 |
|---------|---------|------|
| register_page_view | 进入注册页 | 渠道来源 |
| get_code_click | 点击获取验证码 | 手机号前缀 |
| get_code_success | 验证码发送成功 | 手机号前缀 |
| register_submit | 点击注册按钮 | 注册方式 |
| register_success | 注册成功 | 注册方式、耗时 |

### 5.2 核心指标

| 指标名称 | 计算方式 | 目标值 |
|---------|---------|-------|
| 注册转化率 | 注册成功数/访问注册页数 | 40% |
| 注册成功率 | 注册成功数/提交注册数 | 95% |
| 平均注册时长 | 从进入页面到注册成功的时长 | 30秒 |
| 验证码到达率 | 收到验证码用户数/发送验证码数 | 99% |

## 6. 时间计划

| 里程碑 | 时间 | 负责人 | 交付物 |
|-------|------|-------|--------|
| 需求评审 | 1月16日 | 产品 | 确认PRD |
| UI设计 | 1月18日 | 设计 | UI稿 |
| 开发完成 | 1月25日 | 开发 | 可测试版本 |
| 测试完成 | 1月28日 | 测试 | 测试报告 |
| 上线 | 1月30日 | 全员 | 线上版本 |

## 7. 风险评估

| 风险项 | 概率 | 影响 | 应对措施 |
|-------|------|------|---------|
| 短信通道不稳定 | 中 | 高 | 准备备用通道 |
| 微信接口变更 | 低 | 高 | 关注官方公告 |
| 注册转化率不达标 | 中 | 高 | 准备优化方案 |
```

## 常见问题

### Q1：PRD写到什么程度算详细？
**A：** 
- 开发看完能直接开发
- 测试看完能写测试用例
- 不需要频繁询问细节
- 覆盖正常和异常场景

### Q2：PRD如何管理版本？
**A：**
- 使用文档协作工具
- 明确版本号规范
- 记录变更内容
- 及时通知相关人员

### Q3：如何提高PRD质量？
**A：**
- 多写多练
- 寻求反馈
- 学习优秀案例
- 建立模板库
- 持续迭代优化
