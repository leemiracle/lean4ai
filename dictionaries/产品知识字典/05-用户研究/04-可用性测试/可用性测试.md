# 可用性测试

## 概述
可用性测试(Usability Testing)是通过观察真实用户使用产品,发现可用性问题并改进产品设计的评估方法。它是用户体验研究的核心方法之一。

## 核心价值

### 评估维度
- **有效性(Effectiveness)**: 用户能否完成任务
- **效率(Efficiency)**: 完成任务的时间和步数
- **满意度(Satisfaction)**: 用户的主观感受
- **错误率(Errors)**: 犯错的频率和严重程度
- **易学性(Learnability)**: 学习使用的难易程度

### 何时进行
- 设计初期: 测试原型
- 设计中期: 迭代优化
- 开发完成后: 验证效果
- 产品上线后: 持续改进

## 测试类型

### 1. 按形式分类

#### 实验室测试(Lab Testing)
```
特点:
- 专业环境控制
- 实时观察记录
- 可用性实验室
- 单向镜观察

优点:
- 环境可控
- 深度观察
- 即时追问

缺点:
- 成本较高
- 环境不自然
- 地理限制

适用:
- 复杂产品
- 深度研究
- 客户观摩需求
```

#### 远程测试(Remote Testing)
```
特点:
- 用户在自然环境中
- 通过网络进行
- 屏幕共享录制

优点:
- 地理灵活
- 成本较低
- 环境真实
- 便于招募

缺点:
- 控制力较弱
- 技术依赖
- 观察受限

适用:
- 分布式用户
- 快速迭代
- 预算有限
```

#### 无 moderated 测试(Unmoderated)
```
特点:
- 无主持人
- 用户自主完成
- 自动化工具

优点:
- 效率高
- 成本低
- 样本量大

缺点:
- 无法追问
- 数据深度浅
- 质量控制难

适用:
- 简单任务
- 大规模测试
- 趋势发现
```

### 2. 按时间分类

#### 形成性测试(Formative)
```
目的: 发现问题,指导改进
时机: 设计开发过程中
样本: 5-8人
方法: 定性观察为主
```

#### 总结性测试(Summative)
```
目的: 评估效果,验证目标
时机: 设计完成后
样本: 更多人(10-20+)
方法: 定量指标为主
```

## 测试流程

### 标准流程图
```
计划准备(1-2周)
   ↓
招募用户(1周)
   ↓
测试执行(1-3天)
   ↓
数据分析(1周)
   ↓
报告呈现(2-3天)
```

### 阶段1: 计划准备

#### 制定测试计划
**测试计划模板**
```
【可用性测试计划】

1. 测试目的
   - 要回答什么问题?
   - 验证什么假设?

2. 测试范围
   - 测试哪些功能/页面?
   - 不测试什么?

3. 目标用户
   - 用户画像
   - 招募标准
   - 样本量

4. 测试任务
   - 核心任务列表
   - 任务场景描述
   - 成功标准

5. 测试方法
   - 测试形式
   - 工具设备
   - 流程安排

6. 评估指标
   - 定量指标
   - 定性观察点

7. 时间计划
   - 准备时间
   - 测试时间
   - 分析时间

8. 人员分工
   - 主持人
   - 记录员
   - 观察者
```

#### 设计测试任务

**任务设计原则**
```
✅ 应该:
- 基于真实场景
- 清晰具体
- 难度适中
- 涵盖核心功能

❌ 避免:
- 过于简单
- 引导性太强
- 脱离实际
- 过于复杂
```

**任务示例**
```
❌ 错误示范:
"请点击右上角的'设置'按钮"
→ 引导性太强

✅ 正确示范:
"你最近收到太多通知,想调整通知设置。请找到相关设置并关闭促销类通知。"
→ 真实场景,不提示操作路径
```

**任务结构**
```
1. 情境描述
   "想象你刚买了一双运动鞋..."

2. 任务目标
   "你想查看订单状态和物流信息..."

3. 结束条件
   "找到物流信息后告诉我即可"
```

#### 准备测试材料
- [ ] 原型/产品
- [ ] 任务卡片
- [ ] 前后问卷
- [ ] 知情同意书
- [ ] 录音录像设备
- [ ] 记录表格
- [ ] NDA(如需要)

### 阶段2: 用户招募

#### 招募标准
```
基本维度:
- 使用经验(新手/有经验)
- 人口统计学(年龄、性别、职业)
- 行为特征(频率、场景)
- 态度倾向(喜欢/不喜欢)

特殊要求:
- 竞品用户
- 特定行业
- 特殊需求
```

#### 样本量
```
形成性测试: 5-8人
→ 可发现80%的可用性问题

总结性测试: 10-20人
→ 获得可靠的定量数据

多轮测试:
- 第1轮: 5人 → 修复问题
- 第2轮: 5人 → 验证改进
```

#### 招募渠道
- 现有用户库
- 社交媒体
- 第三方招募公司
- 滚雪球推荐
- 线下拦截

#### 激励标准
```
一般用户: 100-300元/小时
专业用户: 300-800元/小时
学生用户: 50-150元/小时
(参考,因地区和项目而异)
```

### 阶段3: 测试执行

#### 测试环境准备
```
实验室布置:
- 安静的房间
- 舒适的桌椅
- 合适的光线
- 录音录像设备测试
- 网络连接测试
- 备用设备准备
```

#### 测试流程
```
1. 欢迎与介绍(5分钟)
   - 自我介绍
   - 测试说明
   - 签署同意书
   
2. 预测试访谈(5-10分钟)
   - 背景了解
   - 使用习惯
   - 建立关系
   
3. 任务执行(30-60分钟)
   - 朗读任务
   - 用户操作
   - 出声思考
   - 观察记录
   
4. 访谈反馈(10-15分钟)
   - 整体感受
   - 满意度评分
   - 改进建议
   
5. 感谢与结束(5分钟)
   - 道谢
   - 发放报酬
   - 说明后续
```

#### 主持人技巧

**开场话术**
```
"感谢您今天来参加测试。我先说明一下:

1. 我们测试的是产品,不是您
   → 消除紧张,鼓励真实反馈

2. 请真实表达,没有对错
   → 鼓励批评意见

3. 过程中请说出您的想法
   → 引导出声思考

4. 如果找不到某功能,不是您的问题
   → 强调是设计问题

5. 有任何问题随时提出
   → 开放态度

现在,请谈谈您使用类似产品的经历..."
```

**出声思考(Think Aloud)**
```
引导语:
"在操作过程中,请把您的想法说出来,
比如:
- 您在找什么
- 您看到了什么
- 您期待发生什么
- 您的感受是什么"

提示:
- 当用户沉默时提醒
- "您现在在想什么?"
- "您在找什么?"
```

**观察与记录**
```
记录内容:
- 操作路径
- 迟疑停顿
- 错误尝试
- 情绪反应
- 自发评论
- 困惑表情

记录方式:
- 0: 无问题
- 1: 轻微延迟
- 2: 困惑/挫折
- 3: 失败/放弃

示例:
[任务1] 查找订单
- 路径: 首页 → 我的 → 订单(错误)
       → 我的 → 全部订单(正确)
- 问题: "订单"入口不明显
- 严重度: 2
- 用户原话: "应该在个人中心吧...哦在这里"
```

### 阶段4: 数据分析

#### 定量指标

**任务成功率**
```
完成率 = 成功完成任务人数 / 总人数

分级:
- 完全成功
- 部分成功(有困难但完成)
- 失败

目标:
- 核心任务: >90%
- 次要任务: >80%
```

**任务完成时间**
```
记录:
- 开始时间
- 完成时间
- 总耗时

对比:
- 与预期时间对比
- 与竞品对比
- 与上一版本对比

目标:
- 核心任务: <3分钟
- 复杂任务: <5分钟
```

**错误率**
```
错误率 = 出错次数 / 操作总数

错误类型:
- 导航错误: 走错路径
- 输入错误: 填写错误
- 理解错误: 误解含义
- 遗漏错误: 忽略信息
```

**满意度(SUS)**
```
系统可用性量表(System Usability Scale)
10题5点量表
总分0-100

评分标准:
- >80: 优秀
- 68-80: 良好
- 50-68: 一般
- <50: 较差
```

#### 定性分析

**问题分类**
```
问题严重度分级:
4 - 灾难性: 用户无法完成任务
3 - 严重: 显著影响使用
2 - 中等: 造成困扰但可克服
1 - 轻微: 小问题,影响不大
0 - 建议: 非问题,优化建议
```

**问题整理表**
```
| 编号 | 问题描述 | 位置 | 严重度 | 频次 | 建议 |
|------|---------|------|--------|------|------|
| P1   | 找不到订单入口 | 首页 | 3 | 4/5 | 增加明显入口 |
| P2   | 按钮文字不清 | 支付页 | 2 | 3/5 | 修改文案 |
```

**问题优先级**
```
优先级 = 严重度 × 频次

高优先级:
- 严重度高 + 频次高
- 阻碍核心任务

中优先级:
- 严重度中 + 频次高
- 或严重度高 + 频次低

低优先级:
- 严重度低
- 或频次低
```

### 阶段5: 报告呈现

#### 报告结构
```
1. 执行摘要(1页)
   - 测试概况
   - 关键发现
   - 核心建议

2. 研究方法
   - 测试目的
   - 参与者
   - 测试任务
   - 测试环境

3. 主要发现
   - 定量结果
   - 定性问题
   - 用户反馈

4. 详细问题列表
   - 问题描述
   - 严重度
   - 频次
   - 改进建议

5. 建议
   - 优先级排序
   - 具体方案
   - 预期效果

6. 附录
   - 测试任务
   - 问卷结果
   - 视频片段
```

## 特殊测试方法

### 1. 启发式评估(Heuristic Evaluation)
```
定义: 专家根据可用性原则评估产品

Nielsen十大启发式原则:
1. 系统状态可见性
2. 环境与现实匹配
3. 用户控制与自由
4. 一致性与标准
5. 错误预防
6. 识别而非回忆
7. 灵活性和效率
8. 美观与极简设计
9. 帮助用户识别和恢复错误
10. 帮助和文档

优点: 快速,成本低
缺点: 可能遗漏真实用户问题
建议: 与用户测试结合使用
```

### 2. 认知走查(Cognitive Walkthrough)
```
定义: 模拟用户使用产品的认知过程

步骤:
1. 定义用户和任务
2. 逐步走查操作流程
3. 每步评估:
   - 用户知道做什么吗?
   - 能找到正确控件吗?
   - 能识别操作成功吗?
4. 记录问题
```

### 3. A/B测试
```
定义: 对比两个版本的效果差异

详见: A/B测试.md
```

### 4. 树测试(Tree Testing)
```
定义: 测试信息架构的可找性

工具:
- Treejack
- UserZoom Tree Testing

用途:
- 导航结构优化
- 分类方案验证
```

### 5. 卡片分类(Card Sorting)
```
定义: 让用户组织信息结构

类型:
- 开放式: 用户自建分类
- 封闭式: 按给定分类
- 混合式

工具:
- Optimal Workshop
- UserZoom
- 纸质卡片(线下)
```

## 最佳实践

### ✅ 应该做
- 早期测试,频繁测试
- 5人即可发现80%问题
- 真实任务,真实场景
- 鼓励出声思考
- 记录详细
- 迭代改进

### ❌ 避免做
- 等产品完成再测试
- 样本量追求过大
- 引导用户操作
- 忽视小问题
- 只测一次
- 只收集定量数据

## 常见问题

**Q: 需要多少人?**
A: 形成性测试5-8人,总结性测试10-20人。

**Q: 测试原型还是真实产品?**
A: 早期用原型,后期用真实产品。原型级别根据测试目的选择。

**Q: 如何让用户真实反馈?**
A: 强调测试产品而非用户,鼓励批评,不评判。

**Q: 远程还是线下?**
A: 根据预算、时间和用户分布选择。复杂产品推荐线下。

## 案例分析

### 案例: 电商App购物流程优化

**背景**
某电商平台购物车到支付流程转化率偏低

**测试目的**
发现购物流程中的可用性问题,提升转化率

**测试设计**
```
参与者: 8人(4新手+4有经验用户)
形式: 实验室测试
任务:
1. 添加商品到购物车
2. 修改商品数量
3. 使用优惠券
4. 完成支付

方法: 出声思考+观察记录
```

**关键发现**
```
问题P1 (严重度3, 频次6/8):
- 优惠券输入位置不明显
- 用户原话: "优惠券在哪输入?找不到啊"
- 建议: 在结算页突出优惠券入口

问题P2 (严重度2, 频次5/8):
- 修改数量需进入商品详情
- 用户原话: "为什么要跳转才能改数量?"
- 建议: 购物车页直接修改

问题P3 (严重度2, 频次4/8):
- 支付方式选择不明确
- 建议: 默认勾选常用方式,突出显示
```

**改进效果**
- 购物车到支付转化率 +15%
- 支付完成率 +8%
- 用户满意度 3.2→4.1
