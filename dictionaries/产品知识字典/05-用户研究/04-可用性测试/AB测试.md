# A/B测试

## 概述
A/B测试(又称对照实验或桶测试)是将用户随机分配到不同版本,对比关键指标差异,以数据驱动产品决策的科学方法。

## 核心概念

### 基本原理
```
总体用户
   ↓ 随机分配
实验组(B) ←→ 对照组(A)
   ↓              ↓
新版本          原版本
   ↓              ↓
指标测量        指标测量
   ↓              ↓
   对比分析 → 统计检验
         ↓
      结论判断
```

### 核心要素
- **对照组(Control)**: 原版本/基准版本
- **实验组(Treatment)**: 新版本/测试版本
- **随机分配**: 确保组间可比
- **指标(Metric)**: 评估标准
- **统计显著性**: 确保差异非偶然

## 测试类型

### 1. 经典A/B测试
```
对照组A ──→ 原版本
实验组B ──→ 新版本

特点:
- 两组对比
- 单一变量
- 最常用
```

### 2. A/B/n测试
```
对照组A ──→ 原版本
实验组B ──→ 方案1
实验组C ──→ 方案2
实验组D ──→ 方案3

特点:
- 多个方案对比
- 寻找最优方案
- 需要更大样本
```

### 3. 多变量测试(MVT)
```
同时测试多个变量的组合

示例:
按钮颜色: 红色/蓝色
按钮文字: "立即购买"/"马上抢购"
按钮大小: 大/小

2×2×2 = 8种组合

特点:
- 测试多元素组合
- 发现交互效应
- 需要大量样本
```

### 4. 分层测试
```
按用户特征分层后再随机

示例:
      新用户      老用户
     /    \      /    \
    A     B     A     B

特点:
- 保证各组均衡
- 可分析不同人群
- 提高统计功效
```

### 1. 假设提出

#### 优秀假设的标准
```
格式:
如果 [改变], 那么 [指标] 会 [变化], 因为 [原因]

示例:
如果 [增大购买按钮], 那么 [点击率] 会 [提升], 
因为 [按钮更醒目,用户更容易发现]
```

#### 假设来源
```
- 用户反馈
- 数据分析洞察
- 竞品对标
- 设计直觉
- 行业最佳实践
- 前期定性研究
```

### 2. 指标定义

#### 指标类型

**核心指标(OEC - Overall Evaluation Criterion)**
```
定义: 决定实验成败的关键指标

特点:
- 与业务目标强相关
- 可量化测量
- 对变化敏感
- 不易被游戏化

示例:
- 电商: 转化率、客单价、GMV
- 内容: 停留时长、阅读深度、互动率
- 社交: 日活、使用时长、内容发布率
```

**过程指标**
```
定义: 反映中间过程的指标

示例:
- 点击率
- 页面停留时间
- 跳出率
- 任务完成率
```

**护栏指标(Guardrail Metrics)**
```
定义: 确保不产生负面影响的指标

示例:
- 加载时间
- 错误率
- 客服投诉量
- 卸载率
```

#### 指标选择原则
```
✅ 应该:
- 选择1-2个核心指标
- 配合过程指标
- 关注护栏指标
- 可操作性

❌ 避免:
- 指标过多
- 只看短期指标
- 忽视负面影响
```

### 3. 实验设计

#### 样本量计算

**基本公式**
```
n = 2 × [(Z_α/2 + Z_β)² × σ²] / δ²

其中:
n = 每组样本量
Z_α/2 = 显著性水平(α=0.05时为1.96)
Z_β = 统计功效(β=0.2时为0.84)
σ = 标准差
δ = 预期最小检测差异(MDE)

简化公式(比例):
n = 2 × [(Z_α/2 + Z_β)² × p(1-p)] / δ²
```

**在线计算器**
- Optimizely Sample Size Calculator
- VWO SmartStats
- Evan Miller's Calculator

**参数设置**
```
显著性水平(α): 通常0.05
→ 5%概率将偶然差异判定为真实差异

统计功效(1-β): 通常0.8
→ 80%概率检测到真实差异

MDE(最小检测效应):
- 根据业务价值设定
- 过小→样本量太大
- 过大→可能错过效果

示例:
当前转化率: 5%
期望提升: 10% → 新转化率5.5%
MDE = 0.5% (绝对) 或 10% (相对)
```

**实际样本量参考**
```
假设:
- 基准转化率: 5%
- MDE: 相对提升10% → 5.5%
- α = 0.05, 功效 = 0.8

每组样本量 ≈ 30,000
总样本量 ≈ 60,000
```

#### 流量分配

**分配比例**
```
常见比例:
- 50/50: 标准A/B测试
- 80/20: 小流量测试风险方案
- 90/10: 谨慎测试重大变更

考虑因素:
- 风险程度
- 流量规模
- 测试时长
- 统计功效
```

**随机化方法**
```
方法:
- 用户ID哈希
- 设备ID哈希
- Cookie分配
- 账号分配

要求:
- 一致性: 同一用户始终在同一组
- 随机性: 分配均匀
- 可复现: 可重新生成
```

### 4. 技术实现

#### 实验架构
```
用户请求
   ↓
分流服务
   ↓
┌──────┬──────┐
│  A组  │  B组  │
└──────┴──────┘
   ↓      ↓
版本A  版本B
   ↓      ↓
日志记录(带实验标记)
   ↓
数据分析
```

#### 关键技术点

**分流一致性**
```
要求:
- 同一用户多次访问,始终看到同一版本
- 跨平台用户保持一致

实现:
hash(user_id + experiment_id) % 100 < traffic%
```

**埋点与日志**
```
必要字段:
- user_id: 用户标识
- experiment_id: 实验ID
- variation_id: 分组ID
- timestamp: 时间戳
- event_type: 事件类型
- event_value: 事件值
```

**灰度发布**
```
阶段1: 1% 流量测试
  ↓ 观察
阶段2: 5% 流量测试
  ↓ 观察
阶段3: 20% 流量测试
  ↓ 观察
阶段4: 50/50 完整测试
  ↓
全量发布
```

### 5. 测试执行

#### 执行清单
```
□ 分流逻辑正确
□ 埋点数据完整
□ AA测试通过
□ 护栏指标正常
□ 技术性能稳定
□ 用户体验无异常
```

#### AA测试
```
目的:
- 验证分流正确性
- 确保两组基准一致
- 检测系统偏差

方法:
- 对照组 vs 对照组
- 预期差异不显著
- 如显著→系统有问题

时长: 1-3天
```

#### 测试时长

**最短时长**
```
需要达到:
- 样本量充足
- 覆盖完整周期(至少7天)
- 业务周期考量(工作日/周末)

避免:
- 周期效应
- 新奇效应
- 节假日影响
```

**最长时长**
```
考虑:
- 时间成本
- 竞争压力
- 用户疲劳
- 多重检验

建议: 2-4周
```

### 6. 数据分析

#### 描述统计
```
基础指标:
- 样本量
- 均值/比例
- 标准差
- 置信区间

对比:
- 绝对差异
- 相对提升
```

#### 统计检验

**比例差异检验**
```
两比例z检验:

z = (p_B - p_A) / √[p(1-p)(1/n_A + 1/n_B)]

其中:
p = (x_A + x_B) / (n_A + n_B)

显著性判断:
p-value < 0.05 → 显著
```

**均值差异检验**
```
两样本t检验:

t = (x̄_B - x̄_A) / √(s_A²/n_A + s_B²/n_B)
```

**置信区间**
```
95% CI:
差异 ± 1.96 × SE

解读:
- 不包含0 → 显著
- 包含0 → 不显著
- 区间宽度反映精度
```

#### 多重检验校正
```
问题:
多次检验增加假阳性风险

方法:
- Bonferroni: α/m
- Holm-Bonferroni: 逐步校正
- FDR(错误发现率)
```

#### 分层分析
```
维度:
- 新老用户
- 设备类型
- 地域
- 时间段

目的:
- 发现不同群体差异
- 优化定向策略

注意:
- 避免过度分片
- 多重检验校正
```

### 7. 决策落地

#### 决策框架
```
统计显著 + 业务显著 → 全量发布
统计显著 + 业务不显著 → 视情况
统计不显著 + 业务显著 → 延长测试/优化
统计不显著 + 业务不显著 → 放弃
```

#### 效果评估
```
绝对提升:
Δ = 指标_B - 指标_A

相对提升:
Δ% = (指标_B - 指标_A) / 指标_A × 100%

业务价值:
价值 = 用户数 × 提升比例 × 单位价值
```

#### 发布策略
```
方式1: 直接全量
- 效果明确显著
- 风险较低

方式2: 渐进发布
- 逐步扩大流量
- 持续监控指标

方式3: 定向发布
- 仅对特定用户发布
- 如仅对新用户
```

## 常见问题与陷阱

### 1. 新奇效应
```
现象: 用户因新鲜感而行为改变
解决: 延长测试时间,观察衰减

示例:
新UI上线初期点击率高,两周后回归正常
```

### 2. 辛普森悖论
```
现象: 分组结果与总体结果相反

示例:
         A组     B组
新用户   5%      4%
老用户   10%     8%
总体     8%      9%  ← 矛盾!

原因: 用户构成不同
解决: 分层分析,分层抽样
```

### 3. 过早停止
```
问题:
- 样本量不足就下结论
- p值波动导致错误判断

解决:
- 预先计算样本量
- 达到样本量再判断
- 使用序贯检验方法
```

### 4. 选择性报告
```
问题:
- 只报告显著指标
- 忽略负面影响
- 多次检验不校正

解决:
- 预先声明核心指标
- 全面报告所有指标
- 多重检验校正
```

### 5. 污染效应
```
问题:
- 用户跨组
- 信息传播
- 网络效应

示例:
社交产品,看到新功能的用户可能告诉对照组

解决:
- 按社交圈分组
- 延长测试观察
- 谨慎解读结果
```

### 6. 过度优化
```
问题:
- 短期指标提升
- 长期价值损害

示例:
弹窗提高点击率,但损害用户体验

解决:
- 平衡短期长期指标
- 关注护栏指标
```

## 最佳实践

### ✅ 应该做
- 基于假设和数据分析
- 预先计算样本量
- 明确定义核心指标
- AA测试验证
- 等样本量充足再判断
- 全面报告结果
- 知识沉淀

### ❌ 避免做
- 为了测试而测试
- 过早停止
- 选择性报告
- 忽视长期影响
- 过度分片分析
- 不做AA测试

## 案例分析

### 案例1: 按钮文案优化

**背景**
电商详情页购买按钮优化

**假设**
"立即购买"改为"马上抢购"可提升点击率

**设计**
```
A组(对照): "立即购买"
B组(实验): "马上抢购"

样本量: 每组50,000
流量分配: 50/50
时长: 14天
核心指标: 按钮点击率、转化率
```

**结果**
```
        A组    B组    提升    p值
曝光量  50,000 50,000
点击率  12.5%  13.8%  +10.4%  0.002**
转化率  3.2%   3.4%   +6.3%   0.08

结论:
点击率显著提升(p<0.01)
转化率提升但不显著(p>0.05)
建议延长测试或全量发布
```

### 案例2: 首页改版

**背景**
首页信息流布局改版

**假设**
卡片式布局提升用户停留时长

**设计**
```
A组: 列表式
B组: 卡片式

核心指标: 停留时长
护栏指标: 加载时间、跳出率
```

**结果**
```
           A组    B组    变化    显著性
停留时长   180s   195s   +8.3%   p<0.01
跳出率     45%    42%    -6.7%   p<0.05
加载时间   1.2s   1.8s   +50%    -

问题:
虽然停留时长提升,但加载时间显著增加

决策:
优化卡片加载性能后再上线
```
