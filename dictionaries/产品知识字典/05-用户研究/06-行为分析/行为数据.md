# 行为数据

## 概述
行为数据(Behavioral Data)是用户在产品使用过程中产生的客观行为记录,通过收集、分析和解读这些数据,可以深入了解用户真实的使用模式和偏好。

## 核心价值

### 相比自我报告的优势
```
行为数据:
- 客观真实
- 无记忆偏差
- 大规模数据
- 实时更新
- 细粒度细节

自我报告:
- 主观回忆
- 社会期许偏差
- 样本有限
- 滞后性
```

### 应用场景
```
产品优化:
- 发现使用问题
- 验证设计方案
- 评估功能效果

用户洞察:
- 理解使用模式
- 识别用户分群
- 预测用户行为

业务决策:
- 转化率优化
- 留存分析
- 增长策略
```

## 数据类型

### 1. 按层级分类

#### 页面级数据
```
内容:
- PV(Page View): 页面浏览量
- UV(Unique Visitor): 独立访客数
- 访问时长
- 跳出率
- 入口/出口页面

示例:
商品详情页:
- PV: 10,000
- UV: 8,000
- 平均停留: 45秒
- 跳出率: 35%
```

#### 事件级数据
```
内容:
- 点击事件
- 曝光事件
- 自定义事件
- 错误事件

示例:
[点击事件]
event: button_click
element: "add_to_cart"
timestamp: 2024-01-15 10:23:45
user_id: user123
page: /product/12345
device: mobile
```

#### 会话级数据
```
内容:
- 会话时长
- 页面路径
- 事件序列
- 会话内转化

示例:
会话#session_456:
- 开始: 10:00:00
- 结束: 10:15:30
- 时长: 15分30秒
- 页面: A → B → C → D
- 事件: 32个
- 转化: 下单成功
```

## 数据收集

### 1. 埋点方式

#### 代码埋点
```
定义: 在代码中手动添加追踪代码

优点:
- 精确控制
- 信息丰富
- 灵活定制

缺点:
- 开发成本高
- 需要发版
- 维护成本

示例:
// 点击事件埋点
track('button_click', {
  button_name: 'add_to_cart',
  product_id: '12345',
  price: 99.00,
  category: 'electronics'
});
```

#### 可视化埋点
```
定义: 通过可视化界面配置埋点

优点:
- 无需开发
- 快速配置
- 降低门槛

缺点:
- 灵活性有限
- 信息量受限

工具:
- GrowingIO
- Sensors Data
- Mixpanel
```

#### 全埋点/无埋点
```
定义: 自动采集所有交互事件

优点:
- 无需开发
- 数据全面
- 快速部署

缺点:
- 数据噪音多
- 存储成本高
- 缺少业务上下文

工具:
- Heap
- Amplitude
```

#### 混合埋点
```
定义: 结合多种方式

策略:
- 全埋点: 自动采集基础交互
- 代码埋点: 关键业务事件
- 可视化埋点: 快速添加新埋点

推荐做法 ⭐
```

### 2. 数据字段设计

#### 标准字段
```
用户维度:
- user_id: 用户唯一标识
- device_id: 设备标识
- session_id: 会话标识

时间维度:
- timestamp: 事件时间戳
- date: 日期
- hour: 小时

设备维度:
- platform: 平台(iOS/Android/Web)
- device_type: 设备类型
- os_version: 系统版本
- app_version: 应用版本

地理维度:
- country: 国家
- province: 省份
- city: 城市

来源维度:
- utm_source: 来源
- utm_medium: 媒介
- utm_campaign: 活动
```

#### 事件字段
```
基础:
- event_name: 事件名称
- event_type: 事件类型

业务:
- 根据业务需求定义
- 商品ID、订单ID、金额...

示例:
[下单事件]
{
  "event": "order_created",
  "timestamp": "2024-01-15 10:23:45",
  "user_id": "user123",
  "order_id": "order456",
  "product_ids": ["p1", "p2"],
  "total_amount": 299.00,
  "payment_method": "wechat_pay"
}
```

### 3. 埋点规范

#### 命名规范
```
事件命名:
- 格式: [对象]_[动作]
- 示例: button_click, page_view, order_created

属性命名:
- 小写+下划线
- 简洁明确
- 示例: product_id, order_amount

建议:
- 建立埋点字典
- 统一命名约定
- 文档化管理
```

#### 埋点管理
```
埋点文档:
| 事件名称 | 触发时机 | 属性 | 说明 |
|---------|---------|------|------|
| page_view | 页面加载完成 | page_name, page_url | 页面浏览 |
| button_click | 按钮被点击 | button_name, page | 按钮点击 |

埋点流程:
1. 需求提出
2. 方案评审
3. 埋点开发
4. 测试验证
5. 上线监控
```

## 数据分析

### 1. 描述性分析

#### 趋势分析
```
目的: 了解指标随时间变化

维度:
- 日/周/月趋势
- 同比: 与去年同期对比
- 环比: 与上一周期对比

可视化:
- 折线图
- 面积图
```

#### 分布分析
```
目的: 了解数据分布特征

内容:
- 频次分布
- 概率分布
- 分位数

可视化:
- 直方图
- 箱线图
```

#### 构成分析
```
目的: 了解各部分占比

内容:
- 维度分解
- 占比分析

可视化:
- 饼图
- 堆叠柱状图
```

### 2. 诊断性分析

#### 漏斗分析
```
目的: 分析转化流程

步骤:
1. 定义转化流程
2. 统计各步骤人数
3. 计算转化率
4. 识别流失点

示例:
浏览 → 加购 → 下单 → 支付
1000   300    150    120
       30%    50%    80%

发现: 加购到下单流失率高
```

#### 路径分析
```
目的: 分析用户行为路径

方法:
- 桑基图(Sankey Diagram)
- 路径树
- 流向分析

发现:
- 常见路径
- 意外路径
- 流失路径
```

#### 归因分析
```
目的: 分析转化贡献来源

模型:
- 首次触点归因
- 最后触点归因
- 线性归因
- 时间衰减归因
- 数据驱动归因

应用:
- 渠道效果评估
- 营销ROI计算
```

### 3. 预测性分析

#### 行为预测
```
目的: 预测用户未来行为

方法:
- 回归分析
- 分类模型
- 时间序列

示例:
- 预测流失概率
- 预测购买概率
- 预测LTV
```

#### 推荐算法
```
目的: 个性化推荐

方法:
- 协同过滤
- 内容推荐
- 混合推荐

应用:
- 商品推荐
- 内容推荐
- 广告推荐
```


## 案例分析

### 案例: 电商转化漏斗优化

**背景**
某电商平台希望提升转化率

**数据收集**
```
埋点事件:
- page_view: 页面浏览
- product_view: 商品浏览
- add_to_cart: 加入购物车
- order_create: 创建订单
- order_pay: 支付订单
```

**漏斗分析**
```
首页浏览: 100,000
  ↓ 80%
商品详情: 80,000
  ↓ 30%
加入购物车: 24,000
  ↓ 60%
创建订单: 14,400
  ↓ 70%
支付成功: 10,080

整体转化率: 10.08%
```

**问题识别**
```
1. 首页→商品详情: 80%
   → 相对正常

2. 商品详情→加购: 30%
   → 低于预期(基准40%)
   → 可能问题: 价格、信息不全

3. 加购→下单: 60%
   → 流失严重
   → 可能问题: 结算流程复杂

4. 下单→支付: 70%
   → 有提升空间
   → 可能问题: 支付方式、失败处理
```

**优化措施与效果**
```
优化1: 商品页增加限时优惠
  → 加购率: 30% → 38%

优化2: 简化结算流程(3步→1步)
  → 下单率: 60% → 75%

优化3: 增加支付失败自动重试
  → 支付率: 70% → 82%

整体效果:
  → 转化率: 10.08% → 17.1%
  → 提升: 70%
```
