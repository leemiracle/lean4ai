## 📋 结构说明

用户故事是敏捷开发中描述需求的轻量级方法，本模板包含：
- **用户故事卡片**：标准格式的故事描述
- **验收标准**：明确完成定义（DoD）
- **任务分解**：技术实现任务
- **故事地图**：用户故事的组织结构

---

## 📝 填写指南

### 1. 用户故事格式
- 使用标准格式：作为[角色]，我想要[功能]，以便[价值]
- 角色：明确是哪类用户
- 功能：描述用户想要什么
- 价值：说明为什么需要

### 2. 验收标准
- 使用Given-When-Then格式
- 涵盖正常流程和异常流程
- 标准要具体、可测试

### 3. 故事拆分
- 遵循INVEST原则
- 大故事要拆分成可在一个Sprint完成的小故事

### 4. 优先级
- 基于业务价值和技术依赖
- 与干系人共同确定

### 四、INVEST原则检查

| 原则 | 检查项 | 是否符合 |
|------|--------|---------|
| **I**ndependent（独立的）| 故事之间是否相互独立？ | ✅/❌ |
| **N**egotiable（可协商的）| 实现细节是否可协商？ | ✅/❌ |
| **V**aluable（有价值的）| 是否对用户有明确价值？ | ✅/❌ |
| **E**stimable（可估算的）| 是否可以估算工作量？ | ✅/❌ |
| **S**mall（小的）| 是否可以在一个Sprint完成？ | ✅/❌ |
| **T**estable（可测试的）| 是否有明确的验收标准？ | ✅/❌ |

### 示例1：用户注册功能

#### 故事编号：US-001

---

#### 1. 故事描述

**作为** 一个新用户，  
**我想要** 使用手机号注册账号，  
**以便** 我可以使用系统的完整功能。

---

#### 2. 故事详情

| 属性 | 内容 |
|------|------|
| 故事ID | US-001 |
| Epic | 用户管理 |
| Feature | 用户注册登录 |
| 优先级 | ☑ P0 |
| 故事点 | 5 |
| 迭代 | Sprint 1 |
| 负责人 | 张三 |
| 状态 | ☑ 完成 |

---

#### 3. 验收标准

##### 正常场景

**AC-1：手机号注册成功**

- **Given**：用户在注册页面，手机号未被注册
- **When**：用户输入有效手机号13800138000，获取并输入验证码123456，设置密码Abc123!@
- **Then**：注册成功，跳转到首页，显示欢迎消息

**AC-2：邮箱注册成功**

- **Given**：用户选择邮箱注册，邮箱未被注册
- **When**：用户输入有效邮箱test@example.com，获取验证码，设置密码
- **Then**：注册成功，跳转到首页

##### 异常场景

**AC-3：手机号已被注册**

- **Given**：用户在注册页面
- **When**：用户输入已被注册的手机号
- **Then**：显示错误提示"该手机号已被注册，请直接登录"

**AC-4：验证码错误**

- **Given**：用户在注册页面，已获取验证码
- **When**：用户输入错误的验证码
- **Then**：显示错误提示"验证码错误，请重新输入"

**AC-5：密码格式不正确**

- **Given**：用户在注册页面
- **When**：用户输入密码"123456"（不符合密码规则）
- **Then**：显示错误提示"密码需包含大小写字母、数字和特殊字符，长度8-20位"

##### 边界场景

**AC-6：验证码过期**

- **Given**：用户获取验证码后等待超过5分钟
- **When**：用户输入验证码
- **Then**：显示错误提示"验证码已过期，请重新获取"

---

#### 4. 业务规则

| 规则编号 | 规则描述 |
|---------|---------|
| BR-1 | 手机号必须是11位有效中国大陆手机号 |
| BR-2 | 密码长度8-20位，必须包含大小写字母、数字和特殊字符 |
| BR-3 | 验证码6位数字，5分钟内有效 |
| BR-4 | 同一手机号60秒内只能获取一次验证码 |
| BR-5 | 同一IP每天最多获取10次验证码 |

### 示例2：商品搜索功能

#### 故事编号：US-015

---

#### 1. 故事描述

**作为** 一个购物用户，  
**我想要** 搜索商品并按条件筛选，  
**以便** 快速找到我想要购买的商品。

---

#### 2. 故事详情

| 属性 | 内容 |
|------|------|
| 故事ID | US-015 |
| Epic | 商品搜索 |
| Feature | 搜索与筛选 |
| 优先级 | ☑ P1 |
| 故事点 | 8 |
| 迭代 | Sprint 3 |

---

#### 3. 验收标准

##### 正常场景

**AC-1：关键词搜索**

- **Given**：用户在搜索页面
- **When**：用户输入"iPhone"并点击搜索
- **Then**：显示包含"iPhone"关键词的商品列表，按相关度排序

**AC-2：分类筛选**

- **Given**：用户已搜索"手机"
- **When**：用户选择分类"智能手机"
- **Then**：显示智能手机分类下的商品

**AC-3：价格区间筛选**

- **Given**：用户在搜索结果页
- **When**：用户设置价格区间1000-3000元
- **Then**：只显示该价格区间的商品

##### 异常场景

**AC-4：无搜索结果**

- **Given**：用户在搜索页面
- **When**：用户输入"abcdefg"搜索
- **Then**：显示"未找到相关商品"提示和推荐商品

---

#### 4. 业务规则

| 规则编号 | 规则描述 |
|---------|---------|
| BR-1 | 搜索支持模糊匹配 |
| BR-2 | 默认按相关度排序，可切换为价格、销量排序 |
| BR-3 | 每页显示20个商品 |
| BR-4 | 筛选条件可组合使用 |

## ⚠️ 最佳实践

### 1. 故事编写原则
- 从用户视角出发，而非技术视角
- 一个故事只描述一个功能点
- 避免使用技术术语
- 价值导向，而非功能导向

### 2. 验收标准编写
- 使用具体数据而非模糊描述
- 覆盖所有场景（正常、异常、边界）
- 可测试、可验证
- 团队共同讨论确定

### 3. 故事拆分技巧
- 按业务流程拆分
- 按操作类型拆分（增删改查）
- 按数据复杂度拆分
- 按界面元素拆分

### 4. 估算要点
- 使用斐波那契数列（1,2,3,5,8,13,21）
- 团队共同估算
- 参考历史数据
- 考虑复杂度、不确定性、工作量
