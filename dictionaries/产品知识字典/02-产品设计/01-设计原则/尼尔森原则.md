# 尼尔森十大可用性原则

## 定义

尼尔森十大可用性原则（Nielsen's 10 Usability Heuristics）由可用性大师 Jakob Nielsen 于1994年提出，是用户界面设计中最具影响力的可用性评估准则。

## 十大原则详解

### 1. 系统状态可见性 (Visibility of System Status)

**定义**
系统应该通过适当的方式，在合理的时间内，让用户了解系统当前的状态。

**要点**
- 及时反馈用户操作
- 清晰显示当前状态
- 使用进度指示器

**应用示例**
```
✓ 上传文件时显示进度条
✓ 按钮点击后显示加载状态
✓ 表单填写时显示已完成步骤
✗ 点击提交后无任何反馈
```

**实现方式**
| 状态类型 | 展示方式 |
|----------|----------|
| 加载中 | 进度条、骨架屏、转圈 |
| 成功 | 绿色勾、成功提示 |
| 失败 | 红色提示、错误信息 |
| 进行中 | 步骤指示器、时间预估 |

### 2. 系统与现实世界的匹配 (Match Between System and Real World)

**定义**
系统应使用用户熟悉的语言、词汇和概念，而不是技术术语。

**要点**
- 使用用户语言
- 遵循现实世界的惯例
- 信息呈现自然有逻辑

**应用示例**
```
✓ "购物车"而非"临时存储区"
✓ "删除"而非"移除记录"
✓ "我的账户"而非"用户中心"
✗ 使用技术术语："404错误" → "页面未找到"
```

**设计对照**
| 技术语言 | 用户语言 |
|----------|----------|
| HTTP 404 | 页面未找到 |
| 网络异常 | 网络连接失败，请检查网络 |
| 数据库错误 | 系统繁忙，请稍后重试 |

### 3. 用户控制与自由 (User Control and Freedom)

**定义**
用户经常会误操作，需要一个明确的"紧急出口"来离开不需要的状态。

**要点**
- 支持撤销和重做
- 提供退出选项
- 标记明确的功能入口

**应用示例**
```
✓ 邮件删除后可恢复
✓ 表单填写支持保存草稿
✓ 支付前二次确认
✗ 删除操作不可逆且无确认
```

**控制机制**
- **撤销/重做**：支持 Ctrl+Z/Ctrl+Y
- **返回**：明确的返回按钮
- **取消**：操作可取消
- **草稿**：自动保存

### 4. 一致性与标准化 (Consistency and Standards)

**定义**
用户不应猜测不同词语、情境或动作是否意味着同一件事。

**要点**
- 平台一致性
- 内部一致性
- 行业标准遵循

**应用示例**
```
✓ 所有"删除"按钮使用相同的图标和颜色
✓ 返回按钮始终位于左上角
✗ 不同页面的"确认"按钮位置不同
```

**一致性层次**
| 层级 | 说明 |
|------|------|
| 平台一致性 | 遵循 iOS/Android 设计规范 |
| 产品一致性 | 产品内各模块统一 |
| 行业一致性 | 符合行业通用惯例 |

### 5. 错误预防 (Error Prevention)

**定义**
比好的错误消息更好的是精心设计以防止错误发生。

**要点**
- 预见可能的错误
- 提供约束和确认
- 减少错误发生机会

**应用示例**
```
✓ 输入框格式限制（手机号只允许数字）
✓ 删除操作二次确认
✓ 自动检测并提示可能的错误
✗ 允许输入无效格式后再报错
```

**预防策略**
- **输入约束**：限制输入格式
- **确认对话**：危险操作二次确认
- **智能提示**：实时校验和提示
- **默认值**：提供合理的默认选项

### 6. 识别而非回忆 (Recognition Rather Than Recall)

**定义**
通过使对象、动作和选项可见，最小化用户的记忆负担。

**要点**
- 信息可见
- 不要求记忆
- 提供必要的提示

**应用示例**
```
✓ 下拉菜单显示选项而非要求输入
✓ 搜索历史记录展示
✓ 最近使用的功能高亮
✗ 要求用户记住复杂密码而无提示
```

**应用技巧**
- 可见选项 > 记忆输入
- 图标配合文字
- 最近使用/常用功能突出
- 上下文提示

### 7. 使用的灵活性与效率 (Flexibility and Efficiency of Use)

**定义**
系统应满足有经验和无经验的用户，提供加速器提高效率。

**要点**
- 快捷操作支持
- 可定制化
- 新手和专家模式

**应用示例**
```
✓ 支持键盘快捷键
✓ 专业模式/简单模式切换
✓ 常用操作快速入口
✗ 所有用户只能通过繁琐的菜单操作
```

**效率提升方式**
| 方式 | 示例 |
|------|------|
| 快捷键 | Ctrl+S 保存 |
| 批量操作 | 批量删除、批量编辑 |
| 模板 | 预设模板快速创建 |
| 自动化 | 自动填充、智能推荐 |

### 8. 美观与极简的设计 (Aesthetic and Minimalist Design)

**定义**
对话不应包含无关或极少需要的信息。

**要点**
- 每个信息都有价值
- 避免视觉混乱
- 简洁但功能完整

**应用示例**
```
✓ Google 首页简洁但功能完整
✓ 表单只包含必要字段
✗ 页面堆砌大量无关信息
```

**极简原则**
- 删除不必要的元素
- 信息层级清晰
- 视觉焦点突出
- 留白的合理使用

### 9. 帮助用户识别、诊断并从错误中恢复 (Help Users Recognize, Diagnose, and Recover from Errors)

**定义**
错误消息应以简明的语言表达，精确指出问题并建设性地提出解决方案。

**要点**
- 明确指出错误
- 解释错误原因
- 提供解决方法

**应用示例**
```
✓ "密码太短，至少需要8个字符"
✓ "用户名已存在，请尝试其他用户名或登录"
✗ "错误代码: 0x80042"
```

**错误提示结构**
```
问题说明 → 原因分析 → 解决方案
"无法连接网络 → 请检查网络设置 → 点击重试"
```

### 10. 帮助与文档 (Help and Documentation)

**定义**
系统应提供帮助文档和用户指南。

**要点**
- 易于搜索
- 聚焦用户任务
- 提供具体操作步骤

**应用示例**
```
✓ 上下文相关的帮助提示
✓ 可搜索的FAQ
✓ 图文并茂的操作指南
✗ 难以找到帮助入口
```

**帮助类型**
| 类型 | 说明 |
|------|------|
| 入门引导 | 新手教程、引导页 |
| 上下文帮助 | 点击问号显示说明 |
| 帮助中心 | 文档、FAQ |
| 客服支持 | 在线客服、反馈 |

## 评估方法

### 启发式评估流程

1. **准备阶段**
   - 确定评估范围
   - 选择评估者（3-5人为宜）
   - 准备评估材料

2. **评估阶段**
   - 独立评估
   - 记录问题
   - 标注严重程度

3. **分析阶段**
   - 汇总问题
   - 优先级排序
   - 提出改进建议

### 问题严重程度分级

| 级别 | 说明 | 示例 |
|------|------|------|
| 0 | 不影响可用性 | 微小的视觉问题 |
| 1 | 表面问题 | 文案不清晰 |
| 2 | 主要问题 | 关键功能难用 |
| 3 | 灾难性问题 | 无法完成核心任务 |
| 4 | 不可用 | 系统崩溃 |

## 案例

### 案例1：电商购物流程优化

**问题发现**
- 用户提交订单后无状态反馈（违反原则1）
- 支付失败后无明确错误说明（违反原则9）

**改进方案**
- 添加订单提交进度提示
- 优化错误提示：明确说明失败原因和解决方法

### 案例2：表单设计优化

**问题发现**
- 表单字段过多，包含不必要信息（违反原则8）
- 无输入校验，提交后才报错（违反原则5）

**改进方案**
- 精简表单字段，只保留必要项
- 添加实时输入校验和格式提示

## 最佳实践

### 1. 日常应用

- 设计评审时对照十大原则检查
- 用户测试中识别违反原则的问题
- 持续优化界面可用性

### 2. 团队协作

- 将原则融入设计规范文档
- 定期进行启发式评估
- 分享改进案例

### 3. 常见错误

- ❌ 忽视状态反馈
- ❌ 使用技术语言
- ❌ 错误提示不友好
- ❌ 界面过于复杂
