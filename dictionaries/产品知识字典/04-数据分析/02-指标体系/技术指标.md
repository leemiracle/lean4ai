# 技术监控指标

## 1. 概述

技术指标用于监控系统的性能、稳定性、可靠性等技术状态，是保障产品服务质量和用户体验的重要工具。

### 技术指标框架

```
技术指标体系：

性能指标
├─ 响应时间：请求处理速度
├─ 吞吐量：系统处理能力
├─ 并发数：同时处理请求数
└─ 资源利用率：CPU、内存、磁盘

可用性指标
├─ 可用率：系统正常运行时间
├─ 故障率：系统故障频率
├─ MTTR：平均恢复时间
└─ MTBF：平均故障间隔

质量指标
├─ 错误率：请求失败比例
├─ 异常数：系统异常数量
├─ Bug率：代码缺陷密度
└─ 测试覆盖率：测试完整度

运维指标
├─ 部署频率：发布节奏
├─ 变更成功率：发布成功率
├─ 回滚率：发布回滚比例
└─ 告警数量：系统告警频次
```

## 2. 性能指标

### 2.1 响应时间指标

**核心指标**

```
响应时间（Response Time）
├─ 定义：从请求发出到收到响应的时间
├─ 计算：响应时间 = 网络时间 + 服务处理时间 + 数据库时间
└─ 标准：P95 < 200ms, P99 < 500ms

关键统计量：
├─ 平均值（Mean）：平均响应时间
├─ 中位数（Median）：P50响应时间
├─ P95：95%的请求响应时间
└─ P99：99%的请求响应时间

TTFB（Time To First Byte）
├─ 定义：首字节时间
├─ 计算：从请求到收到第一个字节的时间
└─ 标准：< 100ms

页面加载时间
├─ 定义：页面完全加载的时间
├─ 计算：DOMContentLoaded / Load事件时间
└─ 标准：< 3秒
```

**SQL示例：响应时间分析**
```sql
-- 接口响应时间分析
SELECT 
    api_endpoint,
    COUNT(*) as request_count,
    AVG(response_time) as avg_time_ms,
    PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY response_time) as p50,
    PERCENTILE_CONT(0.95) WITHIN GROUP (ORDER BY response_time) as p95,
    PERCENTILE_CONT(0.99) WITHIN GROUP (ORDER BY response_time) as p99,
    MAX(response_time) as max_time_ms
FROM api_logs
WHERE request_time >= DATE_SUB(NOW(), INTERVAL 24 HOUR)
GROUP BY api_endpoint
HAVING COUNT(*) > 100  -- 只看有足够样本的接口
ORDER BY avg_time_ms DESC;
```

**Python分析示例**
```python
import numpy as np
import pandas as pd

# 响应时间分析
def analyze_response_times(response_times):
    """
    分析响应时间分布
    """
    stats = {
        'count': len(response_times),
        'mean': np.mean(response_times),
        'median': np.median(response_times),
        'p90': np.percentile(response_times, 90),
        'p95': np.percentile(response_times, 95),
        'p99': np.percentile(response_times, 99),
        'max': np.max(response_times),
        'std': np.std(response_times)
    }
    
    # 分段统计
    bins = [0, 100, 200, 500, 1000, 2000, float('inf')]
    labels = ['0-100ms', '100-200ms', '200-500ms', '500ms-1s', '1-2s', '>2s']
    distribution = pd.cut(response_times, bins=bins, labels=labels).value_counts()
    
    return stats, distribution
```

### 2.2 吞吐量指标

**核心指标**

```
QPS（Queries Per Second）
├─ 定义：每秒查询数
├─ 计算：请求数 / 秒数
└─ 用途：衡量系统处理能力

TPS（Transactions Per Second）
├─ 定义：每秒事务数
├─ 计算：事务数 / 秒数
└─ 用途：衡量数据库处理能力

RPS（Requests Per Second）
├─ 定义：每秒请求数
├─ 计算：HTTP请求数 / 秒数
└─ 用途：衡量Web服务处理能力

带宽（Bandwidth）
├─ 定义：网络传输速率
├─ 计算：传输数据量 / 时间
└─ 用途：衡量网络吞吐能力
```

**SQL示例：吞吐量监控**
```sql
-- 实时QPS监控
SELECT 
    DATE_FORMAT(request_time, '%Y-%m-%d %H:%i') as time_minute,
    COUNT(*) / 60 as qps,
    COUNT(DISTINCT user_id) as unique_users,
    SUM(CASE WHEN status_code = 200 THEN 1 ELSE 0 END) as success_count,
    SUM(CASE WHEN status_code >= 400 THEN 1 ELSE 0 END) as error_count
FROM api_logs
WHERE request_time >= DATE_SUB(NOW(), INTERVAL 2 HOUR)
GROUP BY DATE_FORMAT(request_time, '%Y-%m-%d %H:%i')
ORDER BY time_minute DESC;
```

### 2.3 并发指标

**核心指标**

```
并发连接数
├─ 定义：同时建立的连接数
├─ 计算：当前活跃连接数
└─ 标准：< 系统最大并发数

并发用户数
├─ 定义：同时在线的用户数
├─ 计算：当前在线用户数
└─ 用途：评估系统负载

并发请求数
├─ 定义：同时处理的请求数
├─ 计算：当前处理中的请求数
└─ 标准：< 线程池/连接池大小

队列长度
├─ 定义：等待处理的请求数
├─ 计算：队列中的请求数
└─ 标准：< 队列容量
```

**SQL示例：并发监控**
```sql
-- 并发连接数监控
SELECT 
    TIMESTAMP(request_time) as time_second,
    COUNT(*) as concurrent_requests,
    COUNT(DISTINCT connection_id) as concurrent_connections,
    AVG(response_time) as avg_response_time
FROM (
    SELECT 
        request_time,
        connection_id,
        response_time,
        request_time + INTERVAL response_time/1000 SECOND as end_time
    FROM api_logs
    WHERE request_time >= DATE_SUB(NOW(), INTERVAL 5 MINUTE)
) t
WHERE TIMESTAMP(request_time) <= NOW()
    AND TIMESTAMP(end_time) >= NOW()
GROUP BY TIMESTAMP(request_time)
ORDER BY time_second DESC;
```

### 2.4 资源利用率

**核心指标**

```
CPU使用率
├─ 定义：CPU使用百分比
├─ 计算：CPU使用时间 / 总时间 × 100%
├─ 标准：< 70%（警戒线）
└─ 预警：> 80%

内存使用率
├─ 定义：内存使用百分比
├─ 计算：已用内存 / 总内存 × 100%
├─ 标准：< 80%（警戒线）
└─ 预警：> 90%

磁盘使用率
├─ 定义：磁盘使用百分比
├─ 计算：已用空间 / 总空间 × 100%
├─ 标准：< 80%（警戒线）
└─ 预警：> 90%

磁盘I/O
├─ IOPS：每秒I/O操作数
├─ 读写吞吐量：MB/s
└─ 响应时间：ms

网络I/O
├─ 入站流量：入站带宽使用
├─ 出站流量：出站带宽使用
└─ 连接数：网络连接数
```

**SQL示例：资源监控**
```sql
-- 服务器资源使用监控
SELECT 
    server_name,
    TIMESTAMP(record_time) as time,
    cpu_usage,
    memory_usage,
    disk_usage,
    disk_io_read,
    disk_io_write,
    network_in,
    network_out,
    CASE 
        WHEN cpu_usage > 80 OR memory_usage > 90 OR disk_usage > 90 THEN 'critical'
        WHEN cpu_usage > 70 OR memory_usage > 80 OR disk_usage > 80 THEN 'warning'
        ELSE 'normal'
    END as status
FROM server_metrics
WHERE record_time >= DATE_SUB(NOW(), INTERVAL 1 HOUR)
ORDER BY record_time DESC, server_name;
```

## 3. 可用性指标

### 3.1 核心指标

```
可用性（Availability）
├─ 定义：系统正常运行时间占比
├─ 计算：可用时间 / 总时间 × 100%
├─ 标准：99.9%（3个9）
│         99.99%（4个9）
│         99.999%（5个9）
└─ 允许停机时间：
    99.9%: 8.76小时/年
    99.99%: 52.6分钟/年
    99.999%: 5.26分钟/年

故障率（Failure Rate）
├─ 定义：系统故障的比例
├─ 计算：故障次数 / 总次数 × 100%
└─ 标准：< 0.1%

MTBF（Mean Time Between Failures）
├─ 定义：平均故障间隔时间
├─ 计算：总运行时间 / 故障次数
└─ 用途：评估系统可靠性

MTTR（Mean Time To Recovery）
├─ 定义：平均故障恢复时间
├─ 计算：总故障时间 / 故障次数
└─ 标准：< 1小时
```

**SQL示例：可用性计算**
```sql
-- 系统可用性计算
WITH uptime_data AS (
    SELECT 
        DATE(record_time) as date,
        SUM(CASE WHEN status = 'up' THEN 1 ELSE 0 END) * 60 as uptime_minutes,
        SUM(CASE WHEN status = 'down' THEN 1 ELSE 0 END) * 60 as downtime_minutes,
        COUNT(*) * 60 as total_minutes
    FROM system_status
    WHERE record_time >= DATE_SUB(NOW(), INTERVAL 30 DAY)
    GROUP BY DATE(record_time)
)
SELECT 
    date,
    uptime_minutes,
    downtime_minutes,
    ROUND(uptime_minutes * 100.0 / total_minutes, 2) as availability_pct,
    CASE 
        WHEN uptime_minutes * 100.0 / total_minutes >= 99.99 THEN 'excellent'
        WHEN uptime_minutes * 100.0 / total_minutes >= 99.9 THEN 'good'
        WHEN uptime_minutes * 100.0 / total_minutes >= 99.0 THEN 'acceptable'
        ELSE 'poor'
    END as sla_status
FROM uptime_data
ORDER BY date DESC;
```

### 3.2 故障指标

```
故障等级分类：

P0 - 致命故障
├─ 影响：核心服务不可用
├─ 响应：立即响应，5分钟内
├─ 恢复：30分钟内恢复
└─ 通知：全员通知

P1 - 严重故障
├─ 影响：核心功能受损
├─ 响应：15分钟内响应
├─ 恢复：2小时内恢复
└─ 通知：技术团队+管理层

P2 - 一般故障
├─ 影响：部分功能异常
├─ 响应：1小时内响应
├─ 恢复：1个工作日内
└─ 通知：技术团队

P3 - 轻微故障
├─ 影响：用户体验受影响
├─ 响应：1个工作日内
├─ 恢复：下个版本
└─ 通知：相关团队
```

**SQL示例：故障统计**
```sql
-- 故障统计分析
SELECT 
    DATE_TRUNC('month', incident_time) as month,
    severity_level,
    COUNT(*) as incident_count,
    AVG(TIMESTAMPDIFF(HOUR, incident_time, resolved_time)) as avg_resolution_hours,
    SUM(TIMESTAMPDIFF(MINUTE, incident_time, resolved_time)) as total_downtime_minutes,
    SUM(affected_users) as total_affected_users
FROM incidents
WHERE incident_time >= DATE_SUB(NOW(), INTERVAL 12 MONTH)
GROUP BY DATE_TRUNC('month', incident_time), severity_level
ORDER BY month DESC, severity_level;
```

## 4. 质量指标

### 4.1 错误率指标

**核心指标**

```
HTTP错误率
├─ 4xx错误率：客户端错误
├─ 5xx错误率：服务端错误
└─ 总错误率：(4xx + 5xx) / 总请求 × 100%

业务错误率
├─ 定义：业务逻辑失败的比例
├─ 计算：失败次数 / 总次数 × 100%
└─ 标准：< 1%

异常率
├─ 定义：系统异常的比例
├─ 计算：异常次数 / 总次数 × 100%
└─ 标准：< 0.1%

超时率
├─ 定义：请求超时的比例
├─ 计算：超时次数 / 总次数 × 100%
└─ 标准：< 0.5%
```

**SQL示例：错误率监控**
```sql
-- API错误率监控
SELECT 
    api_endpoint,
    COUNT(*) as total_requests,
    SUM(CASE WHEN status_code >= 400 AND status_code < 500 THEN 1 ELSE 0 END) as client_errors,
    SUM(CASE WHEN status_code >= 500 THEN 1 ELSE 0 END) as server_errors,
    ROUND(SUM(CASE WHEN status_code >= 400 THEN 1 ELSE 0 END) * 100.0 / COUNT(*), 2) as error_rate,
    ROUND(SUM(CASE WHEN status_code >= 500 THEN 1 ELSE 0 END) * 100.0 / COUNT(*), 2) as server_error_rate,
    ROUND(SUM(CASE WHEN is_timeout = 1 THEN 1 ELSE 0 END) * 100.0 / COUNT(*), 2) as timeout_rate
FROM api_logs
WHERE request_time >= DATE_SUB(NOW(), INTERVAL 24 HOUR)
GROUP BY api_endpoint
HAVING error_rate > 0.1  -- 只看错误率超过0.1%的接口
ORDER BY error_rate DESC;
```

```
└─ 标准：> 80%

代码复杂度

└─ 标准：圈复杂度 < 10

代码重复率

└─ 标准：< 5%

技术债务

├─ 计算：代码问题数量
```

## 5. 运维指标

### 5.1 发布指标

**核心指标**

```
部署频率
├─ 定义：单位时间内的部署次数
├─ 计算：部署次数 / 时间周期
└─ 标准：每天多次（精英级）

变更失败率
├─ 定义：部署失败的比例
├─ 计算：失败部署数 / 总部署数 × 100%
└─ 标准：< 5%

平均恢复时间
├─ 定义：从部署失败到恢复的时间
├─ 计算：平均(恢复时间 - 失败时间)
└─ 标准：< 1小时

回滚率
├─ 定义：部署后回滚的比例
├─ 计算：回滚次数 / 总部署数 × 100%
└─ 标准：< 10%
```

**SQL示例：发布统计**
```sql
-- 发布统计分析
SELECT 
    DATE_TRUNC('week', deploy_time) as week,
    COUNT(*) as total_deploys,
    SUM(CASE WHEN status = 'success' THEN 1 ELSE 0 END) as successful,
    SUM(CASE WHEN status = 'failed' THEN 1 ELSE 0 END) as failed,
    SUM(CASE WHEN is_rollback = 1 THEN 1 ELSE 0 END) as rollbacks,
    ROUND(SUM(CASE WHEN status = 'success' THEN 1 ELSE 0 END) * 100.0 / COUNT(*), 2) as success_rate,
    ROUND(AVG(CASE WHEN status = 'failed' 
        THEN TIMESTAMPDIFF(MINUTE, deploy_time, rollback_time) END), 2) as avg_recovery_min
FROM deployments
WHERE deploy_time >= DATE_SUB(NOW(), INTERVAL 12 WEEK)
GROUP BY DATE_TRUNC('week', deploy_time)
ORDER BY week DESC;
```

### 5.2 监控告警

**核心指标**

```
告警数量
├─ 定义：单位时间内的告警次数
├─ 计算：告警次数 / 时间周期
└─ 用途：评估系统健康度

告警响应时间
├─ 定义：从告警到响应的时间
├─ 计算：平均(响应时间 - 告警时间)
└─ 标准：< 5分钟

告警准确率
├─ 定义：真实问题的告警比例
├─ 计算：真实告警 / 总告警 × 100%
└─ 标准：> 90%

告警处理率
├─ 定义：已处理告警的比例
├─ 计算：已处理 / 总告警 × 100%
└─ 标准：100%
```

**SQL示例：告警统计**
```sql
-- 告警统计分析
SELECT 
    alert_type,
    severity,
    COUNT(*) as alert_count,
    SUM(CASE WHEN is_true_positive = 1 THEN 1 ELSE 0 END) as true_positives,
    AVG(TIMESTAMPDIFF(MINUTE, alert_time, ack_time)) as avg_ack_minutes,
    AVG(TIMESTAMPDIFF(MINUTE, alert_time, resolved_time)) as avg_resolve_minutes,
    ROUND(SUM(CASE WHEN is_resolved = 1 THEN 1 ELSE 0 END) * 100.0 / COUNT(*), 2) as resolve_rate
FROM alerts
WHERE alert_time >= DATE_SUB(NOW(), INTERVAL 30 DAY)
GROUP BY alert_type, severity
ORDER BY alert_count DESC;
```

## 7. 告警策略

### 7.1 告警规则设计

```yaml
# 告警规则配置示例
alert_rules:
  - name: high_error_rate
    expr: error_rate > 0.05
    for: 5m
    severity: critical
    annotations:
      summary: "高错误率告警"
      description: "接口 {{ $labels.api }} 错误率 {{ $value }}%"
    
  - name: high_response_time
    expr: histogram_quantile(0.95, response_time_seconds) > 1
    for: 10m
    severity: warning
    annotations:
      summary: "响应时间过长"
      description: "P95响应时间 {{ $value }}秒"
    
  - name: low_availability
    expr: availability < 0.999
    for: 1m
    severity: critical
    annotations:
      summary: "可用性降低"
      description: "系统可用性 {{ $value }}"
```

## 8. 监控最佳实践

### 8.1 监控设计原则

```
1. 监控全覆盖
   ├─ 基础设施
   ├─ 应用性能
   ├─ 业务指标
   └─ 用户体验

2. 告警精准
   ├─ 减少噪音
   ├─ 避免告警疲劳
   └─ 提高准确率

3. 响应及时
   ├─ 快速响应
   ├─ 明确责任
   └─ 有效处理
```

## 10. 总结

技术监控指标是保障系统稳定运行的基础：

**核心原则**
- 全面覆盖：不遗漏关键环节
- 重点突出：聚焦核心指标
- 及时响应：快速发现问题
- 持续优化：不断提升监控能力

**关键指标**
- 性能：响应时间、吞吐量、并发
- 可用性：可用率、MTTR、MTBF
- 质量：错误率、代码质量
- 运维：部署频率、告警管理

**实施要点**
- 建立完善的监控体系
- 设计合理的告警策略
- 确保快速响应机制
- 持续优化监控效果

记住：**没有监控的系统就像闭着眼睛开车，随时可能出事故。好的监控系统能够让你洞察系统状态，快速定位和解决问题。**
