# 敏捷开发

## 一、定义

**敏捷开发(Agile Development)**是一种以人为核心、迭代、循序渐进的软件开发方法,强调快速响应变化、持续交付价值、紧密协作。

```
敏捷宣言

个体和互动   重于  流程和工具
可工作的软件 重于  详尽的文档
客户合作     重于  合同谈判
响应变化     重于  遵循计划

虽然右项也有价值
但我们更重视左项
```

## 二、敏捷方法论

### 2.1 Scrum框架

```
Scrum框架

┌────────────────────────────────┐
│          Product Owner         │
│         (产品负责人)            │
└────────────┬───────────────────┘
             │
┌────────────┴───────────────────┐
│      Product Backlog           │
│         (产品待办列表)          │
└────────────┬───────────────────┘
             ↓
    ┌────────┴────────┐
    │  Sprint Planning │
    │    (冲刺规划)     │
    └────────┬────────┘
             ↓
    ┌────────┴────────┐
    │  Sprint Backlog  │
    │   (冲刺待办)     │
    └────────┬────────┘
             ↓
    ┌────────┴────────┐
    │  Daily Scrum     │
    │   (每日站会)     │  2-4周
    │      ↕          │
    │  Sprint开发      │
    └────────┬────────┘
             ↓
    ┌────────┴────────┐
    │ Sprint Review   │
    │   (冲刺评审)     │
    └────────┬────────┘
             ↓
    ┌────────┴────────┐
    │Retrospective    │
    │   (回顾会议)     │
    └─────────────────┘
```

**Scrum角色**:
```
1. Product Owner(产品负责人)
   职责:
   - 管理Product Backlog
   - 定义产品愿景
   - 决定发布日期
   - 接受或拒绝工作成果

2. Scrum Master(敏捷教练)
   职责:
   - 保证Scrum流程
   - 移除障碍
   - 促进团队协作
   - 教练团队

3. Development Team(开发团队)
   职责:
   - 自组织
   - 交付产品增量
   - 5-9人最佳
   - 跨职能
```

**Scrum会议**:
```
1. Sprint Planning(冲刺规划)
   时长: 2-4小时
   内容:
   - 选择Sprint目标
   - 选择Backlog项
   - 制定计划
   
2. Daily Scrum(每日站会)
   时长: 15分钟
   内容:
   - 昨天做了什么
   - 今天计划做什么
   - 有什么障碍
   
3. Sprint Review(冲刺评审)
   时长: 1-2小时
   内容:
   - 演示完成的工作
   - 收集反馈
   - 更新Backlog
   
4. Retrospective(回顾会议)
   时长: 1-1.5小时
   内容:
   - 哪些做得好
   - 哪些需要改进
   - 行动计划
```

### 2.2 看板方法(Kanban)

```
看板方法

┌─────────┬─────────┬─────────┬─────────┐
│ 待办    │  进行中 │  测试   │  完成   │
│  To Do  │ In Progress│ Testing│  Done  │
├─────────┼─────────┼─────────┼─────────┤
│ 任务1   │ 任务4   │ 任务7   │ 任务10  │
│ 任务2   │ 任务5   │         │ 任务11  │
│ 任务3   │ 任务6   │         │ 任务12  │
│         │         │         │         │
│   3     │   2     │   1     │    5    │
└─────────┴─────────┴─────────┴─────────┘
        WIP限制

核心原则:
1. 可视化工作流
2. 限制在制品数量(WIP)
3. 管理流动
4. 持续改进
```

**看板实践**:
```
1. 可视化工作流
   - 创建看板
   - 显示所有工作项
   - 明确流程

2. 限制WIP
   - 每列设置上限
   - 避免多任务
   - 提高专注度

3. 管理流动
   - 监控周期时间
   - 识别瓶颈
   - 优化流程

4. 持续改进
   - 定期回顾
   - 数据驱动
   - 小步改进
```

### 2.3 用户故事

```
用户故事格式

作为 [角色]
我想要 [功能]
以便于 [价值]

示例:
作为 用户
我想要 搜索商品
以便于 快速找到需要的商品

验收标准:
Given [前置条件]
When [操作]
Then [预期结果]

示例:
Given 用户在搜索页面
When 输入关键词并点击搜索
Then 显示匹配的商品列表
```

**用户故事拆分**:
```
拆分原则(INVEST):
- Independent(独立的)
- Negotiable(可协商的)
- Valuable(有价值的)
- Estimable(可估算的)
- Small(小的)
- Testable(可测试的)

拆分方法:
1. 按工作流程拆分
2. 按操作拆分
3. 按数据类型拆分
4. 按简单/复杂拆分

示例:
大故事: 用户管理
↓ 拆分
- 用户注册
- 用户登录
- 用户资料编辑
- 密码重置
```

## 三、敏捷实践

### 3.1 迭代开发

```
迭代开发周期

Sprint(2-4周)
   │
   ├─ Day 1: Sprint规划
   │
   ├─ Day 2-n: 开发
   │   ├─ 每日站会
   │   ├─ 开发测试
   │   └─ 代码评审
   │
   ├─ 倒数第2天: Sprint评审
   │
   └─ 最后1天: 回顾会议

迭代原则:
1. 固定时间盒
2. 可交付增量
3. 持续改进
```

CI/CD流程

### 3.3 产品待办列表管理

```
Product Backlog管理

优先级排序:
P0(紧急重要)
  ↓
P1(重要不紧急)
  ↓
P2(紧急不重要)
  ↓
P3(不紧急不重要)

Backlog精化(Backlog Grooming):
频率: 每周1次
时长: 1-2小时
内容:
- 添加新需求
- 调整优先级
- 拆分大故事
- 估算故事点

估算方法:
1. 故事点(Story Points)
   - 相对估算
   - 斐波那契数列(1,2,3,5,8,13...)
   
2. 规划扑克(Planning Poker)
   - 团队共同估算
   - 讨论差异
   - 达成共识
```

## 四、敏捷团队

### 4.1 团队特征

```
敏捷团队特征

1. 跨职能
   - 产品、设计、开发、测试
   - 自给自足
   - 减少依赖

2. 自组织
   - 自主决策
   - 共同负责
   - 团队ownership

3. 小团队
   - 5-9人最佳
   - 沟通成本低
   - 效率高

4. 同地办公
   - 面对面沟通
   - 快速反馈
   - 白板协作
```

### 4.2 团队协作

```
团队协作最佳实践

1. 信息透明
   - 看板可视化
   - 进度公开
   - 指标共享

2. 高频沟通
   - 每日站会
   - 即时沟通
   - 快速反馈

3. 代码协作
   - 代码评审
   - 结对编程
   - 共享代码权

4. 文档协作
   - 在线文档
   - 实时协作
   - 版本管理
```

## 五、敏捷度量

### 5.1 关键指标

```
敏捷度量指标

1. 速度(Velocity)
   - 每Sprint完成的故事点
   - 用于预测
   
2. 周期时间(Cycle Time)
   - 从开始到完成的时间
   - 越短越好

3. 吞吐量(Throughput)
   - 单位时间完成的任务数
   - 衡量产能

4. 燃尽图(Burndown Chart)
   剩余工作量
       ↑
       │╲
       │ ╲
       │  ╲
       │   ╲
       └────→ 时间

5. 燃起图(Burnup Chart)
   完成工作量
       ↑
       │    ╱
       │   ╱
       │  ╱
       │ ╱
       └────→ 时间
```

### 5.2 度量陷阱

```
避免度量陷阱

✗ 用速度考核个人
  → 导致虚高估算
  
✗ 牺牲质量追求速度
  → 技术债务
  
✗ 伪造数据
  → 失去意义
  
✗ 过度优化指标
  → 忽视价值

正确做法:
✓ 指标用于改进,不是考核
✓ 关注趋势,不是绝对值
✓ 团队指标,不是个人
✓ 数据驱动对话
```

## 七、敏捷转型

### 7.1 转型步骤

```
敏捷转型路径

阶段1: 试点(1-3个月)
- 选择1-2个团队
- 培训敏捷方法
- 小范围实践

阶段2: 扩展(3-6个月)
- 更多团队加入
- 建立最佳实践
- 调整流程

阶段3: 深化(6-12个月)
- 全组织敏捷
- 文化转型
- 持续改进

成功要素:
1. 高层支持
2. 文化转型
3. 培训教练
4. 工具支持
5. 持续改进
```

### 7.2 常见问题

```
敏捷转型常见问题

1. 形式敏捷
   问题: 只有仪式,没有实质
   解决: 理解敏捷本质

2. 缺乏授权
   问题: 团队没有自主权
   解决: 管理层授权

3. 文档缺失
   问题: 误解"工作的软件重于文档"
   解决: 适度文档

4. 规划不足
   问题: 误解"响应变化"
   解决: 适度规划

5. 技术债务
   问题: 追求速度忽视质量
   解决: 重构时间
```

## 八、最佳实践

### 8.1 敏捷原则

```
敏捷开发12原则

1. 最高优先级是通过尽早和持续交付
   有价值的软件满足客户

2. 欣然面对需求变化,即使在开发后期

3. 经常交付可工作的软件,
   周期越短越好

4. 业务人员和开发人员必须
   每天一起工作

5. 围绕有激情的个人构建项目

6. 面对面交谈是最有效的沟通方式

7. 可工作的软件是进度的首要度量

8. 可持续的开发速度

9. 持续关注技术卓越和良好设计

10. 简单化-最大化未做工作的艺术

11. 最好的架构、需求和设计
    出自自组织团队

12. 团队定期反思如何提高效率
```

### 8.2 实践建议

```
敏捷实践建议

DO:
✓ 小步快跑,持续交付
✓ 面对面沟通
✓ 拥抱变化
✓ 团队协作
✓ 持续改进
✓ 技术卓越

DON'T:
✗ 追求完美
✗ 长期规划
✗ 大批量开发
✗ 忽视质量
✗ 形式主义
✗ 个人英雄主义
```

## 十、总结

```
敏捷开发核心

价值观:
个体和互动 > 流程和工具
可工作的软件 > 详尽的文档
客户合作 > 合同谈判
响应变化 > 遵循计划

关键:
1. 小步快跑
2. 持续交付
3. 拥抱变化
4. 团队协作
5. 技术卓越

记住:
敏捷不是目的,是手段
目标是持续交付价值
```

**敏捷的本质**:
> "敏捷不是一种方法论,
> 而是一种思维方式,
> 一种快速响应变化、持续创造价值的
> 能力和文化。"
