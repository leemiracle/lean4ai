# 风险评估

## 一、定义

**风险评估**是识别、分析和评估潜在风险，并制定应对策略的过程，目的是将风险控制在可接受范围内。

```
风险管理流程

┌──────────┐
│ 风险识别 │ 发现潜在风险
└─────┬────┘
      ↓
┌──────────┐
│ 风险分析 │ 评估影响和概率
└─────┬────┘
      ↓
┌──────────┐
│ 风险评估 │ 确定风险等级
└─────┬────┘
      ↓
┌──────────┐
│ 风险应对 │ 制定应对策略
└─────┬────┘
      ↓
┌──────────┐
│ 风险监控 │ 持续跟踪风险
└──────────┘
```

## 二、风险矩阵

### 2.1 基本概念

**风险矩阵**是通过风险发生概率和影响程度来评估风险等级的工具。

```
风险矩阵

                    影响程度
              低      中      高
         ┌────────┬────────┬────────┐
      高 │   中   │   高   │  极高  │
   概    ├────────┼────────┼────────┤
   率    中 │   低   │   中   │   高   │
         ├────────┼────────┼────────┤
      低 │  极低  │   低   │   中   │
         └────────┴────────┴────────┘

风险等级：
- 极高：立即处理
- 高：优先处理
- 中：计划处理
- 低：监控即可
- 极低：可接受
```

### 2.2 风险矩阵应用

```
步骤1: 识别风险
列出所有可能的风险

步骤2: 评估概率
- 高：>70%可能性发生
- 中：30-70%可能性发生
- 低：<30%可能性发生

步骤3: 评估影响
- 高：项目失败、重大损失
- 中：项目延期、中等损失
- 低：轻微影响、小损失

步骤4: 绘制矩阵
在矩阵中标出风险位置

步骤5: 制定策略
根据风险等级制定应对措施
```

### 2.3 产品风险矩阵示例

```
某产品上线风险矩阵

影响程度
              低        中        高
         ┌─────────┬─────────┬─────────┐
      高 │ 竞品模仿 │ 用户投诉 │系统崩溃 │
   概    │   (中)  │   (高)  │ (极高)  │
   率    ├─────────┼─────────┼─────────┤
      中 │UI调整   │ 功能延期 │数据泄露 │
         │  (低)   │   (中)  │  (高)   │
         ├─────────┼─────────┼─────────┤
      低 │ 文档缺失│ 预算超支 │ 核心人员│
         │ (极低)  │  (低)   │离职(中) │
         └─────────┴─────────┴─────────┘

优先处理：
1. 系统崩溃（极高）
2. 用户投诉、数据泄露（高）
3. 功能延期、竞品模仿（中）
```

## 三、风险评估方法

### 3.1 FMEA失效模式分析

```
FMEA (Failure Mode and Effects Analysis)

步骤1: 列出所有可能的失效模式
步骤2: 评估严重度(S)：1-10分
步骤3: 评估发生度(O)：1-10分
步骤4: 评估检测度(D)：1-10分
步骤5: 计算RPN = S × O × D
步骤6: 优先处理RPN高的风险

示例：支付功能FMEA

| 失效模式 | 严重度 | 发生度 | 检测度 | RPN |
|---------|-------|-------|-------|-----|
| 支付失败 | 8 | 3 | 3 | 72 |
| 重复扣款 | 10 | 2 | 5 | 100 |
| 延迟到账 | 5 | 5 | 4 | 100 |
| 金额错误 | 9 | 2 | 2 | 36 |

优先处理：重复扣款、延迟到账（RPN=100）
```

### 3.2 敏感性分析

```
敏感性分析 - 识别关键风险因素

方法：分析各因素变化对结果的影响程度

示例：产品收益敏感性分析

基准收益：100万

| 因素 | 变化 | 收益变化 | 敏感性 |
|-----|------|---------|-------|
| 用户数 | -10% | -15万 | 高 |
| 转化率 | -10% | -12万 | 高 |
| 客单价 | -10% | -10万 | 中 |
| 成本 | +10% | -8万 | 中 |

结论：用户数和转化率是最敏感因素
策略：重点监控用户数和转化率
```

### 3.3 蒙特卡洛模拟

```
蒙特卡洛模拟 - 量化不确定性风险

原理：通过大量随机模拟，得到结果分布

示例：项目工期估算

输入：
- 乐观时间：3个月
- 最可能时间：4个月
- 悲观时间：6个月

模拟10000次后：

结果分布：
      │
频率  │    ████
      │   ██████
      │  ████████
      │ ██████████
      └─────────────→ 工期
        3  4  5  6  月

结论：
- 80%概率在4-5个月内完成
- 95%概率在5.5个月内完成
- 建议预留6个月工期
```

### 3.4 情景分析

```
情景分析 - 多种可能情景下的风险评估

步骤1: 定义关键变量
步骤2: 设计多种情景
步骤3: 评估各情景影响
步骤4: 制定应对策略

示例：新产品销售预测

        市场好      市场一般     市场差
        ─────────────────────────────
营销强  销售1亿     销售7000万   销售3000万
营销中  销售8000万  销售5000万   销售2000万
营销弱  销售5000万  销售3000万   销售1000万

策略：
- 最优情景：销售1亿（概率20%）
- 基础情景：销售5000万（概率50%）
- 最差情景：销售1000万（概率10%）

期望值 = 1亿×20% + 7000万×30% + 5000万×40% + 3000万×10%
       = 5800万
```

## 四、风险类型与识别

### 4.1 产品风险分类

```
产品风险分类

1. 市场风险
   - 市场需求不足
   - 竞争激烈
   - 市场时机不对
   - 定价不当

2. 技术风险
   - 技术方案不可行
   - 技术难度超预期
   - 技术选型错误
   - 技术债务累积

3. 资源风险
   - 资金不足
   - 人员流失
   - 供应商问题
   - 设备故障

4. 管理风险
   - 项目延期
   - 需求变更频繁
   - 沟通不畅
   - 决策失误

5. 合规风险
   - 政策法规变化
   - 知识产权问题
   - 数据安全
   - 用户隐私
```

### 4.2 风险识别方法

```
1. 头脑风暴法
   召集团队列出所有可能风险

2. 德尔菲法
   多轮专家意见收集

3. 检查表法
   基于历史经验的风险清单
   
   示例检查表：
   □ 需求是否清晰？
   □ 技术方案是否验证？
   □ 资源是否充足？
   □ 时间计划是否合理？
   □ 是否有Plan B？

4. 根本原因分析
   追问5个"为什么"

5. 流程图分析
   分析各环节可能风险点

6. 历史数据分析
   参考类似项目风险
```

## 五、风险应对策略

### 5.1 四种基本策略

```
风险应对策略

1. 规避(Avoid)
   消除风险源头
   
   示例：
   - 风险：新技术不稳定
   - 策略：改用成熟技术

2. 转移(Transfer)
   将风险转给第三方
   
   示例：
   - 风险：系统故障
   - 策略：购买保险、外包运维

3. 缓解(Mitigate)
   降低风险概率或影响
   
   示例：
   - 风险：需求变更
   - 策略：加强需求评审、预留缓冲

4. 接受(Accept)
   承受风险，做好应急预案
   
   示例：
   - 风险：小概率延迟
   - 策略：准备应急方案
```

### 5.2 风险应对策略选择

```
风险应对策略选择矩阵

        │影响小          │影响大
────────┼────────────────┼──────────────
概率高  │ 缓解或接受      │ 规避或转移
        │                │
────────┼────────────────┼──────────────
概率低  │ 接受           │ 转移或缓解
        │（监控即可）     │（制定预案）
```

```
风险应对计划模板

同时由备份人员接手
```

## 六、风险评估案例

### 6.1 案例：新产品上线风险评估

```
场景：社交产品新版本上线

风险识别：

| 风险 | 概率 | 影响 | 等级 | 策略 |
|-----|------|------|------|------|
| 服务器压力 | 高 | 高 | 极高 | 缓解 |
| 用户接受度低 | 中 | 高 | 高 | 缓解 |
| Bug导致崩溃 | 中 | 高 | 高 | 缓解 |
| 竞品模仿 | 高 | 中 | 中 | 接受 |
| 预算超支 | 低 | 中 | 低 | 接受 |

风险应对措施：

1. 服务器压力（极高）
   - 灰度发布，逐步放量
   - 压力测试，预留容量
   - 弹性扩容方案
   - 实时监控告警

2. 用户接受度低（高）
   - 提前用户调研
   - 小范围测试
   - A/B测试验证
   - 快速迭代优化

3. Bug导致崩溃（高）
   - 充分测试
   - 代码审查
   - 灰度发布
   - 回滚预案

风险监控：
- 每日检查风险状态
- 每周风险评估会议
- 关键指标实时监控
```

### 6.2 案例：项目延期风险评估

```
场景：功能开发项目

FMEA分析：

| 失效模式 | 严重度(S) | 发生度(O) | 检测度(D) | RPN | 措施 |
|---------|----------|----------|----------|-----|------|
| 需求变更 | 7 | 6 | 4 | 168 | 需求评审机制 |
| 技术难题 | 8 | 4 | 3 | 96 | 技术预研 |
| 人员流失 | 9 | 3 | 5 | 135 | 团队激励 |
| 联调延期 | 6 | 5 | 4 | 120 | 提前沟通 |

敏感性分析：

| 因素 | 变化 | 工期影响 | 应对 |
|-----|------|---------|------|
| 人员 | -20% | +15天 | 预留人力缓冲 |
| 需求 | +20% | +10天 | 需求冻结机制 |
| 效率 | -10% | +7天 | 优化流程 |

蒙特卡洛模拟结果：
- 50%概率按时完成
- 80%概率延期1周内
- 95%概率延期2周内

建议工期：计划工期 + 2周缓冲
```

## 八、风险评估最佳实践

DO:
✓ 尽早识别风险
✓ 定期评估风险
✓ 量化风险影响
✓ 制定应急预案
✓ 持续监控风险
✓ 团队共同参与
✓ 记录风险历史

DON'T:
✗ 忽视小概率高风险
✗ 只评估不应对
✗ 风险应对不落地
✗ 一次性评估就结束
✗ 风险管理流于形式
✗ 无人负责风险
✗ 隐瞒风险

记住：风险管理的目的不是消除所有风险，而是将风险控制在可接受范围内，并做好应对准备。
