# 用户故事

## 一、定义

**用户故事(User Story)**是一种从用户角度描述需求的简短描述,强调用户价值而非功能列表。

```
用户故事格式

作为 <角色>
我想要 <功能/活动>
以便 <价值/目的>

(As a <role>, I want <feature>, so that <benefit>)

示例:
作为 注册用户
我想要 重置密码
以便 我忘记密码时能重新访问账户
```

## 二、用户故事组成

### 2.1 三要素(3C)

```
用户故事三要素

1. Card(卡片)
   - 简短描述
   - 便签大小
   - 只写要点

2. Conversation(对话)
   - 细节通过沟通
   - 口头交流
   - 补充信息

3. Confirmation(确认)
   - 验收标准
   - 测试用例
   - 完成定义
```

### 2.2 验收标准(Acceptance Criteria)

```
验收标准格式

Given(给定) 初始条件
When(当) 执行操作
Then(那么) 预期结果

示例:
Given 用户已登录且购物车有商品
When 用户点击结算按钮
Then 跳转到订单确认页
And 显示商品总价和运费
```

## 三、用户故事编写

### 3.1 INVEST原则

```
好的用户故事遵循INVEST原则

I - Independent(独立的)
    - 故事间相互独立
    - 可以单独开发
    例: ✗ "注册并登录"
        ✓ "注册" 和 "登录"分开

N - Negotiable(可协商的)
    - 不是合同
    - 细节可以讨论
    例: "搜索功能"(不写具体实现)

V - Valuable(有价值的)
    - 对用户有价值
    - 不是技术任务
    例: ✗ "创建数据库表"
        ✓ "保存用户信息"

E - Estimable(可估算的)
    - 可以估算工作量
    - 范围明确
    例: ✗ "优化性能"(太模糊)
        ✓ "页面加载时间<2秒"

S - Small(小的)
    - 一个Sprint能完成
    - 范围适中
    例: ✗ "电商系统"(太大)
        ✓ "添加商品到购物车"

T - Testable(可测试的)
    - 可以验证
    - 有明确标准
    例: ✗ "用户体验好"(不可测)
        ✓ "3步内完成下单"
```

### 3.2 故事拆分

```
大故事拆分方法

1. 按工作流程拆分
   用户管理 → 用户注册/登录/资料编辑

2. 按简单/复杂拆分
   搜索 → 简单搜索/高级搜索

3. 按数据类型拆分
   支付 → 支付宝/微信/银行卡

4. 按操作拆分
   商品管理 → 增/删/改/查

示例:
大故事: 用户可以管理商品
↓ 拆分
- 用户可以添加商品
- 用户可以编辑商品
- 用户可以删除商品
- 用户可以查看商品列表
```

## 四、用户故事地图

```
用户故事地图

骨架(Backbone):
用户旅程的主要步骤

┌────┬────┬────┬────┐
│发现 │选择 │购买 │评价 │  ← 骨架
├────┼────┼────┼────┤
│搜索 │详情 │下单 │打分 │
│推荐 │对比 │支付 │评论 │  ← 故事
│分类 │收藏 │物流 │分享 │
└────┴────┴────┴────┘

优先级从上到下:
- 第一行:MVP必须
- 第二行:重要功能
- 第三行:锦上添花
```

## 五、优先级排序

### 5.1 MoSCoW方法

```
MoSCoW优先级

M - Must have(必须有)
    - 没有就不能上线
    - P0级别

S - Should have(应该有)
    - 重要但可推迟
    - P1级别

C - Could have(可以有)
    - 锦上添花
    - P2级别

W - Won't have(不会有)
    - 本次不做
    - P3级别
```

### 5.2 相对优先级

```
优先级排序考虑因素

1. 用户价值
   - 解决痛点强度
   - 使用频率

2. 业务价值
   - 收入贡献
   - 战略重要性

3. 开发成本
   - 工作量
   - 技术复杂度

4. 风险
   - 技术风险
   - 业务风险

公式:
优先级分数 = (用户价值 + 业务价值) / (成本 + 风险)
```

## 六、故事点估算

### 6.1 故事点

```
故事点(Story Points)

定义: 相对复杂度度量单位

斐波那契数列:
1, 2, 3, 5, 8, 13, 21, 34, 55...

基准故事:
1点 = 简单文本修改
2点 = 简单UI调整
3点 = 基础CRUD功能
5点 = 中等复杂功能
8点 = 复杂功能
13点 = 非常复杂(需拆分)

示例:
- 修改文案: 1点
- 添加输入框: 2点
- 用户注册: 3点
- 搜索功能: 5点
- 支付集成: 8点
```

### 6.2 规划扑克

```
Planning Poker流程

1. 选择一个故事
2. 每个人独立估算
3. 同时亮牌
4. 差异大的说明理由
5. 讨论后重新估算
6. 达成共识

示例:
故事: 用户登录功能

产品: 介绍需求
开发A: 出5
开发B: 出8
开发C: 出3

讨论:
- A: 需要多种登录方式
- B: 要考虑安全,复杂
- C: 只是表单提交,简单

结论: 考虑到安全验证,最终定5点
```

## 七、完成的定义(DoD)

```
Definition of Done

用户故事完成的标志:

□ 代码完成并提交
□ 单元测试通过
□ 代码评审通过
□ 功能测试通过
□ 产品验收通过
□ 文档已更新
□ 无严重Bug
□ 已部署到测试环境

只有满足所有条件
才算真正完成
```

## 八、最佳实践

### 8.1 编写技巧

```
用户故事编写技巧

DO:
✓ 从用户角度出发
✓ 保持简洁
✓ 关注价值而非功能
✓ 包含验收标准
✓ 可以估算

DON'T:
✗ 技术导向
✗ 太大或太小
✗ 模糊不清
✗ 缺少价值说明
✗ 无法测试
```

### 8.2 常见错误

```
常见错误示例

错误1: 技术故事
✗ "使用Redis缓存数据"
✓ "商品列表加载时间<2秒"

错误2: 太大
✗ "电商系统"
✓ 拆分为多个小故事

错误3: 缺少价值
✗ "作为用户,我想要搜索"
✓ "作为用户,我想要搜索,以便快速找到商品"

错误4: 不可测试
✗ "体验要好"
✓ "3步内完成下单"
```

## 十、总结

```
用户故事核心

格式:
作为<角色>,我想要<功能>,以便<价值>

INVEST原则:
Independent, Negotiable, Valuable, 
Estimable, Small, Testable

关键:
1. 用户视角
2. 价值导向
3. 小而独立
4. 可估算可测试

记住:
用户故事不是需求文档
而是沟通工具
```
