# PRD编写

## 一、定义

**PRD(Product Requirements Document)**即产品需求文档,是产品经理用来描述产品功能、业务逻辑、用户场景等需求的详细文档。

```
PRD定位

产品愿景
   ↓
┌──┴──┐
│ PRD  │ ← 连接愿景与执行
└──┬──┘
   ↓
开发实现

核心作用:
- 对齐团队理解
- 指导开发
- 验收标准
- 知识沉淀
```

## 三、需求编写方法

### 3.1 需求描述方法

```
需求描述框架

WHO: 用户是谁
WHAT: 需要什么
WHY: 为什么需要
WHEN: 什么时候需要
WHERE: 在哪里使用
HOW: 如何使用

示例:
❌ 错误: 增加搜索功能
✓ 正确: 
作为用户(WHO),
我想要搜索商品(WHAT),
以便快速找到想要的商品(WHY),
在浏览商品列表时使用(WHEN/WHERE),
通过输入关键词搜索(HOW)
```

### 3.2 功能需求模板

```markdown
## 功能需求

### 1. 功能名称
**优先级**: P0/P1/P2/P3

**需求描述**:
[详细描述功能需求]

**用户场景**:
1. 用户进入XXX页面
2. 用户点击XXX按钮
3. 系统显示XXX
4. 用户完成XXX操作

**业务规则**:
- 规则1: XXX
- 规则2: XXX

**前置条件**:
- 条件1: XXX

**后置条件**:
- 结果1: XXX

**异常处理**:
- 异常1: XXX → 处理方式

**界面要求**:
- 界面说明或原型链接
```

### 3.3 非功能需求

```
非功能需求类型

1. 性能需求
   - 响应时间: < 2秒
   - 并发数: 1000 QPS
   - 可用性: 99.9%

2. 安全需求
   - 数据加密
   - 权限控制
   - 防注入攻击

3. 兼容性需求
   - 浏览器: Chrome/Safari/Firefox
   - 设备: iOS/Android/Web
   - 屏幕: 响应式

4. 可维护性
   - 代码规范
   - 文档要求
   - 日志规范

5. 可扩展性
   - 架构设计
   - 扩展预留
```

## 四、PRD写作技巧

### 4.1 写作原则

```
PRD写作原则

CLEAR原则:
C - Complete(完整)
L - Logical(逻辑清晰)
E - Exact(准确)
A - Accessible(易懂)
R - Reviewable(可审核)

具体要求:
1. 简洁明确
   - 一句话能说清楚不用两句
   - 避免模糊词汇(可能、大概、也许)

2. 逻辑清晰
   - 结构化组织
   - 层次分明
   - 前后一致

3. 可视化
   - 用图表代替文字
   - 流程图、状态图
   - 原型图、截图

4. 可追溯
   - 需求来源
   - 决策依据
   - 变更记录
```

### 4.2 常见问题

```
PRD常见问题

✗ 问题1: 需求模糊
   例: "做一个好用的搜索"
   ✓ 改进: "支持关键词、分类、价格区间筛选的搜索"

✗ 问题2: 缺少场景
   例: "用户可以分享"
   ✓ 改进: "用户在查看商品详情页时,可分享到微信朋友圈"

✗ 问题3: 混入解决方案
   例: "用Redis缓存数据"
   ✓ 改进: "商品列表加载时间<2秒"(技术方案留给开发)

✗ 问题4: 忽视异常
   例: 只写正常流程
   ✓ 改进: 包含网络失败、数据为空等异常处理

✗ 问题5: 缺少优先级
   例: 所有功能都P0
   ✓ 改进: 明确优先级排序
```

## 六、PRD评审

### 6.1 评审流程

```
PRD评审流程

1. 自我检查
   - 逻辑完整性
   - 需求清晰度
   - 技术可行性

2. 内部评审
   - 产品团队内部
   - 查漏补缺
   - 完善细节

3. 技术评审
   - 开发参与
   - 评估技术方案
   - 工作量估算

4. 正式评审
   - 相关干系人
   - 最终确认
   - 评审通过

评审要点:
□ 需求是否完整?
□ 描述是否清晰?
□ 逻辑是否合理?
□ 技术是否可行?
□ 优先级是否合理?
```

### 6.2 评审checklist

```
PRD评审检查清单

【完整性】
□ 所有场景都覆盖了吗?
□ 异常情况都考虑了吗?
□ 边界条件都明确了吗?

【清晰性】
□ 需求描述是否明确?
□ 是否有歧义?
□ 是否理解一致?

【可行性】
□ 技术可行吗?
□ 资源足够吗?
□ 时间合理吗?

【优先级】
□ 优先级排序合理吗?
□ P0功能是否必要?
□ 是否可以分期?

【可测试性】
□ 验收标准明确吗?
□ 可以测试吗?
□ 数据埋点设计了吗?
```

### 7.2 最佳实践

```
PRD最佳实践

DO:
✓ 先和开发沟通技术方案
✓ 用图表代替大段文字
✓ 保持简洁但完整
✓ 及时更新版本
✓ 邀请相关方评审

DON'T:
✗ 需求频繁变更不更新文档
✗ 技术细节写太细
✗ 需求描述模糊
✗ 缺少验收标准
✗ 写完不沟通直接扔给开发
```

## 九、总结

```
PRD核心要点

目的:
- 对齐团队理解
- 指导开发实施
- 明确验收标准

原则:
- 完整但不冗余
- 清晰且有逻辑
- 可追溯可评审

记住:
好的PRD不是写出来的
是沟通出来的
持续更新比完美更重要
```
