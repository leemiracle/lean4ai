# 产品评审

## 一、定义

**产品评审**是产品开发过程中各阶段的检查点，确保产品方向正确、质量达标、风险可控。

```
产品评审流程

需求阶段     设计阶段     开发阶段     上线阶段
    │           │           │           │
    ↓           ↓           ↓           ↓
┌───────┐   ┌───────┐   ┌───────┐   ┌───────┐
│需求   │   │设计   │   │代码   │   │上线   │
│评审   │   │评审   │   │评审   │   │评审   │
└───────┘   └───────┘   └───────┘   └───────┘

目的：
- 确保方向正确
- 保证质量
- 控制风险
- 对齐认知
```

## 二、需求评审

### 2.1 评审目标

```
需求评审目标

1. 需求合理性
   ┌─────────────────────┐
   │ - 是否符合产品战略？│
   │ - 是否有用户价值？  │
   │ - 是否有业务价值？  │
   └─────────────────────┘

2. 需求完整性
   ┌─────────────────────┐
   │ - 需求描述是否完整？│
   │ - 是否覆盖边界情况？│
   │ - 是否有遗漏？      │
   └─────────────────────┘

3. 需求可行性
   ┌─────────────────────┐
   │ - 技术是否可行？    │
   │ - 资源是否充足？    │
   │ - 时间是否合理？    │
   └─────────────────────┘

4. 需求优先级
   ┌─────────────────────┐
   │ - 优先级是否合理？  │
   │ - 是否可以分期？    │
   └─────────────────────┘
```

### 2.2 评审流程

```
需求评审流程

会前准备：
┌─────────────────────┐
│ 1. 完成需求文档     │
│ 2. 发送评审邀请     │
│ 3. 提前分享文档     │
│ 4. 准备演示材料     │
└─────────────────────┘
         ↓
评审会议：
┌─────────────────────┐
│ 1. 需求背景介绍     │
│ 2. 需求详细讲解     │
│ 3. 讨论和答疑       │
│ 4. 可行性评估       │
│ 5. 工时估算         │
│ 6. 确认优先级       │
└─────────────────────┘
         ↓
会后跟进：
┌─────────────────────┐
│ 1. 整理评审结论     │
│ 2. 更新需求文档     │
│ 3. 同步相关人员     │
│ 4. 安排下次评审     │
└─────────────────────┘
```

```
需求评审参与者

必选参与者：
┌─────────────────────┐
│ □ 产品经理          │
│ □ 开发负责人        │

│ □ 测试负责人        │

```

```
3. 记录差异和问题
4. 确认修改范围
5. 跟踪修改结果

走查清单：
□ 布局还原度
□ 颜色准确性
□ 字体大小
□ 间距间距
□ 动效效果
□ 交互行为
□ 边界情况

- 字体大小错误

```

```
         ↓
提交评审：
┌─────────────────────┐
│ 1. 提交MR/PR        │

└─────────────────────┘
         ↓
评审过程：
┌─────────────────────┐
│ 1. 查看变更说明     │

│ 4. 讨论技术方案     │
└─────────────────────┘
         ↓
评审结果：
┌─────────────────────┐
│ 通过 → 合并代码     │
│ 修改 → 更新后重审   │
│ 拒绝 → 说明原因     │
└─────────────────────┘
```

### 4.3 评审标准

```
代码评审标准

功能：
□ 是否实现需求功能？

□ 错误处理是否完善？
□ 是否有遗漏场景？

□ 代码是否规范？
□ 命名是否清晰？
□ 逻辑是否简洁？
□ 是否有重复代码？
```

### 4.4 评审技巧

```
代码评审技巧

评审者：
DO:
✓ 关注问题，不是人
✓ 提出建设性建议
✓ 肯定好的代码
✓ 解释为什么
✓ 及时响应

DON'T:
✗ 人身攻击
✗ 只批评不表扬
✗ 过于苛刻
✗ 拖延评审
✗ 不给理由

被评审者：
DO:
✓ 保持开放心态
✓ 解释设计意图
✓ 虚心接受建议
✓ 及时响应问题

DON'T:
✗ 防御心理
✗ 忽视建议
✗ 争论不休
✗ 拖延修改
```

### 5.4 上线决策

```
上线决策矩阵

              影响大
                │
       不能上线  │  需要审批
       (高风险)  │  (中风险)
    ───────────┼───────────
       可以上线  │  正常上线
       (低风险)  │  (标准流程)
                │
         问题少 ────────→ 问题多

决策标准：

不能上线：
- 有严重bug未修复
- 测试未完成
- 回滚方案缺失
- 关键风险未处理

需要审批：
- 有已知非严重bug
- 测试基本完成
- 有回滚方案
- 需要高层决策

可以上线：
- 测试全部通过
- 无已知bug
- 准备充分
```
