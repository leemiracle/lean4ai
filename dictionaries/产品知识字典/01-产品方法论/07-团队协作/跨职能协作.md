# 跨职能协作

## 一、定义

**跨职能协作**是指产品、设计、开发等不同职能团队成员之间的合作，共同完成产品目标。

```
跨职能协作三角

         产品
          ●
         ╱ ╲
        ╱   ╲
       ╱     ╲
      ╱       ╲
     ╱         ╲
    ●───────────●
   设计        开发

核心原则：
- 共同目标
- 相互尊重
- 充分沟通
- 优势互补
```

### 3.1 有效沟通原则

```
有效沟通四要素

1. 明确目的
   ┌─────────────────────┐
   │ 这次会议/沟通要    │
   │ 达成什么目标？      │
   │                     │
   │ - 通知？            │
   │ - 讨论？            │
   │ - 决策？            │
   │ - 解决问题？        │
   └─────────────────────┘

2. 了解受众
   ┌─────────────────────┐
   │ 对方的关注点是什么？│
   │                     │
   │ 产品：业务价值      │
   │ 设计：用户体验      │
   │ 技术：可行性、成本  │
   └─────────────────────┘

3. 选择方式
   ┌─────────────────────┐
   │ 紧急：当面/电话     │

   ┌─────────────────────┐

   └─────────────────────┘
```

### 3.2 跨职能沟通技巧

```
产品与设计师沟通

DO:
✓ 讲用户故事，不只功能
✓ 说明业务目标和约束
✓ 邀请设计师早期参与
✓ 尊重设计专业性
✓ 给出设计原则，不规定细节

DON'T:
✗ "照这个做"（不给理由）
✗ "这里改大一点"（不说原因）
✗ 频繁改动需求
✗ 忽视设计规范
✗ 最后一刻才找设计师

────────────────────────────

产品与开发沟通

DO:
✓ 说明为什么做（业务价值）
✓ 提供完整清晰的需求文档
✓ 尊重技术可行性意见
✓ 给出优先级
✓ 预留合理开发时间

DON'T:
✗ "这个很简单，很快就能做"
✗ 需求描述模糊
✗ 频繁变更需求
✗ 忽视技术债务
✗ 压缩测试时间

────────────────────────────

设计与开发沟通

DO:
✓ 提供完整设计稿和标注
✓ 说明交互细节
✓ 考虑技术实现
✓ 及时响应实现问题
✓ 保持设计一致性

DON'T:
✗ 设计稿不完整
✗ 交互说明不清
✗ 忽视技术限制
✗ 随意变更设计
✗ 不考虑边界情况
```

## 四、冲突解决

### 4.1 常见冲突类型

```
跨职能协作常见冲突

1. 优先级冲突
   ┌─────────────────────┐
   │ 产品：业务价值优先  │
   │ 技术：技术债务优先  │
   │ 设计：体验优化优先  │
   └─────────────────────┘

2. 时间冲突
   ┌─────────────────────┐
   │ 产品：越快越好      │
   │ 设计：需要打磨      │
   │ 技术：需要充分开发  │
   └─────────────────────┘

3. 方案冲突
   ┌─────────────────────┐
   │ 产品：功能完整      │
   │ 设计：简洁优雅      │
   │ 技术：简单可行      │
   └─────────────────────┘

4. 资源冲突
   ┌─────────────────────┐
   │ 多个项目抢资源      │
   │ 关键资源不足        │
   │ 依赖其他团队        │
   └─────────────────────┘

5. 认知冲突
   ┌─────────────────────┐
   │ 对需求理解不一致    │
   │ 对标准理解不同      │
   │ 信息不对称          │
   └─────────────────────┘
```

### 4.2 冲突解决方法

```
冲突解决五步法

步骤1: 识别冲突
┌─────────────────────┐
│ 是什么冲突？        │
│ 涉及哪些人？        │
│ 冲突程度如何？      │
└─────────────────────┘
         ↓
步骤2: 理解各方立场
┌─────────────────────┐
│ 各方的关注点是什么？│
│ 各方的约束是什么？  │
│ 各方的利益是什么？  │
└─────────────────────┘
         ↓
步骤3: 寻找共同目标
┌─────────────────────┐
│ 我们共同的目标是什么│
│ 有什么共识基础？    │
└─────────────────────┘
         ↓
步骤4: 探索解决方案
┌─────────────────────┐
│ 头脑风暴            │
│ 评估各方案          │
│ 寻找双赢方案        │
└─────────────────────┘
         ↓
步骤5: 达成共识
┌─────────────────────┐
│ 形成决议            │
│ 明确执行计划        │
│ 建立复盘机制        │
└─────────────────────┘
```

### 4.3 冲突解决技巧

```
冲突解决技巧

1. 关注问题，不针对人
   ┌─────────────────────┐
   │ 对事不对人          │
   │                     │
   │ 说"这个方案有风险"  │
   │ 不说"你不懂技术"    │

   ┌─────────────────────┐
   │ 先听对方说完        │
   │ 理解对方立场        │
   │ 复述确认理解        │
   └─────────────────────┘

3. 寻找数据支持
   ┌─────────────────────┐
   │ 用数据说话          │
   │ 用案例证明          │
   │ 减少主观判断        │
   └─────────────────────┘

4. 寻求第三方协调
   ┌─────────────────────┐
   │ 无法达成一致时      │
   │ 请上级或第三方协助  │

   └─────────────────────┘

   ┌─────────────────────┐
   │ 有时需要妥协        │

   │ 不是零和游戏        │
   └─────────────────────┘
```

### 4.4 冲突案例与解决

```
案例1: 需求变更冲突

情境：
产品在开发中途变更需求
开发抵制，认为影响进度

解决：
1. 理解原因：为什么变更？
2. 评估影响：需要多少额外时间？
3. 讨论方案：是否可以简化？下个版本？
4. 达成一致：这次做哪些，下个版本做哪些
5. 建立机制：如何减少未来变更

────────────────────────────

案例2: 时间压力冲突

情境：
产品要求1周上线
开发认为需要2周

解决：
1. 讨论必要性：为什么必须1周？
2. 评估范围：哪些功能必须有？
3. 调整方案：能否MVP先行？
4. 资源支持：能否加人或减少其他任务？
5. 明确风险：赶工的风险是什么？

────────────────────────────

案例3: 设计实现冲突

情境：
设计师设计复杂动效
开发认为实现成本高

解决：
1. 讨论价值：这个动效对用户的价值？
2. 评估成本：实现需要多久？
3. 寻找替代：有没有简化方案？
4. 优先级：是否这个版本必须？
5. 折中方案：保核心效果，简化细节
```

## 七、跨职能协作最佳实践

DO:
✓ 建立共同目标和语言
✓ 尊重彼此专业性
✓ 充分沟通，信息透明
✓ 建立清晰的协作流程
✓ 定期回顾和改进
✓ 用数据和事实说话

DON'T:
✗ 各自为政
✗ 不尊重专业意见
✗ 信息不透明
✗ 沟通不及时
✗ 需求频繁变更
✗ 推卸责任

记住：跨职能协作的核心是相互尊重、充分沟通、共同目标。成功的团队不是没有冲突，而是能高效解决冲突。
