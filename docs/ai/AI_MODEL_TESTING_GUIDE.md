| 健壮性 | 20分 | 边界处理 |
| 代码质量 | 20分 | 命名、注释、类型注解 |

【参考答案】
def find_duplicates(nums: list[int]) -> list[int]:
    """找出列表中所有重复的元素"""
    if not nums:
        return []
    seen = set()
    duplicates = set()
    for num in nums:
        if num in seen:
            duplicates.add(num)
        else:
            seen.add(num)
    return list(duplicates)
```

【参考答案】
∫(3x² + 2x - 1)dx

步骤1：将积分拆分
= ∫3x²dx + ∫2xdx - ∫1dx

步骤2：应用幂函数积分公式 ∫xⁿdx = xⁿ⁺¹/(n+1) + C
= 3 · x³/3 + 2 · x²/2 - x + C
= x³ + x² - x + C

最终答案：x³ + x² - x + C

### 1.6 思维链测试提示词

#### 测试集G：复杂推理

```
【TC-REASON-001】困难：多步骤推理

【输入】
问题：一个水池有两个进水管A和B，一个出水管C。
- 单开A管，6小时可以注满水池
- 单开B管，8小时可以注满水池
- 单开C管，12小时可以放完水池

现在水池是空的，如果三管同时打开，多少小时可以注满水池？

要求：请一步步思考，写出完整的推理过程。

【期望输出】
1. 分析每个管子的效率
2. 建立方程
3. 求解
4. 验证答案合理性

【评分标准】
| 维度 | 分值 | 标准 |
|------|------|------|
| 推理过程 | 40分 | 步骤清晰、逻辑正确 |
| 计算正确 | 30分 | 数值计算准确 |
| 答案正确 | 20分 | 最终答案正确 |
| 格式规范 | 10分 | 表达清晰 |

【参考答案】
步骤1：分析各管效率
- A管效率：1/6（池/小时）
- B管效率：1/8（池/小时）
- C管效率：-1/12（池/小时，负号表示出水）

步骤2：计算三管同时开的净效率
净效率 = 1/6 + 1/8 - 1/12
       = 4/24 + 3/24 - 2/24
       = 5/24（池/小时）

步骤3：计算注满时间
时间 = 1 ÷ (5/24) = 24/5 = 4.8小时 = 4小时48分钟

验证：4.8 × (1/6 + 1/8 - 1/12) = 4.8 × 5/24 = 1 ✓
```

│   └── 标准：≥99%
│
├── 错误率：各类错误的比例
│   ├── 超时错误：≤0.5%
│   ├── 格式错误：≤0.3%
│   └── 其他错误：≤0.2%
│
└── 结果一致性：相同输入多次测试的稳定性
    └── 标准：评分标准差≤5分

3. 安全性
├── 拒绝率：对有害请求的拒绝比例
│   └── 标准：100%（有害请求）
│
├── 误拒率：对正常请求的错误拒绝
│   └── 标准：≤1%
│
└── 信息泄露：敏感信息保护
    └── 标准：0次泄露
```

### 3.3 L2：输出质量指标

```
1. 正确性
┌─────────────────────────────────────────────────────┐
│  完全正确：答案完全正确，无任何错误                 │
│  部分正确：主要部分正确，有少量错误                 │
│  完全错误：答案完全错误或无法完成                   │
│                                                     │
│  计算公式：                                         │
│  正确率 = 完全正确数 / 总测试数 × 100%             │
│  部分正确率 = 部分正确数 / 总测试数 × 100%         │
└─────────────────────────────────────────────────────┘

2. 完整性
├── 要点覆盖率：回答包含的关键要点比例
│   └── 计算方式：实际要点数 / 应有要点数
│
├── 步骤完整性：推理/计算步骤的完整性
│   └── 计算方式：完整步骤数 / 必要步骤数
│
└── 格式完整性：输出格式是否符合要求
    └── 计算方式：符合项数 / 要求项数

3. 规范性
├── 格式规范：是否按要求的格式输出
│   └── 评分：0-10分
│
├── 语言规范：表达是否清晰、准确
│   └── 评分：0-10分
│
└── 代码规范：代码风格是否符合规范
    └── 评分：0-10分

4. 可用性
├── 直接可用率：输出可直接使用，无需修改
│   └── 标准：≥60%
│
├── 小改可用率：需要小幅修改后可用
│   └── 标准：≥30%
│
└── 不可用率：需要大幅重写
    └── 标准：≤10%
```

### 3.4 L3：业务价值指标

```
1. 效率提升
├── 时间节省：相比人工完成节省的时间
│   └── 计算方式：(人工时间 - AI辅助时间) / 人工时间
│
├── 返工率：需要重新生成或大幅修改的比例
│   └── 标准：≤20%
│
└── 一轮成功率：第一次生成就满足需求的比例
    └── 标准：≥50%

2. 质量提升
├── 错误减少率：相比人工的 错误减少
│   └── 计算方式：(人工错误率 - AI错误率) / 人工错误率
│
├── 规范提升：代码/文档规范性提升
│   └── 评分：1-5分提升
│
└── 覆盖度提升：考虑的边界情况增加
    └── 评分：1-5分提升

3. 用户满意度
├── 整体满意度：用户对输出的整体评价
│   └── 评分：1-5分，目标≥4.0
│
├── 推荐意愿：用户推荐给他人的意愿
│   └── NPS评分：-100到100，目标≥30
│
└── 持续使用率：用户持续使用的比例
    └── 7日留存率，目标≥60%
```

## 第五部分：实践建议

### 5.1 测试用例设计建议

```
1. 渐进式难度
   简单30% → 中等50% → 困难20%

2. 类型覆盖
   ├── 核心能力（60%）：代码生成、数学推理
   ├── 扩展能力（30%）：代码审查、文档生成
   └── 边界能力（10%）：极端情况、错误处理

3. 真实性
   ├── 使用真实项目中的问题
   ├── 避免过于学术化的题目
   └── 包含实际使用场景

4. 可量化
   ├── 每个用例有明确的评分标准
   ├── 避免主观性强的评价
   └── 便于自动化评估
```

### 5.2 常见陷阱

```
1. 测试用例泄露
   避免使用公开数据集，模型可能已学习

2. 评分主观性
   建立明确评分标准，多人独立评分

3. 单一维度评估
   多维度评估，不只看正确性

4. 忽略可用性
   正确但不可用的输出没有价值

5. 测试量不足
   每类至少10个用例，才有统计意义
```
