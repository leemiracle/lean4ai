# Prompt 设计完全指南

> 从定义到实战，系统掌握 Prompt 设计

---

## 第一部分：定义与作用

### 1.1 什么是 Prompt

```
Prompt = 你给LLM的指令/输入

本质：
  一种与AI沟通的"编程语言"
  一种将人类意图转化为机器可理解形式的表达方式

组成：
  指令 + 上下文 + 输入数据 + 输出格式
```

### 1.2 Prompt 的作用

| 作用 | 说明 | 示例 |
|------|------|------|
| 引导方向 | 告诉LLM做什么 | "翻译这段文字" |
| 提供上下文 | 给必要的背景信息 | "你是Python专家" |
| 约束范围 | 限制输出内容 | "不超过100字" |
| 格式规范 | 指定输出形式 | "用表格呈现" |
| 质量控制 | 提升输出质量 | "请一步步思考" |

### 1.3 好的 Prompt 的影响

```
差的 Prompt：
  "帮我写代码"
  → 输出模糊、可能跑偏、需要多次返工

好的 Prompt：
  "用Python写一个函数：输入整数列表，返回最大值。
   要求：O(n)复杂度，处理空列表情况，添加注释。"
  → 输出精准、一次成功、可直接使用

对比：
  - 返工次数：5+ 次 vs 0-1 次
  - 总耗时：30分钟 vs 5分钟
  - 可用性：需要大改 vs 直接可用
```

## 第二部分：设计的基本原则

### 2.1 清晰性原则

```
核心：让LLM无歧义地理解你的意图

检查方法：
  □ 每句话只有一种理解方式？
  □ 专业术语有解释或上下文？
  □ 代词指代明确？

示例：
❌ "把它改好一点"
✅ "将代码的时间复杂度从O(n²)优化到O(n)"
```

### 2.2 具体性原则

```
核心：提供足够的细节

5W1H检查：
  What - 具体要做什么？
  Why  - 为什么/目的是什么？
  Who  - 目标用户/受众是谁？
  When - 什么场景/时机？
  Where- 什么环境/平台？
  How  - 怎么做/什么标准？

示例：
❌ "写一个函数"
✅ "写一个Python函数，名为find_max，
   输入整数列表，返回最大值，
   处理空列表返回None，
   添加类型注解和docstring"
```

### 2.3 结构化原则

```
核心：用清晰的格式组织信息

推荐结构：
  【角色/背景】
  【任务】
  【约束】
  【示例】（可选）
  【格式】

示例：
【角色】你是Python专家，擅长性能优化
【任务】优化以下代码的执行速度
【代码】
[粘贴代码]
【约束】保持功能不变，优先优化时间复杂度
【格式】给出优化后的代码 + 改动说明
```

### 2.4 渐进性原则

```
核心：复杂任务分步完成

简单任务 → 一次完成
中等任务 → 问关键问题再执行
复杂任务 → 最小方案 → 确认 → 迭代

示例：
复杂任务不要一次问完：
❌ "设计一个完整的电商系统"

✅ 第一轮："我要设计一个电商系统，用户量100万。
          请先问我关键问题，确认理解后再给方案"
  
  第二轮：（回答问题后）"基于这些信息，给最小可行架构"
  
  第三轮："展开订单模块的设计"
```

### 2.5 可验证性原则

```
核心：让输出可以被检验

方法：
  - 指定输出格式
  - 要求给出依据
  - 要求标注置信度
  - 提供验证方法

示例：
❌ "这个答案对吗？"

✅ "请回答并标注：
   1. 置信度（1-10）
   2. 依据来源
   3. 如何验证"
```

## 第三部分：常见设计技巧

### 3.1 角色设定（Role Playing）

```
作用：激活特定领域的知识和表达风格

模板：
"你是[角色]，具有[背景/经验]。
擅长[能力1][能力2][能力3]。
请[任务]。"

示例：
"你是资深Python工程师，有10年后端开发经验。
擅长性能优化、代码重构、系统设计。
请帮我审查这段代码，重点关注性能和可维护性。"

效果对比：
无角色设定 → 通用回答，可能不够深入
有角色设定 → 专业视角，更有针对性
```

### 3.2 思维链（Chain of Thought）

```
作用：引导LLM逐步推理，提升复杂问题准确率

模板：
"请一步步思考：
1. 首先，[第一步]
2. 然后，[第二步]
3. 接着，[第三步]
4. 最后，[结论]"

简化版：
"请一步步思考，并写出推理过程"

研究支持：Wei et al., 2022 - 推理任务准确率提升30%+

示例：
❌ "25 * 4 + 10 = ?"

✅ "请一步步计算：25 * 4 + 10 = ?
   步骤1：先算25 * 4
   步骤2：再加10
   步骤3：得出答案"
```

### 3.3 Few-shot（少样本学习）

```
作用：通过示例让LLM理解期望的输出格式/风格

模板：
"按以下示例格式回答：

示例1：
输入：[...]
输出：[...]

示例2：
输入：[...]
输出：[...]

现在处理：
输入：[实际输入]
输出："

示例：
"按以下格式生成API文档：

示例：
函数：add(a, b)
功能：两数相加
参数：a(int), b(int)
返回：int，两数之和

现在为以下函数生成文档：
函数：find_max(numbers)
"
```

### 3.4 结构化输入

```
作用：让LLM快速定位关键信息

推荐格式：
【标签】内容

常用标签：
【任务】【背景】【约束】【输入】【输出】【示例】

示例：
【任务】代码审查
【语言】Python 3.10
【关注点】性能、安全
【代码】
def process(data):
    result = []
    for item in data:
        if item > 0:
            result.append(item * 2)
    return result
【要求】指出问题并给出改进建议
```

### 3.5 负面约束

```
作用：明确告诉LLM不要做什么

模板：
【必须】
- [要求1]
- [要求2]

【禁止】
- [限制1]
- [限制2]

示例：
【必须】
- 使用Python标准库
- 处理所有异常情况
- 添加类型注解

【禁止】
- 不使用第三方库
- 不使用全局变量
- 代码不超过50行
```

### 3.6 输出格式控制

```
作用：让输出符合你的使用需求

常用格式：
- 表格："用Markdown表格呈现"
- JSON："输出JSON格式"
- 列表："用编号列表"
- 步骤："分步骤输出"
- 代码："用代码块包裹"

示例：
❌ "比较Python和Java"

✅ "比较Python和Java，用表格呈现：
| 维度 | Python | Java |
|------|--------|------|
| 类型 | 动态 | 静态 |
| 性能 | ... | ... |
| 适用场景 | ... | ... |"
```

### 3.7 自我评估（Self-Evaluation）

```
作用：让LLM检查自己的输出

模板：
"回答后请评估：
1. 置信度（1-10）：?
2. 可能的问题：?
3. 需要验证的点：?"

示例：
"请解释什么是递归，并给出一个例子。
回答后评估你的解释是否清晰，例子是否恰当。"
```

### 3.8 多角度分析

```
作用：获得更全面的分析

模板：
"请从以下角度分析：
1. [角度A]：...
2. [角度B]：...
3. [角度C]：...
最后给出综合建议。"

示例：
"请从性能、安全、可维护性三个角度
分析这段代码，并给出改进建议。"
```

### 3.9 思维树（Tree of Thoughts）

```
作用：复杂问题的系统化探索

模板：
"请用思维树方法分析：
1. 生成3个可能方案
2. 每个方案推演2-3步
3. 评估每个方案
4. 选择最优方案"

示例：
问题
├── 方案A
│   ├── 步骤1 → 可行性：高
│   └── 步骤2 → 风险：中
├── 方案B
│   └── ...
└── 方案C
    └── ...

推荐：方案A
```

### 3.10 迭代精炼

```
作用：通过多轮对话逐步优化

模式：
v1 → 反馈 → v2 → 反馈 → v3 → ...

示例：
第一轮："写一个登录功能的代码"
第二轮："增加验证码功能"
第三轮："优化错误处理"

要点：
- 每次只改一小部分
- 基于上一版本迭代
- 明确指出要改什么
```

## 第四部分：编写的艺术

### 4.1 构成要素

```
完整的Prompt包含以下要素（按需组合）：

┌─────────────────────────────────────────┐
│  1. 角色设定（谁在做）                  │
│     "你是XX专家"                        │
├─────────────────────────────────────────┤
│  2. 任务描述（做什么）                  │
│     "请帮我完成XX"                      │
├─────────────────────────────────────────┤
│  3. 背景上下文（为什么/什么情况）       │
│     "背景是...，目标是..."              │
├─────────────────────────────────────────┤
│  4. 输入数据（处理什么）                │
│     "以下是代码/文本/数据"              │
├─────────────────────────────────────────┤
│  5. 约束条件（限制）                    │
│     "必须...，禁止..."                  │
├─────────────────────────────────────────┤
│  6. 输出格式（结果形式）                │
│     "用表格/JSON/步骤呈现"              │
├─────────────────────────────────────────┤
│  7. 示例（可选）                        │
│     "参考以下格式..."                   │
├─────────────────────────────────────────┤
│  8. 思维引导（可选）                    │
│     "请一步步思考..."                   │
└─────────────────────────────────────────┘

最简组合：任务 + 输入
推荐组合：角色 + 任务 + 背景 + 约束 + 格式
完整组合：全部要素
```

### 4.2 编写技巧

**技巧一：先想清楚再写**

```
写Prompt前问自己：
1. 我最终要什么？
2. LLM需要知道什么才能做好？
3. 怎样表达最清晰？
4. 输出要什么格式？
5. 如何判断结果好坏？
```

**技巧二：从简单开始，逐步完善**

```
步骤：
1. 先写最简版本，测试效果
2. 看哪里不满意
3. 针对性补充信息/约束
4. 再次测试
5. 重复直到满意

不要：一开始就想写完美
```

**技巧三：用模板提高效率**

```
建立自己的模板库：
- 日常任务模板
- 代码相关模板
- 写作相关模板
- 分析相关模板

使用时：填空 + 微调
```

**技巧四：记录有效模式**

```
建立"最佳实践"笔记：
- 什么问法效果好？
- 什么技巧解决了什么问题？
- 哪些模板反复有用？

定期回顾和优化
```

**技巧五：针对问题调整**

```
常见问题及调整：
- 答案太泛 → 加具体约束
- 理解偏差 → 加背景说明
- 格式不对 → 明确格式要求
- 深度不够 → 加角色设定
- 逻辑错误 → 加思维链
```

## 第五部分：常见问题判断与解决方案

### 5.1 问题诊断框架

```
当Prompt效果不好时，按以下顺序诊断：

Step 1: 检查理解
  → LLM是否理解了我的意图？
  → 症状：答非所问、偏离主题
  → 解决：增加背景说明、明确任务

Step 2: 检查信息
  → 是否给了足够的信息？
  → 症状：输出太泛、不够具体
  → 解决：补充上下文、数据、约束

Step 3: 检查格式
  → 是否明确了输出格式？
  → 症状：格式混乱、难以使用
  → 解决：指定格式要求、给示例

Step 4: 检查深度
  → 是否需要更深度的分析？
  → 症状：回答太浅、不专业
  → 解决：加角色设定、思维链

Step 5: 检查边界
  → 是否超出了LLM能力范围？
  → 症状：编造事实、明显错误
  → 解决：换方法、验证结果
```

### 5.2 编程类问题

#### 问题类型一：代码生成错误

```
【症状】生成的代码有语法错误或逻辑错误

【原因分析】
- 描述不够具体
- 缺少边界情况说明
- 语言/版本信息缺失

【解决方案】
1. 明确技术栈
   ❌ "写一个排序函数"
   ✅ "用Python 3.10写一个快速排序函数"

2. 说明边界情况
   ✅ "处理空列表、None、重复元素"

3. 提供示例输入输出
   ✅ "示例：输入[3,1,2] → 输出[1,2,3]"

4. 要求解释逻辑
   ✅ "添加注释解释关键步骤"
```

#### 问题类型二：代码理解偏差

```
【症状】生成的代码功能不符合预期

【原因分析】
- 需求描述有歧义
- 缺少上下文

【解决方案】
1. 拆解复杂需求
   ❌ "写一个完整的用户管理系统"
   ✅ "先写用户注册功能，包含：输入验证、密码加密、数据存储"

2. 提供上下文
   ✅ "这是一个电商系统的用户模块，需要与订单模块对接"

3. 明确非功能需求
   ✅ "需要处理并发、考虑性能、方便测试"
```

#### 问题类型三：代码优化方向错误

```
【症状】优化的方向不是想要的

【原因分析】
- 没有明确优化目标
- 没有说明约束条件

【解决方案】
1. 明确优化目标
   ❌ "优化这段代码"
   ✅ "优化这段代码的执行速度"
   ✅ "优化这段代码的内存使用"

2. 说明约束
   ✅ "保持功能不变"
   ✅ "不引入新的依赖"
   ✅ "代码可读性不能降低太多"

3. 量化目标
   ✅ "目标是将执行时间从10秒降到1秒以内"
```

#### 问题类型四：调试帮助无效

```
【症状】给的调试建议不靠谱

【原因分析】
- 没给足够的环境信息
- 错误信息不完整
- 代码片段不完整

【解决方案】
1. 提供完整信息
   ✅ "环境：Python 3.10, Windows 11"
   ✅ "完整错误信息：[粘贴]"
   ✅ "相关代码：[粘贴完整函数]"

2. 说明已尝试的方法
   ✅ "已尝试：打印调试、查文档、搜索Stack Overflow"

3. 提供复现步骤
   ✅ "复现步骤：1. 运行test.py 2. 输入abc 3. 报错"
```

### 5.3 数学类问题

#### 问题类型一：概念解释不清

```
【症状】解释太抽象，听不懂

【原因分析】
- 没有指定受众水平
- 没有要求举例

【解决方案】
1. 指定水平
   ❌ "解释什么是积分"
   ✅ "用高中生能懂的语言解释什么是积分"

2. 要求举例
   ✅ "配合2-3个具体例子说明"

3. 要求类比
   ✅ "用生活中的例子类比"

4. 分层解释
   ✅ "先给一句话总结，再详细展开"
```

#### 问题类型二：计算过程跳步

```
【症状】计算过程省略太多，跟不上

【原因分析】
- 没有要求详细步骤

【解决方案】
1. 要求思维链
   ✅ "请一步步计算，写出每一步"

2. 指定详细程度
   ✅ "每一步都要写出，不要跳步"

3. 要求解释
   ✅ "每一步解释为什么这样做"

示例：
❌ "计算 ∫(2x+1)dx"

✅ "请一步步计算 ∫(2x+1)dx：
   步骤1：识别被积函数
   步骤2：应用积分公式
   步骤3：计算每一项
   步骤4：写出最终结果
   每步解释依据"
```

#### 问题类型三：证明逻辑混乱

```
【症状】证明过程逻辑不清晰

【原因分析】
- 没有要求结构化输出
- 没有指定证明方法

【解决方案】
1. 指定证明方法
   ✅ "用数学归纳法证明"
   ✅ "用反证法证明"

2. 要求结构化
   ✅ "按以下结构输出：
      ## 已知
      ## 求证
      ## 证明过程
      ## 结论"

3. 要求标注每步依据
   ✅ "每一步标注依据的定理或性质"
```

#### 问题类型四：应用题不会转化

```
【症状】无法将实际问题转化为数学问题

【原因分析】
- 缺少建模指导
- 没有要求分析过程

【解决方案】
1. 要求建模过程
   ✅ "请按以下步骤：
      1. 识别变量
      2. 建立关系
      3. 列出方程
      4. 求解
      5. 验证答案是否符合实际"

2. 要求画图/列表
   ✅ "用表格整理已知量和未知量"

3. 要求分析单位
   ✅ "标注每个量的单位，检查量纲"
```

### 5.4 写作类问题

#### 问题类型一：风格不符合

```
【症状】写作风格与期望不符

【解决方案】
1. 给参考样例
   ✅ "参考以下风格：[粘贴样例]"

2. 描述目标风格
   ✅ "风格：正式但不生硬，专业但易懂"

3. 指定受众
   ✅ "读者是：小学生/专家/普通大众"

4. 给正面和反面例子
   ✅ "像XX那样写，不要像YY那样写"
```

#### 问题类型二：内容太泛

```
【症状】内容空洞，没有实质信息

【解决方案】
1. 要求具体化
   ✅ "每个观点用具体例子支撑"

2. 要求数据
   ✅ "引用具体数据或案例"

3. 限制范围
   ✅ "只讨论XX方面，深度展开"
```

#### 问题类型三：结构混乱

```
【症状】文章结构不清晰

【解决方案】
1. 指定结构
   ✅ "按以下结构：
      1. 引言（背景+论点）
      2. 正文（3个分论点）
      3. 结论（总结+展望）"

2. 要求大纲
   ✅ "先给出大纲，确认后再展开"

3. 要求过渡
   ✅ "段落之间要有过渡句"
```
