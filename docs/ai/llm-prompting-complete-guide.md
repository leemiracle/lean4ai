## 第一部分：理解LLM

### 1.1 知识本质

```
LLM知识 = 训练数据的压缩 + 统计模式学习

不是：真正理解、数据库存储、有意识思考
是：  模式提取、上下文激活、概率预测
```

### 1.2 可靠性速查

| 类型 | 可靠性 | 验证策略 |
|------|--------|----------|
| 语言/常识 | ★★★ | 直接信任 |
| 专业知识 | ★★☆ | 交叉验证 |
| 时效/冷门 | ★☆☆ | 必须验证 |

### 1.3 能力边界

**擅长：** 语言任务、代码模式、概念解释、创意组合
**需验证：** 专业领域、精确数据、时效信息
**不擅长：** 最新信息、真正推理、实际验证

### 2.2 职业使用原则

```
高价值场景：
  ✓ 高频重复性工作（文档、邮件、报告）
  ✓ 创意发散（头脑风暴、方案设计）
  ✓ 学习加速（概念理解、代码示例）
  ✓ 质量提升（审查、润色、优化）

需谨慎场景：
  ⚠ 专业判断（医疗、法律、金融决策）
  ⚠ 精确数据（财务数据、法律条款）
  ⚠ 敏感信息（客户隐私、商业机密）

不适合场景：
  ✗ 实时操作（系统运维、交易执行）
  ✗ 物理任务（制造、医疗操作）
  ✗ 最终决策（战略、人事、投资）
```

#### 实践案例：写一个复杂SQL

```
【场景】
需要统计每个用户在过去30天的购买行为，计算RFM指标

【错误做法】
"帮我写一个RFM分析的SQL"

【正确做法】
【数据库】MySQL 8.0
【表结构】
```sql
-- 用户表
users(id, name, created_at)
-- 订单表
orders(id, user_id, amount, created_at, status)
```
【需求】计算每个用户的RFM指标：
- R：最近一次购买距今天数
- F：30天内购买次数
- M：30天内购买金额
- 需要分5档打分

【约束】
- 只统计已完成的订单（status='completed'）
- 时间范围：过去30天
- 用户量：100万，订单量：500万

请：
1. 写出完整SQL
2. 解释关键逻辑
3. 给出优化建议（如果查询慢）

【LLM输出】
```sql
WITH user_rfm AS (
  SELECT 
    u.id as user_id,
    u.name,
    DATEDIFF(CURDATE(), MAX(o.created_at)) as recency,
    COUNT(o.id) as frequency,
    SUM(o.amount) as monetary
  FROM users u
  LEFT JOIN orders o ON u.id = o.user_id 
    AND o.status = 'completed'
    AND o.created_at >= DATE_SUB(CURDATE(), INTERVAL 30 DAY)
  GROUP BY u.id, u.name
),
rfm_score AS (
  SELECT 
    *,
    NTILE(5) OVER (ORDER BY recency DESC) as r_score,
    NTILE(5) OVER (ORDER BY frequency ASC) as f_score,
    NTILE(5) OVER (ORDER BY monetary ASC) as m_score
  FROM user_rfm
)
SELECT * FROM rfm_score;
```
优化建议：
1. 确保created_at有索引
2. 考虑分区表...
```

正确思路：我遇到什么问题？LLM能帮哪部分？

分析框架：
1. 我的最终目标是什么？
2. 当前卡在哪里？
3. 哪些部分我能做？
4. 哪些部分LLM能帮？
5. 哪些部分需要其他资源？
```

### 4.2 思维方式

**结构化思维**

```
金字塔原理：
  结论 → 分论点 → 证据

应用：
  提问时：我要[结论]，因为[分论点]，基于[证据]
  理解时：核心结论是什么？支撑理由？下一步行动？
```

**第一性原理**

```
回归本质：
  不是"别人怎么做"而是"本质是什么"

模板：
"这件事的基本约束是什么？
 从约束出发，最优解是什么？"
```

**系统思维**

```
看整体：
  上游（依赖什么）→ 本身 → 下游（影响什么）

模板：
"这个问题在系统中的位置是什么？
 有哪些连锁反应？
 如何系统性解决？"
```

### 4.3 统筹能力

**资源整合**

```
你的工具箱：
  LLM     → 快速生成、解释、brainstorm
  搜索    → 最新信息、事实查证
  文档    → 权威来源、深度知识
  人      → 经验判断、复杂决策
  实践    → 真实验证、技能内化

原则：让每个工具做最擅长的事
```

**任务分解**

```
目标 → 阶段 → 任务 → 步骤 → 工具

例：做一个数据项目
  目标：分析用户流失原因
  阶段：数据准备→分析→报告
  任务：数据清洗→特征分析→可视化
  步骤：写SQL→跑数据→画图
  工具：LLM（SQL）、Python（分析）、LLM（报告框架）
```

### 4.4 数据导向

**效果指标**

| 指标 | 定义 | 好的标准 |
|------|------|----------|
| 返工次数 | 需要修改几次 | ≤1 |
| 一次理解率 | LLM第一次理解对 | ≥80% |
| 总耗时 | 提问到满意 | 减少50%+ |
| 可用率 | 答案直接可用 | ≥70% |

**A/B测试**

```
同一问题，两种问法，对比效果：
  - 理解速度
  - 输出质量
  - 返工次数
  - 总耗时
```

### 4.5 反馈迭代

```
迭代循环：提问 → 输出 → 评价 → 改进

反馈类型：
  - 方向反馈："不是我要的方向"
  - 细节反馈："第二点不够具体"
  - 格式反馈："用表格呈现"
  - 补充反馈："还需要考虑XX"

迭代次数：
  简单1轮 | 中等2-3轮 | 复杂3-5轮
  
超过5轮：重新定义问题或换工具
```
