## 一、基础概念

### LLM (Large Language Model / 大语言模型)

**一句话：** 读过海量文字的AI，能理解和生成人类语言。

**详细解释：**
```
想象一个读过互联网上几乎所有文字的人。
你给它一段话，它能猜出下一句是什么。
你问它问题，它能根据读过的内容回答。

LLM就是这个"人"，但它不是真正理解，
而是通过统计规律来"猜"。

核心能力：
├── 理解：看懂你说的意思
├── 生成：写出连贯的文字
└── 推理：根据已知推导未知
```

**例子：**
```
输入："天空是"
LLM可能输出："蓝色的"、"晴朗的"、"广阔的"

为什么？因为它读过很多"天空是XX"的句子，
知道哪些词最可能出现在后面。
```

---

### Token (词元)

**一句话：** LLM处理文字的最小单位，约等于0.75个英文单词。

**详细解释：**
```
LLM不是按"字"处理文字，而是按"块"处理。
这个"块"就是Token。

英文：1个单词 ≈ 1-2个Token
中文：1个汉字 ≈ 1-2个Token

LLM有"记忆限制"，这个限制就是按Token算的。
比如GPT-4能记住约128000个Token。
```

**例子：**
```
句子："Hello world"

分Token：
"Hello" → Token 1
" world" → Token 2

共2个Token
```

---

### Context Window (上下文窗口)

**一句话：** LLM一次能"记住"的文字量上限。

**详细解释：**
```
想象你在考试，但只能看一次题目。
题目太长，你看到后面就忘了前面。
LLM也一样，它有一个"记忆容量"。

这个容量就是Context Window。
超过这个容量，早期的内容就会被"忘记"。

各模型Context Window：
├── GPT-3.5: 4K tokens
├── GPT-4: 8K/32K tokens
├── GPT-4 Turbo: 128K tokens
├── Claude 3: 200K tokens
└── Gemini 1.5 Pro: 1M tokens
```

**例子：**
```
你给LLM一本10万字的书，问它第一章讲什么。

如果Context Window只有4K tokens（约3万字）：
它会"忘记"开头的内容，答不出来。

如果Context Window是128K tokens（约10万字）：
它能"记住"整本书，回答正确。
```

### Temperature (温度)

**一句话：** 控制LLM输出"随机性"的参数，0最确定，1最随机。

**详细解释：**
```
LLM生成文字时，每个词都有一个概率。
Temperature决定它是选"最可能的词"，还是"随机选词"。

Temperature = 0：
├── 总是选最可能的词
├── 输出最确定、最一致
└── 适合：代码、数学、事实问答

Temperature = 1：
├── 按概率随机选词
├── 输出多样、有创意
└── 适合：创意写作、头脑风暴

Temperature = 0.7（常用）：
├── 平衡确定性和多样性
└── 适合：日常对话
```

**例子：**
```
输入："天空是"

Temperature = 0：
输出："蓝色的"（最可能）

Temperature = 1：
输出："一片画布"、"做梦的地方"、"蓝色的"（随机）
```

### Embedding (嵌入/向量化)

**一句话：** 把文字变成数字，让电脑能理解文字的"意思"。

**详细解释：**
```
电脑不懂文字，只懂数字。
Embedding就是把文字变成一串数字（向量）。

关键点：意思相近的词，数字也相近。

比如：
"猫" → [0.2, 0.8, 0.1]
"狗" → [0.3, 0.7, 0.2]  （和猫相近）
"汽车" → [0.9, 0.1, 0.8] （和猫相差远）

这让我们可以：
├── 搜索相似内容
├── 分类文本
└── 构建知识库（RAG）
```

**例子：**
```
搜索："可爱的宠物"

传统搜索：找包含"可爱"和"宠物"的文章
Embedding搜索：找意思相近的文章，即使没有这些词

结果可能包含：
├── "我家的猫咪特别萌"
├── "小狗是最好的伙伴"
└── "兔子的眼睛像红宝石"
```

---

### Prompt (提示词)

**一句话：** 你给LLM的指令，告诉它要做什么。

**详细解释：**
```
Prompt就是你跟LLM说的话。
说得好，LLM就做得好。
说得差，LLM就乱来。

好Prompt = 说清楚要什么 + 给足够的背景 + 告诉格式

就像点菜：
├── 差："我要吃的"
├── 中："我要一份面"
└── 好："我要一份牛肉拉面，不要香菜，辣一点，大碗"
```

**例子：**
```
差Prompt：
"写代码"

好Prompt：
"用Python写一个函数：
├── 函数名：calculate_average
├── 输入：数字列表
├── 输出：平均值
├── 要求：处理空列表，添加注释"
```

## 二、模型架构

### Transformer (变换器)

**一句话：** 现代LLM的核心"引擎"，让AI能理解长句子的意思。

**详细解释：**
```
Transformer是一种神经网络结构，2017年发明。
它让AI能"同时"看整句话，而不是一个字一个字看。

核心创新：注意力机制（Attention）

以前：读"我喜欢苹果"，先看"我"，再看"喜欢"...
现在：一次看完，并知道每个词和其他词的关系

这就像：
├── 以前：用放大镜看书，一次看一个字
└── 现在：用眼睛看书，一眼看一行
```

**例子：**
```
句子："苹果是一家公司，我也喜欢吃苹果。"

Transformer能理解：
├── 第一个"苹果"→ 公司（和"公司"关联）
└── 第二个"苹果"→ 水果（和"喜欢"关联）
```

---

### Attention Mechanism (注意力机制)

**一句话：** 让LLM知道哪些词重要、哪些词相关的技术。

**详细解释：**
```
读句子时，我们不是每个词都同等关注。
"我喜欢红色的苹果"
关注"喜欢"和"苹果"，"的"不太重要。

Attention就是让AI学会"关注重点"。

具体做法：
├── 每个词和其他所有词计算"关联度"
├── 关联度高的，多"注意"
└── 关联度低的，少"注意"

就像看书时：
├── 重要的句子画线
├── 关键的词圈出来
└── 不重要的扫过
```

**例子：**
```
句子："小明把苹果给了小红，因为她饿了。"

Attention会计算：
├── "她"和"小红"关联度高 → "她"="小红"
├── "因为"和"饿了"关联度高 → 饿是原因
└── "小明"和"给"关联度高 → 小明是动作发出者
```

### Pre-training (预训练)

**一句话：** 让LLM"读"海量文字，学习语言的基本规律。

**详细解释：**
```
预训练就是"大量阅读"阶段。

给LLM看数万亿字的文字，让它学习：
├── 哪些词经常一起出现
├── 句子是怎么组织的
├── 不同语境下词的意思
└── 各种知识

这个阶段：
├── 耗时：数周到数月
├── 成本：数百万美元
├── 数据：互联网、书籍、论文等
└── 目标：学会"说话"和"常识"

预训练后的模型：
├── 能接话（预测下一个词）
├── 但不一定听话（不懂指令）
└── 需要进一步训练
```

**例子：**
```
预训练数据：
├── 维基百科（所有词条）
├── 书籍（数百万本）
├── 网页（Common Crawl）
├── 代码（GitHub）
└── 论文

学习结果：
输入："法国的首都是"
输出："巴黎"（因为它"读过"）
```

### Fine-tuning (微调)

**一句话：** 在预训练基础上，用特定数据让LLM更擅长某个任务。

**详细解释：**
```
预训练后的LLM像个"通才"，
什么都知道一点，但不专精。

Fine-tuning就是"专业训练"：
├── 用特定领域的数据
├── 调整模型参数
└── 让它在该领域更强

比如：
├── 医疗微调：用医学文献训练 → 更擅长医疗问答
├── 代码微调：用代码数据训练 → 更擅长编程
└── 对话微调：用对话数据训练 → 更擅长聊天

类比：
├── 预训练：读完所有书的大学生
└── 微调：专门培训某项技能
```

**例子：**
```
通用LLM：
问："写一个排序函数"
答：给出一个基础实现

代码微调后的LLM：
问："写一个排序函数"
答：给出优化实现 + 复杂度分析 + 测试用例
```

### RLHF (Reinforcement Learning from Human Feedback)

**一句话：** 让人类教LLM什么是"好回答"，让输出更符合人类期望。

**详细解释：**
```
问题：LLM可能说出技术正确但不好的话
比如：有偏见、不安全、不礼貌

RLHF的做法：
Step 1: 让LLM生成多个回答
Step 2: 人类排序哪个回答更好
Step 3: 训练一个"打分模型"
Step 4: 用这个模型调整LLM

结果：
├── 回答更安全
├── 回答更符合人类期望
├── 减少有害输出
└── 更"听话"

类比：
├── 预训练：自己看书学习
├── 微调：老师教特定知识
└── RLHF：老师告诉你什么是"好学生"
```

**例子：**
```
问题："如何制作炸弹？"

预训练模型：
可能详细回答（因为它"读过"）

RLHF后的模型：
"我不能提供制作危险物品的信息。
如果您对化学感兴趣，我可以推荐一些
安全的学习资源。"
```

---

### SFT (Supervised Fine-Tuning)

**一句话：** 用"问题-答案"配对数据，教LLM如何回答问题。

**详细解释：**
```
SFT是最简单直接的微调方式。

准备数据：
├── 输入：用户问题
├── 输出：理想回答
└── 大量这样的配对

让LLM学习：
├── 看到这样的问题
├── 应该这样回答
└── 调整参数

常用于：
├── 指令跟随（回答问题）
├── 特定任务（翻译、摘要）
└── 特定风格（正式/口语）
```

**例子：**
```
训练数据：
输入："什么是Python？"
输出："Python是一种高级编程语言，
       以简洁易读著称，广泛用于..."

训练后：
问同样问题 → 给出类似的好回答
```

## 三、高级技术

### RAG (Retrieval-Augmented Generation / 检索增强生成)

**一句话：** 让LLM在回答前先"查资料"，提高准确性。

**详细解释：**
```
LLM的问题：
├── 知识有截止日期
├── 可能"编造"答案
└── 不懂你的私有信息

RAG的做法：
Step 1: 把你的文档存入向量数据库
Step 2: 用户提问时，先检索相关文档
Step 3: 把相关文档 + 问题一起给LLM
Step 4: LLM基于文档回答

类比：
├── 普通LLM：闭卷考试
└── RAG：开卷考试，可以翻书

优点：
├── 答案更准确
├── 可以引用来源
├── 知识可更新
└── 能用私有数据
```

**例子：**
```
问题："公司2024年的营收是多少？"

普通LLM：
"我不知道，我的知识截止到2023年"

RAG：
1. 检索公司财报文档
2. 找到相关段落："2024年营收1.2亿"
3. 回答："根据公司财报，2024年营收为1.2亿元"
```

### Agent (智能体)

**一句话：** 能自主规划、使用工具、完成复杂任务的AI系统。

**详细解释：**
```
普通LLM：你问它答，一次一个回合
Agent：你给目标，它自己想办法完成

Agent的能力：
├── 规划：把大任务拆成小任务
├── 工具使用：能调用API、执行代码
├── 记忆：记住之前的交互
├── 反思：判断结果好不好
└── 迭代：做得不好就重试

工作流程：
1. 接收目标
2. 制定计划
3. 执行动作
4. 观察结果
5. 调整策略
6. 继续执行
7. 完成目标

类比：
├── LLM：咨询顾问（只给建议）
└── Agent：项目经理（负责执行）
```

**例子：**
```
任务："帮我分析竞品的定价策略"

Agent执行：
1. 搜索竞品信息
2. 提取价格数据
3. 整理成表格
4. 分析定价模式
5. 生成分析报告
6. 给出建议

全程自动，不需要你一步步指导
```

### Chain-of-Thought (CoT / 思维链)

**一句话：** 让LLM"一步步想"，提高复杂问题的准确率。

**详细解释：**
```
问题：直接问答案，容易出错

CoT的做法：
让LLM把思考过程写出来，一步一步推理

为什么有效：
├── 每一步都是简单推理
├── 降低单步难度
├── 错误容易发现
└── 准确率大幅提升

效果：
├── 数学题：准确率提升30%+
├── 逻辑推理：准确率提升25%+
└── 复杂问题：效果明显
```

**例子：**
```
问题："小明有5个苹果，给了小红2个，
      又买了3个，现在有几个？"

不用CoT：
"6个"（可能算错）

用CoT：
"让我一步步思考：
1. 小明开始有5个苹果
2. 给了小红2个，剩下5-2=3个
3. 又买了3个，现在有3+3=6个
答案：6个"

准确率更高
```

### Few-shot Learning (少样本学习)

**一句话：** 给LLM几个例子，它就能学会怎么做。

**详细解释：**
```
让LLM学会新任务，不需要重新训练。
只需要在Prompt里给几个例子。

做法：
├── 在Prompt中放2-5个示例
├── 示例格式：输入 + 期望输出
├── 然后给出新输入
└── LLM会模仿示例输出

为什么有效：
├── LLM在预训练时学过"找规律"
├── 示例让它理解你要什么格式
└── 模仿比从零开始容易
```

**例子：**
```
Prompt：
"把情感分类：

示例1：
句子：'这部电影太棒了！'
情感：正面

示例2：
句子：'太失望了，浪费钱'
情感：负面

现在分类：
句子：'服务态度很差'
情感："

LLM输出：
"负面"
```

### Function Calling (函数调用)

**一句话：** 让LLM能调用外部API，获取实时信息或执行操作。

**详细解释：**
```
LLM本身：
├── 不能上网搜索
├── 不能查数据库
└── 不能执行代码

Function Calling让它能：
├── 调用你定义的函数/API
├── 获取实时数据
└── 执行实际操作

工作流程：
1. 你定义可用函数（名称、参数、描述）
2. 用户提问
3. LLM判断需要调用哪个函数
4. 输出函数调用指令
5. 你的系统执行函数
6. 把结果返回给LLM
7. LLM生成最终回答
```

**例子：**
```
定义函数：
├── get_weather(city) → 获取天气
├── send_email(to, subject, body) → 发邮件

用户："北京今天天气怎么样？"

LLM判断需要调用get_weather
输出：{"function": "get_weather", "args": {"city": "北京"}}

系统执行函数，返回："晴天，25°C"

LLM最终回答："北京今天是晴天，气温25°C，适合出行。"
```

### LoRA (Low-Rank Adaptation)

**一句话：** 一种高效微调方法，用很少的参数就能定制LLM。

**详细解释：**
```
问题：微调整个LLM需要很多算力和存储

LoRA的解决：
├── 不动原有的参数
├── 只训练少量额外参数（约1%）
├── 效果接近全量微调
└── 存储和计算成本低很多

类比：
├── 全量微调：整本书重写
└── LoRA：只加几页注释

优点：
├── 微调成本低（约1/10）
├── 可以训练多个版本
├── 切换方便
└── 效果损失小
```

**例子：**
```
场景：想让LLM懂医疗知识

全量微调：
├── 需要数十万美元
├── 需要数十张GPU
└── 微调后的模型很大

LoRA：
├── 只需要几千美元
├── 只需要几张GPU
└── 额外参数只有几十MB
```

### Quantization (量化)

**一句话：** 压缩模型，用更少的内存运行更大的模型。

**详细解释：**
```
模型参数通常是32位或16位浮点数。
量化就是用更少的位数表示。

常见量化：
├── FP16：16位（半精度）
├── INT8：8位（整数）
├── INT4：4位
└── INT2：2位（极限压缩）

效果：
├── 模型体积减小
├── 内存占用降低
├── 推理速度提升
└── 精度略有下降

例子：
70B参数的模型
├── FP16：需要140GB内存
├── INT8：需要70GB内存
└── INT4：需要35GB内存
```

**例子：**
```
在笔记本电脑上运行大模型：

不量化：需要专业服务器
量化（INT4）：普通笔记本就能跑
```

### Multimodal (多模态)

**一句话：** 能同时理解文字、图片、视频、音频等多种内容的LLM。

**详细解释：**
```
传统LLM：只能处理文字
多模态LLM：能处理多种"模态"

支持的模态：
├── 文字（Text）
├── 图片（Image）
├── 视频（Video）
├── 音频（Audio）
└── 文档（PDF、Word等）

能力：
├── 看图说话：描述图片内容
├── 图文理解：回答关于图片的问题
├── 视频分析：总结视频内容
├── 语音识别+理解：听懂说话
└── 生成图片：根据文字画图（部分模型）
```

**例子：**
```
输入图片 + 文字：
[一张表格图片]
"帮我把这个表格转成Markdown格式"

多模态LLM输出：
| 名称 | 价格 | 数量 |
|------|------|------|
| 苹果 | 5元  | 10   |
| 香蕉 | 3元  | 20   |
```

## 四、应用技术

### Prompt Engineering (提示工程)

**一句话：** 设计好的Prompt，让LLM输出更好结果的技术。

**详细解释：**
```
Prompt就是给LLM的指令。
Prompt好，输出就好。
Prompt差，输出就差。

Prompt工程的核心技巧：
├── 角色设定："你是XX专家"
├── 具体明确：说清楚要什么
├── 给示例：让LLM模仿
├── 分步骤：复杂任务拆解
├── 格式要求：指定输出格式
└── 负面约束：告诉它不要做什么

类比：
├── 差Prompt：模糊的命令
└── 好Prompt：详细的任务书
```

**例子：**
```
差Prompt：
"写个函数"

好Prompt：
"【角色】你是Python专家
【任务】写一个排序函数
【要求】
- 输入：整数列表
- 输出：升序排列
- 时间复杂度：O(n log n)
- 添加注释
【格式】代码块 + 解释"
```

---

### Context Caching (上下文缓存)

**一句话：** 记住之前处理过的内容，避免重复计算，省钱省时。

**详细解释：**
```
问题：
├── 长Prompt每次都要处理
├── 重复计算浪费钱和时间
└── API按Token收费，很贵

Context Cache的做法：
├── 把处理过的内容缓存起来
├── 下次遇到相同内容，直接用缓存
├── 只计算新增的部分
└── 节省时间和费用

适用场景：
├── 系统提示词（不变的）
├── 长文档（RAG中的知识库）
├── 代码上下文
└── 多轮对话的历史
```

**例子：**
```
没有缓存：
每次对话都把10000字的文档重新处理
成本：每次10000 tokens

有缓存：
第一次：处理10000 tokens
之后：只处理新增的几十tokens
成本：第一次10000，之后约100

节省：90%+
```

---

### Streaming (流式输出)

**一句话：** LLM边生成边显示，不用等全部生成完才看到。

**详细解释：**
```
传统方式：
├── LLM生成完整回答
├── 全部生成完
└── 一次性返回

流式输出：
├── 生成一点，显示一点
├── 用户马上看到反馈
└── 体验更流畅

类比：
├── 传统：下载完整个视频才看
└── 流式：边下载边看
```

**例子：**
```
用户问："什么是Python？"

传统方式：
等待3秒...
然后一次性显示全部回答

流式输出：
第0.5秒："Python"
第0.6秒："Python是一种"
第0.7秒："Python是一种高级"
...
用户能马上看到内容
```

---

### Rate Limiting (速率限制)

**一句话：** API调用次数/频率的限制，防止滥用。

**详细解释：**
```
LLM API通常有限制：

按时间：
├── 每分钟请求数（RPM）
├── 每分钟Token数（TPM）
└── 每天请求数

按金额：
├── 每月消费上限
└── 预付费余额

常见限制：
├── GPT-3.5: 3500 RPM
├── GPT-4: 500 RPM
└── Claude: 取决于套餐

超限怎么办：
├── 等待（自动恢复）
├── 升级套餐
└── 多账号分担
```

**例子：**
```
你的应用每秒发送100个请求
但API限制每分钟60个

结果：
├── 前60个成功
├── 后40个失败（429错误）
└── 需要等待或降低频率
```

---

### JSON Mode (JSON模式)

**一句话：** 强制LLM输出合法的JSON格式，方便程序处理。

**详细解释：**
```
问题：
├── LLM可能输出不规范格式
├── 程序解析会出错
└── 需要额外处理

JSON Mode：
├── 告诉LLM必须输出JSON
├── LLM保证格式正确
└── 可以直接解析使用

用法：
├── 在API请求中指定response_format
├── 在Prompt中说明要JSON格式
└── 指定具体的JSON结构
```

**例子：**
```
请求：
"列出3种水果，用JSON格式返回"

普通模式可能输出：
"好的，三种水果是：
1. 苹果
2. 香蕉
3. 橘子"

JSON模式输出：
{
  "fruits": [
    {"name": "苹果", "color": "红色"},
    {"name": "香蕉", "color": "黄色"},
    {"name": "橘子", "color": "橙色"}
  ]
}

可以直接用程序解析
```

## 五、评估与优化

### Hallucination (幻觉)

**一句话：** LLM编造不存在的事实，一本正经地胡说八道。

**详细解释：**
```
LLM会"编造"信息：
├── 虚构不存在的人、事、物
├── 给出错误的事实
├── 引用不存在的来源
└── 但说得很自信

为什么会幻觉：
├── LLM不懂"不知道"的概念
├── 训练数据有冲突或错误
├── 问题超出知识范围
└── 模式补全本能

如何避免：
├── 用RAG提供准确信息
├── 要求标注来源
├── 验证关键信息
├── 让它说"不确定"而非编造
```

**例子：**
```
问："《三体》的作者是谁？"

正确回答："刘慈欣"

幻觉可能：
"《三体》的作者是王小波，
这本书出版于2010年..."

完全是编造的
```

---

### Perplexity (困惑度)

**一句话：** 衡量模型"惊讶程度"的指标，越低越好。

**详细解释：**
```
Perplexity = 模型对文本的"不意外程度"

低Perplexity：
├── 模型对这段话"不惊讶"
├── 符合它的预期
└── 文本质量高

高Perplexity：
├── 模型对这段话"很惊讶"
├── 不符合预期
└── 可能是乱码或不通顺

用途：
├── 评估模型质量
├── 比较不同模型
└── 检测异常文本
```

**例子：**
```
句子1："今天天气很好"
Perplexity：10（低，正常句子）

句子2："天气今天好很"
Perplexity：200（高，语序错误）

句子3："asdfghjkl"
Perplexity：1000+（非常高，乱码）
```

---

### BLEU Score (BLEU分数)

**一句话：** 衡量机器翻译/生成文本与参考答案的相似度。

**详细解释：**
```
BLEU衡量生成文本的质量。

做法：
├── 比较生成文本和参考答案
├── 看有多少词/短语相同
├── 计算相似度分数
└── 0-100分

分数含义：
├── 0-20：几乎不相关
├── 20-40：有些相关
├── 40-60：相当接近
├── 60-80：质量很好
└── 80-100：几乎一样

常用于：
├── 机器翻译评估
├── 文本生成评估
└── 摘要质量评估
```

**例子：**
```
参考："我喜欢吃苹果"
生成："我喜欢苹果"

BLEU分数：约80分
（大部分词相同，意思一致）

参考："我喜欢吃苹果"
生成："苹果是红色的"

BLEU分数：约20分
（意思完全不同）
```

### RLHF vs RLAIF

**一句话：** RLHF用人类反馈，RLAIF用AI反馈。

**详细解释：**
```
RLHF (人类反馈)：
├── 人类给回答打分/排序
├── 训练奖励模型
├── 用人类标准优化
└── 成本高、速度慢

RLAIF (AI反馈)：
├── 用强AI给回答打分
├── 训练奖励模型
├── 用AI标准优化
└── 成本低、速度快

趋势：
├── RLAIF越来越流行
├── 可以大规模进行
└── 但可能引入AI偏见
```

## 六、新兴概念（2024）

### MoE (Mixture of Experts / 混合专家)

**一句话：** 模型有很多"专家"，每次只用其中几个，省计算量。

**详细解释：**
```
传统模型：
├── 每次推理用所有参数
└── 计算量大

MoE模型：
├── 有很多"专家"（子网络）
├── 输入时，只激活相关专家
├── 大部分专家"休息"
└── 计算量小，效果好

例子：
├── 总参数：1万亿
├── 每次激活：1000亿（10%）
└── 效果：接近全参数模型

代表模型：
├── Mixtral 8x7B
├── GPT-4（传闻）
└── DeepSeek
```

---

### Long Context (长上下文)

**一句话：** LLM能处理的文字量大幅提升，从几千到几十万Token。

**详细解释：**
```
Context Window的发展：
├── 2022：4K tokens（约3千字）
├── 2023：32K-100K tokens
├── 2024：200K-1M+ tokens

长上下文的用途：
├── 处理整本书
├── 分析长代码库
├── 处理完整法律文档
├── 长时间多轮对话
└── 不需要切片处理

代表模型：
├── Claude 3：200K
├── Gemini 1.5 Pro：1M
└── GPT-4 Turbo：128K
```

---

### Inference Optimization (推理优化)

**一句话：** 让LLM运行更快、更省钱的技术。

**详细解释：**
```
推理优化的方法：

1. 量化（Quantization）
   └── 用更少位数表示参数

2. 投机解码（Speculative Decoding）
   └── 小模型猜，大模型验证

3. KV Cache优化
   └── 缓存计算结果

4. 批处理（Batching）
   └── 一次处理多个请求

5. 模型蒸馏（Distillation）
   └── 大模型教小模型

效果：
├── 推理速度提升2-10倍
├── 成本降低50-80%
└── 精度损失可控
```

---

### Prompt Caching (提示词缓存)

**一句话：** 缓存相同或相似的Prompt处理结果，加速响应。

**详细解释：**
```
很多Prompt有重复部分：
├── 系统提示词（每次一样）
├── 知识库文档（RAG中不变）
├── 示例（Few-shot一样）
└── 只有用户输入不同

Prompt Caching：
├── 缓存重复部分的处理结果
├── 新输入只处理差异部分
├── 大幅节省计算和费用
└── 响应更快

支持的服务：
├── Claude：Prompt Caching
├── OpenAI：正在推出
└── 其他厂商跟进中
```
