# 用户故事

## 概述

用户故事（User Story）是敏捷开发中描述需求的一种方式，从用户角度出发，简短描述用户需要什么功能以及为什么需要。

## 基本格式

### 标准模板

```
作为 <角色>
我想要 <功能/行为>
以便于 <价值/目的>
```

### 示例

```markdown
作为 一名电商用户
我想要 查看订单物流状态
以便于 知道包裹什么时候能送到
```

## 用户故事的三要素（3C）

### 1. Card（卡片）

**定义**：用户故事的书面描述

**特点**：
- 简短精炼
- 写在卡片或电子工具上
- 便于管理和排序

**示例**：

```
┌────────────────────────────────┐
│  用户故事                       │
├────────────────────────────────┤
│  作为：电商用户                 │
│  我想要：查看订单物流状态        │
│  以便于：知道包裹送达时间        │
├────────────────────────────────┤
│  优先级：P1                     │
│  故事点：3                      │
└────────────────────────────────┘
```

### 2. Conversation（对话）

**定义**：关于用户故事的讨论

**内容**：
- 需求的详细说明
- 实现方式的讨论
- 技术细节的澄清

**讨论要点**：
- 这个功能具体是什么？
- 为什么要做这个功能？
- 如何判断完成？
- 有什么限制条件？

### 3. Confirmation（确认）

**定义**：验收标准（Acceptance Criteria）

**作用**：
- 确认需求理解一致
- 作为测试依据
- 判断故事完成的标准

## 验收标准（Acceptance Criteria）

### 格式

```markdown
## 验收标准

**Given**（给定）：初始条件
**When**（当）：触发动作
**Then**（那么）：预期结果
```

### 示例

```markdown
## 用户故事：用户登录

**验收标准**：

1. 正确登录
   - Given：用户在登录页面
   - When：输入正确的用户名和密码并点击登录
   - Then：登录成功，跳转到首页

2. 密码错误
   - Given：用户在登录页面
   - When：输入正确的用户名和错误的密码
   - Then：显示"密码错误"提示

3. 账号不存在
   - Given：用户在登录页面
   - When：输入不存在的用户名
   - Then：显示"账号不存在"提示

4. 表单验证
   - Given：用户在登录页面
   - When：用户名或密码为空时点击登录
   - Then：显示"请输入用户名/密码"提示
```

### 验收标准清单

| 检查项 | 描述 |
|--------|------|
| 功能性 | 功能是否按预期工作 |
| 边界条件 | 边界值是否正确处理 |
| 异常处理 | 错误情况是否正确处理 |
| 性能 | 性能是否满足要求 |
| 安全 | 安全要求是否满足 |
| 用户体验 | 交互是否流畅 |

## INVEST 原则

好的用户故事应该符合 INVEST 原则：

### I - Independent（独立的）

**含义**：用户故事之间相互独立

**为什么重要**：
- 便于优先级排序
- 可以独立开发测试
- 灵活调整计划

**示例**：

```
❌ 依赖的：
故事A：用户可以查看订单
故事B：用户可以查看订单的物流信息（依赖A）

✓ 独立的：
故事A：用户可以查看订单列表
故事B：用户可以查看订单详情（包括物流信息）
```

### N - Negotiable（可协商的）

**含义**：用户故事的细节可以讨论

**为什么重要**：
- 鼓励对话
- 不是合同
- 允许调整

**示例**：

```
❌ 不可协商的：
系统必须使用 MD5 加密存储密码，必须支持 16 位密码

✓ 可协商的：
用户密码需要安全存储，具体实现方式可讨论
```

### V - Valuable（有价值的）

**含义**：用户故事对用户或客户有价值

**为什么重要**：
- 确保做正确的事
- 避免无意义的工作
- 提供优先级依据

**示例**：

```
❌ 没有价值的：
数据库需要使用 MySQL 5.7

✓ 有价值的：
用户可以快速搜索商品（通过优化数据库实现）
```

### E - Estimable（可估算的）

**含义**：开发团队可以估算工作量

**为什么重要**：
- 支持迭代规划
- 帮助优先级决策
- 发现不确定因素

**示例**：

```
❌ 不可估算的：
系统需要性能优化

✓ 可估算的：
商品搜索响应时间从 3 秒降低到 1 秒以内
```

### S - Small（小的）

**含义**：用户故事足够小，一个 Sprint 能完成

**为什么重要**：
- 便于进度跟踪
- 降低风险
- 提高灵活性

**建议**：
- 一个故事不超过 8-13 故事点
- 如果太大，考虑拆分

### T - Testable（可测试的）

**含义**：用户故事有明确的验收标准

**为什么重要**：
- 明确完成定义
- 支持 QA 测试
- 避免理解偏差

**示例**：

```
❌ 不可测试的：
用户体验要好

✓ 可测试的：
页面加载时间 < 2 秒
表单提交成功率 > 95%
```

## 用户故事拆分

### 何时需要拆分

- 故事点 > 13
- 包含多个功能
- 一个 Sprint 完不成
- 包含多个验收标准

### 拆分方法

#### 1. 按工作流程拆分

```
原故事：用户可以管理订单

拆分后：
- 用户可以查看订单列表
- 用户可以查看订单详情
- 用户可以取消订单
- 用户可以申请退款
```

#### 2. 按业务规则拆分

```
原故事：用户可以使用优惠券

拆分后：
- 用户可以使用满减券
- 用户可以使用折扣券
- 用户可以使用免邮券
- 用户可以叠加使用优惠券
```

#### 3. 按界面元素拆分

```
原故事：用户可以搜索商品

拆分后：
- 用户可以输入关键词搜索
- 用户可以按分类筛选
- 用户可以按价格排序
- 用户可以查看搜索历史
```

#### 4. 按简单/复杂拆分

```
原故事：用户可以登录

拆分后：
- 用户可以用用户名密码登录（简单）
- 用户可以用手机验证码登录（中等）
- 用户可以用第三方账号登录（复杂）
```

#### 5. 按数据操作拆分

```
原故事：用户可以管理地址

拆分后：
- 用户可以添加地址
- 用户可以编辑地址
- 用户可以删除地址
- 用户可以设置默认地址
```

### 拆分原则

1. **保持价值**：每个拆分后的故事仍然有价值
2. **符合 INVEST**：拆分后的故事要符合 INVEST 原则
3. **横向切分**：按功能切分而非按技术层切分
4. **避免依赖**：尽量减少故事间的依赖

## 用户故事地图

### 概念

用户故事地图是一种可视化工具，将用户故事按用户活动和工作流组织排列。

### 结构

```
                用户目标
                   │
    ┌──────────────┼──────────────┐
    │              │              │
 用户活动1     用户活动2     用户活动3
    │              │              │
 用户任务      用户任务      用户任务
    │              │              │
  故事1.1       故事2.1       故事3.1
  故事1.2       故事2.2       故事3.2
  故事1.3       故事2.3       故事3.3

──────────────────────────────────────
  MVP         V1.0         V2.0
```

### 示例：电商用户故事地图

```
                    购物
                      │
    ┌─────────┬───────┴───────┬─────────┐
    │         │               │         │
  浏览商品   搜索商品      选择商品   下单支付
    │         │               │         │
 分类浏览   关键词搜索    查看详情    加入购物车
    │         │               │         │
 ┌──┴──┐   ┌──┴──┐        ┌──┴──┐   ┌──┴──┐
 │首页  │   │搜索页│        │详情页│   │购物车│
 │分类页│   │结果页│        │评价页│   │结算页│
 └─────┘   └─────┘        └─────┘   └─────┘

──────────────────────────────────────────
   MVP            V1.0           V2.0
```

### 使用方法

1. **建立骨架**：定义用户活动和任务
2. **填充故事**：添加具体用户故事
3. **确定优先级**：从上到下，从左到右
4. **规划发布**：用横线划分发布版本

## 用户故事估算

### 故事点（Story Points）

**定义**：表示用户故事相对复杂度的单位

**斐波那契数列**：1, 2, 3, 5, 8, 13, 21, 34, 55...

**含义**：

| 故事点 | 复杂度 | 时间估算 |
|--------|--------|---------|
| 1 | 非常简单 | 几小时 |
| 2 | 简单 | 半天 |
| 3 | 中等简单 | 1天 |
| 5 | 中等 | 2-3天 |
| 8 | 复杂 | 1周 |
| 13 | 非常复杂 | 2周 |
| 21+ | 太复杂，需要拆分 | - |

### 规划扑克（Planning Poker）

**流程**：

```
1. 主持人朗读用户故事
2. 团队成员提问澄清
3. 每人选择估算卡片（不展示）
4. 同时亮牌
5. 差异大的成员解释理由
6. 讨论后重新估算
7. 达成共识
```

**示例**：

```
用户故事：用户可以使用手机号登录

成员A：5（需要短信服务对接）
成员B：3（简单的表单验证）
成员C：8（需要考虑多种异常情况）
成员D：5（和A想的一样）

讨论：考虑短信服务商选择、验证码有效期、重试次数等

重新估算：
成员A：5
成员B：5
成员C：5
成员D：5

结果：5 故事点
```

## 用户故事与敏捷实践

### Sprint 规划

1. PO 介绍高优先级用户故事
2. 团队讨论澄清需求
3. 团队估算故事点
4. 承诺本 Sprint 完成的用户故事
5. 分配任务

### 每日站会

每人回答：
1. 昨天完成了哪些用户故事？
2. 今天计划完成哪些用户故事？
3. 遇到什么障碍？

### Sprint 评审

1. 演示完成的用户故事
2. PO 验收
3. 收集反馈

### Sprint 回顾

1. 回顾用户故事完成情况
2. 讨论改进点
3. 制定行动计划

## 常见问题

### 问题1：用户故事太大怎么办？

**解决**：
1. 识别故事中的多个功能点
2. 按拆分方法分解
3. 确保每个故事符合 INVEST

### 问题2：用户故事之间有依赖怎么办？

**解决**：
1. 重新拆分，减少依赖
2. 在 Backlog 中标注依赖关系
3. 优先完成被依赖的故事

### 问题3：非功能性需求如何写用户故事？

**示例**：

```markdown
作为 运维人员
我想要 系统支持 1000 并发用户
以便于 保证系统在高负载下稳定运行

验收标准：
- 1000 并发用户下响应时间 < 2秒
- 错误率 < 0.1%
- 系统资源使用 < 80%
```

### 问题4：技术任务如何处理？

**方式1**：转化为用户故事

```markdown
作为 开发人员
我想要 重构用户模块代码
以便于 提高代码可维护性
```

**方式2**：作为技术任务

```markdown
技术任务：升级数据库到 MySQL 8.0
原因：性能优化
工作量：5 故事点
```

## 总结

用户故事的核心是：

1. **用户视角**：从用户角度描述需求
2. **价值导向**：关注价值而非功能
3. **可协商**：鼓励讨论而非合同
4. **小而独立**：便于管理和开发
5. **可测试**：有明确的验收标准
6. **持续优化**：通过对话不断完善
