# 需求分析

## 概述

需求分析是产品开发的核心环节，通过系统化方法挖掘、理解和定义用户真实需求的过程。

## 需求的层次

### Kano 模型

```
满意度
  ↑
  │           ╱
  │          ╱  兴奋型需求
  │         ╱
  │        ╱
  │       ╱────────── 期望型需求
  │      ╱
  │  ───╱───────────── 基本型需求
  │
  └────────────────────→ 实现程度
```

### 需求类型

| 类型 | 特点 | 用户态度 | 优先级 |
|------|------|---------|--------|
| 基本型 | 必须有 | 没有会不满，有了也不满意 | 最高 |
| 期望型 | 越多越好 | 没有会不满，有了会满意 | 高 |
| 兴奋型 | 意外惊喜 | 没有也不不满，有了很满意 | 中 |
| 无差异 | 有无都行 | 不关心 | 低 |
| 反向 | 越多越糟 | 不喜欢 | 避免 |

## 需求挖掘方法

### 1. 用户访谈

#### 访谈类型

| 类型 | 适用场景 | 时长 | 人数 |
|------|---------|------|------|
| 深度访谈 | 了解深层需求 | 60-90 分钟 | 1 对 1 |
| 焦点小组 | 激发群体讨论 | 90-120 分钟 | 6-8 人 |
| 专家访谈 | 获取专业见解 | 30-60 分钟 | 1 对 1 |

#### 访谈技巧

**开放式问题**：

```
❌ 你喜欢这个功能吗？（封闭式）
✅ 你平时怎么完成这个任务？（开放式）

❌ 你会用这个产品吗？（封闭式）
✅ 能告诉我你目前是怎么解决这个问题的？（开放式）
```

**追问技巧**：

```
用户：我想要一个搜索功能

追问1：你为什么要搜索？
用户：因为内容太多了

追问2：内容多会带来什么问题？
用户：找不到想要的东西

追问3：找不到东西会有什么影响？
用户：浪费时间，影响效率

根本需求：提高内容查找效率
```

### 2. 问卷调查

#### 问卷设计原则

1. **简洁**：问题数量控制在 10-15 个
2. **清晰**：避免歧义和双重否定
3. **中性**：避免引导性提问
4. **逻辑**：问题顺序由浅入深

#### 常用问题类型

| 类型 | 适用场景 | 示例 |
|------|---------|------|
| 单选题 | 互斥选项 | 您的性别是？ |
| 多选题 | 多个选项 | 您使用过哪些产品？ |
| 李克特量表 | 态度测量 | 您对这个功能的满意度（1-5分） |
| 开放题 | 自由表达 | 您对产品有什么建议？ |

#### 量表设计

```
满意度量表：
1 - 非常不满意
2 - 不满意
3 - 一般
4 - 满意
5 - 非常满意

NPS 量表：
0-6 - 贬损者
7-8 - 中立者
9-10 - 推荐者
```

### 3. 用户观察

#### 观察方法

| 方法 | 描述 | 优势 | 劣势 |
|------|------|------|------|
| 伴随观察 | 跟随用户使用 | 真实场景 | 可能有霍桑效应 |
| 可用性测试 | 给定任务观察 | 可控性强 | 可能偏离真实 |
| 日记研究 | 用户自我记录 | 长期行为 | 依赖用户配合 |

#### 观察要点

```
观察清单：
□ 用户在哪里使用产品？
□ 使用什么设备？
□ 周围有什么干扰？
□ 遇到什么困难？
□ 如何解决问题？
□ 有什么情绪反应？
```

### 4. 数据分析

#### 行为数据分析

| 指标 | 分析目的 |
|------|---------|
| 页面访问路径 | 了解用户行为流程 |
| 功能使用率 | 识别核心功能 |
| 停留时长 | 判断内容吸引力 |
| 跳出率 | 发现体验问题 |
| 转化漏斗 | 定位流失节点 |

#### 数据分析方法

1. **趋势分析**：指标随时间的变化
2. **对比分析**：不同群体的差异
3. **漏斗分析**：转化过程的流失
4. **留存分析**：用户的持续使用
5. **归因分析**：找到影响结果的因素

### 3. 需求真伪判断

| 伪需求特征 | 真需求特征 |
|-----------|-----------|
| 少数人提及 | 多数人反馈 |
| 表述模糊 | 描述具体 |
| 没有场景 | 有明确场景 |
| 不愿意付费 | 愿意付出成本 |
| "锦上添花" | "雪中送炭" |

## 需求分解

### 1. Epic → Story → Task

```
Epic（史诗）
  └── Story（用户故事）
        └── Task（任务）
```

**示例**：

```
Epic：用户管理
  ├── Story：用户注册
  │     ├── Task：注册页面设计
  │     ├── Task：表单验证逻辑
  │     └── Task：注册接口开发
  ├── Story：用户登录
  │     ├── Task：登录页面设计
  │     ├── Task：认证逻辑开发
  │     └── Task：Token 管理
  └── Story：密码找回
        ├── Task：找回流程设计
        └── Task：邮件服务对接
```

### 2. INVEST 原则

| 原则 | 说明 | 检查问题 |
|------|------|---------|
| Independent | 独立性 | 能否单独开发？ |
| Negotiable | 可协商 | 细节能否讨论？ |
| Valuable | 有价值 | 对用户有价值吗？ |
| Estimable | 可估算 | 能评估工作量吗？ |
| Small | 足够小 | 能在一个 Sprint 完成？ |
| Testable | 可测试 | 有验收标准吗？ |

## 需求分析常见问题

### 1. 把用户说的当用户要的

**问题**：用户说"我要一匹更快的马"

**解决**：追问"为什么"，找到真实需求"更快到达目的地"

### 2. 以自我为中心

**问题**："我觉得用户需要..."

**解决**：基于用户研究和数据分析

### 3. 需求镀金

**问题**：添加用户没要求的"高级功能"

**解决**：聚焦核心需求，避免过度设计

### 4. 需求蔓延

**问题**：需求不断增加，项目延期

**解决**：明确范围，严格变更控制

## 总结

需求分析的核心是：

1. **多源验证**：用多种方法交叉验证
2. **深挖本质**：找到真实需求而非表面需求
3. **结构化**：系统化整理和管理需求
4. **持续迭代**：需求会变化，需要持续关注
5. **沟通协作**：需求分析需要多方参与
