# 需求优先级

## 概述

需求优先级排序是产品管理中的关键决策，它决定了有限资源的分配方式，直接影响产品的发展方向和用户价值。

## 核心原则

### 优先级决策三角

```
         用户价值
            △
           ╱ ╲
          ╱   ╲
         ╱     ╲
        ╱       ╲
       ╱         ╲
 商业价值 ───────── 企业能力
```

### 基本原则

1. **价值导向**：优先做高价值需求
2. **成本考量**：考虑实现成本和机会成本
3. **战略一致**：与产品战略方向一致
4. **风险平衡**：平衡高风险和低风险需求
5. **依赖关系**：考虑需求间的依赖关系

### 模型概述

### 需求类型详解

#### 1. 基本型需求（Must-have）

**特点**：
- 产品的基础功能
- 没有会导致用户极度不满
- 有了用户也不会特别满意
- 必须满足

**处理策略**：最高优先级，必须实现

**示例**：
- 电商：商品展示、购物车、支付
- 社交：注册、发布内容、评论

#### 2. 期望型需求（Performance）

**特点**：
- 用户明确期望的功能
- 做得越好，用户越满意
- 做得不好，用户不满意

**处理策略**：根据资源和竞争情况决定优先级

**示例**：
- 电商：搜索速度、推荐准确度
- 社交：消息推送速度、加载速度

#### 3. 兴奋型需求（Delighters）

**特点**：
- 用户没预期到的功能
- 没有用户也不会不满
- 有了会带来惊喜

**处理策略**：作为差异化竞争点，适当投入

**示例**：
- 电商：AR 试穿、智能客服
- 社交：特效滤镜、创意工具

### Kano 评估矩阵

| 正向 \ 反向 | 喜欢 | 理应如此 | 无所谓 | 我能忍受 | 不喜欢 |
|-------------|------|---------|--------|---------|--------|
| 喜欢 | Q | A | A | A | O |
| 理应如此 | R | I | I | I | M |
| 无所谓 | R | I | I | I | M |
| 我能忍受 | R | I | I | I | M |
| 不喜欢 | R | R | R | R | Q |

**符号说明**：
- M：Must-have（基本型）
- O：One-dimensional（期望型）
- A：Attractive（兴奋型）
- I：Indifferent（无差异）
- R：Reverse（反向）
- Q：Questionable（可疑结果）

## RICE 模型

### 公式

```
RICE 得分 = (Reach × Impact × Confidence) / Effort
```

### 组成要素

#### 1. Reach（覆盖范围）

**定义**：在给定时间内会接触到该功能的用户数量

**量化方法**：

| 指标 | 数值 |
|------|------|
| 每季度影响用户数 | 实际数字 |
| 占总用户比例 | 百分比 |

**示例**：
- 3个月内会影响 10,000 用户 = 10,000
- 影响 50% 的活跃用户 = 50

#### 2. Impact（影响力）

**定义**：该功能对目标的影响程度

**评分标准**：

| 分数 | 描述 |
|------|------|
| 3 | 质变影响（Massive impact） |
| 2 | 高影响（High impact） |
| 1 | 中等影响（Medium impact） |
| 0.5 | 低影响（Low impact） |
| 0.25 | 极小影响（Minimal impact） |

**示例**：
- 显著提升转化率 = 3
- 中等提升留存率 = 2
- 小幅提升满意度 = 1

#### 3. Confidence（信心度）

**定义**：对 Reach 和 Impact 评估的信心程度

**评分标准**：

| 分数 | 描述 |
|------|------|
| 100% | 高信心（有定量数据支持） |
| 80% | 中等信心（有一些数据支持） |
| 50% | 低信心（主要基于直觉） |

**评估维度**：
- 是否有用户调研数据？
- 是否有竞品参考？
- 是否有历史数据？

#### 4. Effort（工作量）

**定义**：完成该需求所需的总工作量

**估算单位**：人月

**包含内容**：
- 产品设计
- UI/UX 设计
- 前端开发
- 后端开发
- 测试
- 文档

**示例**：
- 小功能：0.5 人月
- 中等功能：2 人月
- 大功能：6 人月

### RICE 计算示例

```markdown
## 需求：增加搜索功能

**Reach**：每月 50,000 次搜索 = 50,000
**Impact**：显著提升用户找到内容的效率 = 2
**Confidence**：有用户调研支持 = 80% = 0.8
**Effort**：预计 2 人月 = 2

**RICE 得分**：
(50,000 × 2 × 0.8) / 2 = 40,000
```

## MoSCoW 方法

### 分类定义

| 分类 | 含义 | 描述 | 优先级 |
|------|------|------|--------|
| **M**ust have | 必须有 | 不做会导致项目失败 | 最高 |
| **S**hould have | 应该有 | 重要但非关键 | 高 |
| **C**ould have | 可以有 | 有更好，没有也可 | 中 |
| **W**on't have | 暂不做 | 这个周期不做 | 低 |

### 判断标准

#### Must have

- 没有它产品无法正常工作
- 法律或合规要求
- 核心业务流程必需

**测试问题**：如果没有这个功能，产品还能发布吗？

#### Should have

- 对用户体验有重要影响
- 能带来显著商业价值
- 有变通的解决方案

**测试问题**：如果没有这个功能，用户会非常不满吗？

#### Could have

- 能提升用户体验
- 成本不高，可以做
- 不会造成严重后果

**测试问题**：这个功能能让产品更好，但不是必需的吗？

#### Won't have

- 好的需求但不是当期重点
- 资源有限，需要取舍
- 放到后续版本

**测试问题**：这个需求是否应该放到下一期？

### MoSCoW 分配建议

```
Must have：   约 60%
Should have： 约 20%
Could have：  约 15%
Won't have：  约 5%
```

## 优先级矩阵

### 价值-成本矩阵

```
价值
  ↑
  │  P1：优先做    │  P0：必须做
  │  高价值低成本  │  高价值高成本
  ├────────────────┼────────────────┤
  │  P3：可能做    │  P2：计划做
  │  低价值低成本  │  低价值高成本
  │                │
  └────────────────┴────────────→ 成本
```

### 紧急-重要矩阵（艾森豪威尔矩阵）

```
重要
  ↑
  │  P1：计划做    │  P0：立即做
  │  重要不紧急    │  重要且紧急
  ├────────────────┼────────────────┤
  │  P3：不做      │  P2：授权做
  │  不重要不紧急  │  不重要但紧急
  │                │
  └────────────────┴────────────→ 紧急
```

## 其他优先级方法

### 1. WSJF（Weighted Shortest Job First）

**公式**：

```
WSJF = (用户价值 + 时间价值 + 风险降低) / 工作量
```

**适用**：SAFe（Scaled Agile Framework）框架

### 2. ICE 评分

```
ICE 得分 = Impact × Confidence × Ease
```

| 维度 | 说明 |
|------|------|
| Impact | 影响力 |
| Confidence | 信心度 |
| Ease | 实现难度（越容易分数越高） |

### 3. 价值 vs 复杂度

```
优先级 = 价值 / 复杂度
```

**评估标准**：

| 价值 | 复杂度 | 优先级 |
|------|--------|--------|
| 高 | 低 | 最高 |
| 高 | 高 | 中高 |
| 低 | 低 | 中低 |
| 低 | 高 | 最低 |

## 优先级决策流程

### 流程图

```
收集需求
    ↓
需求分析（分类、去重）
    ↓
价值评估
    ↓
成本估算
    ↓
优先级评分
    ↓
团队评审
    ↓
最终排序
    ↓
纳入计划
```

### 评审会议

#### 会议准备

```markdown
## 评审准备清单

- [ ] 需求描述清晰
- [ ] 价值评估完成
- [ ] 成本估算完成
- [ ] 相关数据收集
- [ ] 竞品分析完成
```

#### 评审要点

1. **价值讨论**
   - 用户价值是什么？
   - 商业价值是什么？
   - 数据支持吗？

2. **成本讨论**
   - 技术实现复杂度？
   - 需要多少资源？
   - 有什么风险？

3. **优先级讨论**
   - 为什么是这个优先级？
   - 有没有依赖关系？
   - 是否可以调整？

## 优先级调整

### 调整时机

1. 市场环境变化
2. 竞争对手动作
3. 用户反馈变化
4. 技术条件变化
5. 战略方向调整

### 调整原则

1. **数据驱动**：基于数据而非直觉
2. **透明沟通**：及时告知相关方
3. **记录原因**：记录调整的原因
4. **影响评估**：评估调整的影响

### 常见问题应对

**问题1：为什么我的需求优先级低？**

**回答**：基于 RICE 评分，当前资源需要优先投入更高价值的需求，您的需求已纳入后续计划。

**问题2：能不能提高优先级？**

**回答**：请提供更多数据支持该需求的价值，我们可以在下次评审中重新评估。

## 总结

需求优先级排序的关键：

1. **建立标准**：使用系统化的评估方法
2. **数据驱动**：基于数据而非主观判断
3. **透明沟通**：让相关方理解优先级理由
4. **动态调整**：根据情况变化及时调整
5. **团队共识**：优先级需要团队共同认可
