# 风险评估

## 概述

风险评估是识别、分析和评估产品开发过程中可能遇到的风险，并制定相应应对策略的过程。

## 风险类型

### 1. 按来源分类

| 类型 | 描述 | 示例 |
|------|------|------|
| 市场风险 | 市场需求变化 | 用户需求变化 |
| 竞争风险 | 竞争对手威胁 | 竞品功能超越 |
| 技术风险 | 技术实现困难 | 技术方案不可行 |
| 资源风险 | 资源不足 | 人才、资金不足 |
| 政策风险 | 政策法规变化 | 合规要求变化 |
| 运营风险 | 运营问题 | 服务不稳定 |

### 2. 按影响分类

| 类型 | 影响 | 优先级 |
|------|------|--------|
| 致命风险 | 项目失败 | 最高 |
| 严重风险 | 重大影响 | 高 |
| 一般风险 | 中等影响 | 中 |
| 轻微风险 | 小影响 | 低 |

### 3. 按时间分类

| 类型 | 时间跨度 | 示例 |
|------|---------|------|
| 短期风险 | 几天到几周 | 版本延期 |
| 中期风险 | 几周到几月 | 人员离职 |
| 长期风险 | 几月到几年 | 市场变化 |

## 风险管理流程

### 流程图

```
风险识别 → 风险分析 → 风险评估 → 风险应对 → 风险监控
    ↑                                              │
    └──────────────────────────────────────────────┘
```

### 详细步骤

#### 步骤1：风险识别

**方法**：

1. **头脑风暴**：团队讨论可能的风险
2. **检查清单**：使用历史风险清单
3. **专家访谈**：咨询专家意见
4. **历史分析**：分析历史项目问题
5. **SWOT 分析**：识别威胁和劣势

**风险识别清单**：

```markdown
## 市场风险
- [ ] 用户需求可能发生变化
- [ ] 市场规模可能不如预期
- [ ] 竞品可能推出类似功能

## 技术风险
- [ ] 技术方案可能不可行
- [ ] 性能可能不达标
- [ ] 安全可能存在漏洞

## 资源风险
- [ ] 核心人员可能离职
- [ ] 资金可能不足
- [ ] 时间可能不够

## 外部风险
- [ ] 政策法规可能变化
- [ ] 第三方服务可能不稳定
- [ ] 供应链可能中断
```

#### 步骤2：风险分析

**分析方法**：

1. **定性分析**
   - 高/中/低
   - 严重/一般/轻微

2. **定量分析**
   - 概率值
   - 影响金额

**风险登记表**：

| ID | 风险描述 | 类型 | 概率 | 影响 | 风险等级 |
|----|---------|------|------|------|---------|
| R001 | 核心开发离职 | 资源 | 中 | 高 | 高 |
| R002 | 技术方案失败 | 技术 | 低 | 高 | 中 |
| R003 | 需求变更 | 市场 | 高 | 中 | 中 |
| R004 | 竞品超越 | 竞争 | 中 | 中 | 中 |

#### 步骤3：风险评估

**风险矩阵**：

```
影响程度
  ↑
  │  ┌─────────┐  ┌─────────┐  ┌─────────┐
高 │  │  中风险  │  │  高风险  │  │  极高风险 │
  │  │  (3,3)  │  │  (3,4)  │  │  (3,5)  │
  │  └─────────┘  └─────────┘  └─────────┘
  │  ┌─────────┐  ┌─────────┐  ┌─────────┐
中 │  │  低风险  │  │  中风险  │  │  高风险  │
  │  │  (2,2)  │  │  (2,3)  │  │  (2,4)  │
  │  └─────────┘  └─────────┘  └─────────┘
  │  ┌─────────┐  ┌─────────┐  ┌─────────┐
低 │  │ 极低风险 │  │  低风险  │  │  中风险  │
  │  │  (1,1)  │  │  (1,2)  │  │  (1,3)  │
  │  └─────────┘  └─────────┘  └─────────┘
  └────────────────────────────────────────→ 发生概率
           低           中           高
```

**风险等级定义**：

| 等级 | 分数 | 描述 | 处理优先级 |
|------|------|------|-----------|
| 极高风险 | 15-25 | 必须立即处理 | 最高 |
| 高风险 | 10-14 | 需要优先处理 | 高 |
| 中风险 | 5-9 | 需要关注 | 中 |
| 低风险 | 1-4 | 可以接受 | 低 |

**风险评分公式**：

```
风险评分 = 概率 × 影响

示例：
概率：高（4）× 影响：高（5）= 风险评分：20（极高风险）
```

#### 步骤4：风险应对

**应对策略**：

| 策略 | 描述 | 适用场景 |
|------|------|---------|
| 规避 | 消除风险源 | 风险可避免 |
| 转移 | 转移给第三方 | 风险可转移 |
| 缓解 | 降低风险影响 | 风险可降低 |
| 接受 | 接受风险存在 | 风险影响小 |

**风险应对计划**：

```markdown
## 风险应对计划

### R001：核心开发离职

**风险等级**：高

**应对策略**：缓解

**应对措施**：
1. 建立知识文档
2. 代码审查制度
3. 关键代码多人掌握
4. 人员备份计划

**责任人**：张三

**预算**：¥X

**触发条件**：人员提出离职

**应急预案**：
1. 立即启动备份人员
2. 加快知识转移
3. 必要时外部招聘
```

#### 步骤5：风险监控

**监控内容**：
- 风险状态变化
- 新风险出现
- 应对措施有效性

**监控方式**：
- 定期风险评审
- 风险指标监控
- 预警机制

## 常见风险及应对

### 1. 需求风险

**风险**：需求频繁变更、需求理解偏差

**应对**：
- 建立需求变更流程
- 加强需求评审
- 使用原型验证
- 保持与业务方沟通

### 2. 技术风险

**风险**：技术方案不可行、性能问题

**应对**：
- 技术预研
- 技术评审
- 原型验证
- 预留缓冲时间

### 3. 资源风险

**风险**：人员不足、资金不足

**应对**：
- 人员备份计划
- 合理分配资源
- 外部资源支持
- 优先级管理

### 4. 时间风险

**风险**：项目延期

**应对**：
- 合理估算时间
- 预留缓冲
- 分阶段交付
- 定期检查进度

### 5. 质量风险

**风险**：Bug 过多、体验差

**应对**：
- 代码审查
- 自动化测试
- 灰度发布
- 用户测试

### 6. 外部依赖风险

**风险**：第三方服务不稳定

**应对**：
- 多服务商备份
- 服务降级方案
- 监控告警
- 合同保障

## 风险评估工具

### 1. FMEA（失效模式与影响分析）

```markdown
## FMEA 分析

| 失效模式 | 失效原因 | 失效影响 | 严重度 | 发生度 | 检测度 | RPN |
|---------|---------|---------|-------|-------|-------|-----|
| 服务器宕机 | 硬件故障 | 服务不可用 | 10 | 2 | 3 | 60 |
| 数据丢失 | 未备份 | 用户数据丢失 | 9 | 1 | 5 | 45 |

RPN = 严重度 × 发生度 × 检测度
```

### 2. 蒙特卡洛模拟

**应用**：评估项目完成时间的不确定性

**步骤**：
1. 确定变量和分布
2. 进行大量随机模拟
3. 分析结果分布

### 3. 敏感性分析

**目的**：识别对结果影响最大的因素

**方法**：
1. 确定关键变量
2. 改变单一变量
3. 观察结果变化

### 4. 决策树分析

```
                风险事件
                   │
         ┌─────────┴─────────┐
         │                   │
       发生                 不发生
      (概率P)              (概率1-P)
         │                   │
         ↓                   ↓
    采取应对              正常进行
    成本+C1               成本+C2
```

## 总结

风险评估的关键：

1. **主动识别**：提前发现潜在风险
2. **系统分析**：科学评估风险等级
3. **有效应对**：制定可行应对策略
4. **持续监控**：跟踪风险状态变化
5. **及时沟通**：保持风险信息透明
6. **持续改进**：从风险中学习
