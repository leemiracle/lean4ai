# 产品评审

## 概述

产品评审是产品开发过程中各方对产品方案、进度、质量进行评估和确认的活动，是确保产品质量和团队协作的重要机制。

## 评审类型

### 按阶段分类

| 类型 | 时间点 | 参与者 | 目的 |
|------|--------|--------|------|
| 需求评审 | 需求阶段 | PM、设计、开发、QA | 确认需求理解一致 |
| 设计评审 | 设计阶段 | PM、设计、开发 | 确认设计方案 |
| 技术评审 | 开发前 | PM、开发 | 确认技术方案 |
| 代码评审 | 开发中 | 开发团队 | 保证代码质量 |
| 测试评审 | 测试阶段 | PM、QA | 确认测试方案 |
| 验收评审 | 发布前 | PM、QA、设计 | 确认可以发布 |
| 复盘评审 | 发布后 | 全团队 | 总结经验教训 |

## 需求评审

### 目的

- 确保需求理解一致
- 评估技术可行性
- 估算工作量
- 识别风险

### 评审前准备

```markdown
## 需求评审准备清单

### PM 准备
- [ ] PRD 文档完成
- [ ] 业务背景说明
- [ ] 用户场景描述
- [ ] 数据预估
- [ ] 竞品分析（如有）

### 通知参与者
- [ ] 发送会议邀请
- [ ] 提前发送 PRD
- [ ] 预留足够时间
```

### 评审流程

```markdown
## 需求评审流程

### 1. 背景介绍（5分钟）
- 为什么要做这个需求？
- 业务价值是什么？
- 用户价值是什么？

### 2. 需求讲解（15-20分钟）
- 用户故事
- 功能描述
- 业务流程
- 边界条件

### 3. 讨论澄清（20-30分钟）
- 问题澄清
- 可行性讨论
- 方案建议

### 4. 工作量估算（10-15分钟）
- 设计工作量
- 开发工作量
- 测试工作量

### 5. 总结确认（5-10分钟）
- 确认需求范围
- 确认时间节点
- 确认后续行动
```

### 评审检查点

```markdown
## 需求评审检查点

### 需求完整性
- [ ] 需求背景清晰
- [ ] 用户故事完整
- [ ] 功能描述详细
- [ ] 验收标准明确
- [ ] 边界条件考虑

### 技术可行性
- [ ] 技术方案可行
- [ ] 性能要求合理
- [ ] 安全要求明确
- [ ] 依赖关系清晰

### 工作量评估
- [ ] 设计工作量合理
- [ ] 开发工作量合理
- [ ] 测试工作量合理
- [ ] 总时间可接受
```

## 设计评审

### 目的

- 确认设计符合需求
- 评估用户体验
- 确认技术可行性
- 统一设计方向

### 评审内容

```markdown
## 设计评审内容

### 1. 信息架构
- 页面结构合理
- 层级清晰
- 导航易懂

### 2. 交互设计
- 流程顺畅
- 操作简单
- 反馈及时
- 异常处理

### 3. 视觉设计
- 风格统一
- 视觉美观
- 品牌一致

### 4. 响应式设计
- 多端适配
- 不同屏幕适配
```

### 评审流程

```markdown
## 设计评审流程

### 1. 设计背景（5分钟）
- 需求回顾
- 设计目标
- 用户场景

### 2. 方案展示（15-20分钟）
- 整体设计思路
- 关键页面展示
- 交互演示
- 设计亮点

### 3. 讨论反馈（20-30分钟）
- 体验问题
- 技术可行性
- 改进建议

### 4. 确认方案（10分钟）
- 方案确认
- 修改点记录
- 后续计划
```

## 代码评审

### 目的

- 保证代码质量
- 发现潜在问题
- 知识分享
- 保持代码一致性

### 评审标准

```markdown
## 代码评审标准

### 1. 功能正确性
- [ ] 代码实现符合需求
- [ ] 边界条件处理
- [ ] 异常情况处理

### 2. 代码质量
- [ ] 代码可读性好
- [ ] 命名规范
- [ ] 注释清晰
- [ ] 无冗余代码

### 3. 性能
- [ ] 无性能问题
- [ ] 资源使用合理

### 4. 安全
- [ ] 无安全漏洞

### 5. 可维护性
- [ ] 代码结构清晰

- [ ] 易于测试
```

## 验收评审

### 目的

- 确认功能完整实现
- 确认质量达标
- 决定是否发布

### 验收清单

```markdown
## 验收评审清单

### 功能验收
- [ ] 所有功能已实现
- [ ] 符合 PRD 要求
- [ ] 边界条件处理正确

### 设计验收
- [ ] 设计还原度达标
- [ ] 交互符合预期
- [ ] 多端适配完成

### 质量验收
- [ ] 无 P0/P1 Bug
- [ ] P2 Bug 数量可控
- [ ] 性能达标
- [ ] 兼容性测试通过

### 发布准备
- [ ] 发布计划完成
- [ ] 运营物料准备
- [ ] 监控告警配置
- [ ] 回滚方案准备
```

## 复盘评审

### 目的

- 总结经验教训
- 发现改进机会
- 提升团队能力

### 复盘流程

```markdown
## 复盘流程

### 1. 回顾目标（10分钟）
- 原定目标是什么？
- 实际结果如何？

### 2. 分析过程（20分钟）
- 做得好的地方？
- 做得不好的地方？
- 意外的情况？

### 3. 总结经验（15分钟）
- 学到了什么？
- 可以复制的经验？

### 4. 改进计划（15分钟）
- 需要改进什么？
- 谁负责改进？
- 何时完成？
```

## 评审最佳实践

### 1. 提前准备

```markdown
## 评审准备

### 发起方
- 提前发送材料
- 预留阅读时间
- 准备演示

### 参与方
- 提前阅读材料
- 准备问题和建议
- 了解背景信息
```

### 2. 控制时间

```markdown
## 时间控制

### 会议时长
- 需求评审：60-90 分钟
- 设计评审：45-60 分钟
- 技术评审：30-45 分钟
- 代码评审：线上异步
- 复盘评审：60-90 分钟

### 时间分配
- 讲解：30%
- 讨论：50%
- 总结：20%
```

### 3. 有效讨论

```markdown
## 有效讨论原则

### 1. 聚焦主题
- 不要偏离主题
- 记录离题问题稍后讨论

### 2. 尊重专业
- 让专业的人发言
- 不要越俎代庖

### 3. 基于事实
- 用数据说话
- 避免主观臆断

### 4. 建设性反馈
- 提出问题同时提供建议
- 关注改进而非批评
```

### 4. 明确结论

```markdown
## 评审结论

### 必须明确
- 通过/不通过/有条件通过
- 需要修改的内容
- 责任人
- 时间节点

### 记录存档
- 评审纪要
- 决策记录
- 行动项
```

## 评审文化

### 建立评审文化

```markdown
## 评审文化

### 1. 开放心态
- 接受不同意见
- 欢迎建设性反馈

### 2. 互相尊重
- 尊重专业
- 尊重时间

### 3. 结果导向
- 评审是为了更好的结果
- 不是形式主义

### 4. 持续改进
- 不断优化评审流程
- 提高评审效率
```

## 总结

好的产品评审：

1. **目的明确**：清楚为什么要评审
2. **准备充分**：提前准备材料和问题
3. **过程高效**：控制时间、聚焦主题
4. **结论清晰**：明确下一步行动
5. **持续改进**：不断优化评审流程
6. **文化建设**：建立开放的评审文化
