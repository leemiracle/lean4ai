# 敏捷开发

## 概述

敏捷开发是一种以人为核心、迭代、循序渐进的软件开发方法，强调快速响应变化、持续交付价值。

## 敏捷宣言

### 四大价值观

```
个体和互动      高于  流程和工具
工作的软件      高于  详尽的文档
客户合作        高于  合同谈判
响应变化        高于  遵循计划
```

### 十二原则

1. 最高优先级是通过尽早和持续交付有价值的软件满足客户
2. 欣然面对需求变化，即使在开发后期
3. 经常交付可工作的软件，周期越短越好
4. 业务人员和开发人员必须天天一起工作
5. 围绕有激情的个人构建项目
6. 面对面交谈是最有效的沟通方式
7. 可工作的软件是进度的首要度量标准
8. 敏捷过程倡导可持续的开发速度
9. 持续关注技术卓越和良好设计
10. 简单一最大化不做的工作的艺术
11. 最好的架构、需求和设计出自自组织团队
12. 团队定期反思如何提高工作效率

## Scrum 框架

### 三个角色

#### 1. 产品负责人（Product Owner）

**职责**：
- 定义产品愿景
- 管理产品待办列表
- 确定需求优先级
- 验收交付成果

**能力要求**：
- 业务理解能力
- 决策能力
- 沟通能力
- 数据分析能力

#### 2. Scrum Master

**职责**：
- 保证 Scrum 流程执行
- 移除团队障碍
- 促进团队协作
- 持续改进流程

**能力要求**：
- 敏捷知识
- 引导能力
- 问题解决能力
- 领导力

#### 3. 开发团队（Development Team）

**职责**：
- 承诺完成 Sprint 目标
- 自主决定工作方式
- 交付可工作的产品增量

**特点**：
- 跨职能
- 自组织
- 3-9 人

### 三个工件

#### 1. 产品待办列表（Product Backlog）

**特点**：
- 有序的需求列表
- 持续更新
- 由 PO 负责维护

**格式**：

```markdown
## 用户故事

**作为** <角色>
**我想要** <功能>
**以便于** <价值>

**验收标准**：
- [ ] 标准1
- [ ] 标准2
- [ ] 标准3

**优先级**：P0/P1/P2
**故事点**：X
```

#### 2. Sprint 待办列表（Sprint Backlog）

**内容**：
- Sprint 目标
- 选定的用户故事
- 任务分解

#### 3. 产品增量（Product Increment）

**定义**：
- 可交付的产品版本
- 满足完成的定义（DoD）
- 经过测试验收

### 五个事件

#### 1. Sprint 规划会

**时长**：8 小时（2 周 Sprint）

**议程**：
```
Part 1：做什么？
- PO 介绍优先级最高的需求
- 团队讨论澄清
- 确定 Sprint 目标

Part 2：怎么做？
- 任务分解
- 工作量估算
- 承诺交付内容
```

**输出**：
- Sprint 目标
- Sprint 待办列表

#### 2. 每日站会（Daily Scrum）

**时长**：15 分钟

**每人回答**：
1. 昨天完成了什么？
2. 今天计划做什么？
3. 遇到什么障碍？

**目的**：
- 同步进度
- 识别障碍
- 促进协作

#### 3. Sprint 评审会

**时长**：4 小时（2 周 Sprint）

**议程**：
- PO 说明已完成的和未完成的
- 团队演示完成的功能
- PO 验收
- 讨论下一步计划

**输出**：
- 产品增量
- 更新的产品待办列表

#### 4. Sprint 回顾会

**时长**：3 小时（2 周 Sprint）

**议程**：
```
1. 数据回顾
   - 速率（Velocity）
   - 燃尽图
   - 质量指标

2. 讨论环节
   - 做得好的（Keep）
   - 需要改进的（Problem）
   - 尝试的行动（Try）

3. 行动计划
   - 选择 1-3 个改进项
   - 下个 Sprint 执行
```

#### 5. Sprint

**时长**：1-4 周（通常 2 周）

**特点**：
- 固定时长
- 目标不变
- 时间盒（Time-box）

## 看板方法（Kanban）

### 核心原则

1. **可视化工作**：看板展示所有工作项
2. **限制在制品（WIP）**：控制并行任务数量
3. **管理流动**：优化工作流程
4. **显式流程规则**：明确定义规则
5. **建立反馈循环**：定期回顾改进
6. **协作改进**：团队共同优化

### 看板设计

#### 基本看板

```
┌──────────┬──────────┬──────────┬──────────┬──────────┐
│  待办    │  进行中  │  开发    │  测试    │  完成    │
│          │   (3)    │   (2)    │   (2)    │          │
├──────────┼──────────┼──────────┼──────────┼──────────┤
│ Task 1   │ Task 3   │ Task 5   │ Task 7   │ Task 9   │
│ Task 2   │ Task 4   │ Task 6   │ Task 8   │ Task 10  │
│ Task 11  │          │          │          │ Task 12  │
└──────────┴──────────┴──────────┴──────────┴──────────┘
         ← 限制 WIP →
```

#### WIP 限制

**原则**：停止开始，开始完成

**计算方法**：
```
WIP 限制 = 团队人数 × 1.5

示例：6 人团队
WIP = 6 × 1.5 = 9
```

### 度量指标

#### 1. 周期时间（Cycle Time）

```
周期时间 = 完成时间 - 开始时间
```

#### 2. 吞吐量（Throughput）

```
吞吐量 = 单位时间完成的任务数
```

#### 3. 在制品（WIP）

```
WIP = 当前进行中的任务数
```

#### 利特尔法则

```
平均周期时间 = 平均 WIP / 平均吞吐量

启示：减少 WIP 可以缩短周期时间
```

## 用户故事

### 格式

```markdown
作为 <角色>
我想要 <功能>
以便于 <价值>
```

### 示例

```markdown
作为 一名电商用户
我想要 查看订单物流状态
以便于 知道包裹什么时候能到
```

### INVEST 原则

| 原则 | 说明 |
|------|------|
| Independent | 独立的，不依赖其他故事 |
| Negotiable | 可协商的，细节可以讨论 |
| Valuable | 有价值的，对用户有价值 |
| Estimable | 可估算的，能够评估工作量 |
| Small | 小的，一个 Sprint 能完成 |
| Testable | 可测试的，有明确的验收标准 |

### 故事拆分

**拆分方法**：

```
大故事
├── 按工作流程拆分
├── 按业务规则拆分
├── 按界面拆分
├── 按操作拆分
└── 按数据拆分
```

**示例**：

```
用户管理
├── 用户注册
├── 用户登录
├── 密码找回
├── 个人信息编辑
└── 账号注销
```

## 估算方法

### 故事点（Story Points）

**斐波那契数列**：1, 2, 3, 5, 8, 13, 21, 34...

**参考基准**：
- 1：最简单的任务
- 2：简单任务
- 3：中等难度
- 5：较复杂
- 8：复杂
- 13：非常复杂（考虑拆分）

### 规划扑克（Planning Poker）

**流程**：
1. 主持人朗读用户故事
2. 团队提问澄清
3. 每人选择估算卡片
4. 同时亮牌
5. 差异大的说明理由
6. 讨论后重新估算
7. 达成共识

### 速率（Velocity）

```
速率 = 一个 Sprint 完成的故事点总和

示例：
Sprint 1: 30 点
Sprint 2: 28 点
Sprint 3: 32 点
平均速率: 30 点
```

## 完成的定义（DoD）

### 示例 DoD

```markdown
## 完成的定义

- [ ] 代码已提交
- [ ] 代码已 Review
- [ ] 单元测试通过
- [ ] 集成测试通过
- [ ] UI/UX 验收通过
- [ ] 产品验收通过
- [ ] 文档已更新
- [ ] 无 P0/P1 Bug
```

### 不同级别的 DoD

| 级别 | 标准 |
|------|------|
| 故事级 | 功能完成、测试通过 |
| Sprint 级 | 所有故事完成、回归测试通过 |
| 发布级 | 全部测试通过、文档齐全、部署就绪 |

## 敏捷度量

### 燃尽图（Burndown Chart）

```
剩余工作量
  ↑
  │╲
  │ ╲      理想线
  │  ╲    ╱
  │   ╲  ╱  实际线
  │    ╲╱
  │     ╲
  │      ╲
  └─────────────→ 时间
   Start      End
```

### 燃起图（Burnup Chart）

```
已完成工作量
  ↑
  │         ╱ 总工作量
  │        ╱
  │       ╱
  │      ╱ 已完成
  │     ╱
  │    ╱
  │   ╱
  └─────────────→ 时间
```

### 关键指标

| 指标 | 计算方法 | 目标 |
|------|---------|------|
| 速率 | 完成故事点 | 稳定 |
| 周期时间 | 完成-开始 | 缩短 |
| 缺陷率 | Bug 数/故事点 | 降低 |
| 预测准确度 | 实际/计划 | 接近 100% |

## 敏捷转型

### 转型步骤

```
1. 培训教育
   └─ 敏捷理念和方法的培训

2. 试点项目
   └─ 选择小项目试点

3. 总结改进
   └─ 复盘试点经验

4. 推广扩展
   └─ 扩展到更多团队

5. 持续优化
   └─ 根据实际情况调整
```

### 常见问题

**问题1：站会变成汇报**

**解决**：关注同步和障碍识别，而非汇报进度

**问题2：Sprint 经常延期**

**解决**：
- 合理规划工作量
- 提高估算准确性
- 控制需求变更

**问题3：回顾会没有改进**

**解决**：
- 限制改进项数量
- 分配责任人
- 跟踪执行情况

5. 学习曲线

## 总结

敏捷开发的核心是：

1. **迭代交付**：小步快跑，持续交付
2. **响应变化**：拥抱变化，快速调整
3. **团队协作**：自组织，高效沟通
4. **持续改进**：定期反思，不断优化
5. **客户价值**：以价值为导向
