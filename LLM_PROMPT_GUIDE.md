## 一、理解LLM

### 1.1 知识本质

```
LLM知识 = 训练数据压缩 + 统计模式学习

不是：真正理解、数据库存储、有意识思考
是：  模式提取、上下文激活、概率预测
```

### 1.2 可靠性判断

| 知识类型 | 可靠性 | 示例 | 策略 |
|----------|--------|------|------|
| 语言/常识 | ★★★ | "水往低处流" | 直接信任 |
| 专业知识 | ★★☆ | 代码、数学 | 交叉验证 |
| 时效信息 | ★☆☆ | 最新新闻 | 必须验证 |
| 冷门知识 | ★☆☆ | 冷门人物 | 必须验证 |

### 1.3 能力边界

```
擅长：
  ✓ 语言任务（写作、翻译、润色）
  ✓ 代码模式（生成、调试、审查）
  ✓ 概念解释（多角度说明）
  ✓ 创意组合（头脑风暴、方案生成）

需验证：
  ⚠ 专业领域（医疗、法律、金融）
  ⚠ 精确数据（数字、日期、引用）
  ⚠ 时效信息（新闻、价格、政策）

不擅长：
  ✗ 最新信息（训练后发生的事）
  ✗ 真正推理（复杂逻辑推导）
  ✗ 实际验证（无法运行/测试）
```

### 1.4 使用原则

```
高价值场景：
  高频重复 | 创意发散 | 学习加速 | 质量提升

需谨慎场景：
  专业判断 | 精确数据 | 敏感信息

不适合场景：
  实时操作 | 物理任务 | 最终决策
```

## 二、Prompt设计

### 2.1 核心公式

```
好 Prompt = 角色 + 任务 + 背景 + 约束 + 格式

示例：
【角色】你是Python专家
【任务】写一个排序函数
【背景】处理百万级数据
【约束】O(n log n)复杂度，稳定排序
【格式】代码 + 注释 + 复杂度分析
```

### 2.2 设计原则

| 原则 | 核心 | 检查 |
|------|------|------|
| 清晰性 | 无歧义 | 每句话只有一种理解？ |
| 具体性 | 给细节 | 5W1H都清楚？ |
| 结构化 | 用格式 | 用【】标记各部分？ |
| 渐进性 | 分步做 | 复杂任务拆解了吗？ |
| 可验证 | 能检验 | 知道什么是好结果？ |

### 2.3 核心技巧

**技巧1：角色设定**

```
"你是[角色]，具有[背景]。
擅长[能力]。请[任务]。"

示例：
"你是Python专家，10年后端经验。
擅长性能优化。请审查这段代码。"
```

**技巧2：思维链（CoT）**

```
"请一步步思考：
1. 首先，[第一步]
2. 然后，[第二步]
3. 最后，[结论]"

效果：推理准确率提升30%+
```

**技巧3：Few-shot**

```
"按以下示例格式回答：
示例1：输入... 输出...
示例2：输入... 输出...

现在处理：输入... 输出："
```

**技巧4：结构化输入**

```
【任务】代码审查
【语言】Python 3.10
【关注点】性能、安全
【代码】
[粘贴]
【要求】指出问题并给出建议
```

**技巧5：负面约束**

```
【必须】
- 用Python标准库
- 处理异常

【禁止】
- 不用第三方库
- 不用全局变量
```

**技巧6：格式控制**

```
"用表格/JSON/步骤呈现"
"每项不超过10字"
"分3点说明"
```

### 2.4 复杂度策略

| 类型 | 特征 | 策略 |
|------|------|------|
| 简单 | 单一明确 | 直接问 |
| 中等 | 需上下文 | 给结构+关键问题 |
| 复杂 | 多步骤 | 最小方案→确认→迭代 |

## 三、职业实践

### 3.1 职业价值矩阵

| 职业 | LLM价值 | 核心场景 |
|------|---------|----------|
| 程序员 | ★★★★☆ | 代码生成、调试、审查 |
| 产品经理 | ★★★★☆ | PRD、用户故事、竞品分析 |
| 数据分析师 | ★★★★☆ | SQL、可视化、报告 |
| 市场/运营 | ★★★★☆ | 文案、活动、用户画像 |
| 教师/培训 | ★★★★☆ | 教案、练习、解释 |
| 研究员 | ★★★☆☆ | 文献、写作、翻译 |
| 律师/法务 | ★★☆☆☆ | 文书草稿（需验证） |
| 医生/医疗 | ★★☆☆☆ | 文献解读（需验证） |

### 3.2 职业使用要点

**程序员**

```
擅长：代码生成、调试、审查、学习新技术
注意：给完整上下文（环境、代码、错误）
模板：【语言】【代码】【错误】【期望】
```

**产品经理**

```
擅长：PRD、用户故事、竞品分析
注意：说明产品定位、目标用户、业务约束
模板：【产品】【功能】【目标】【用户】
```

**数据分析师**

```
擅长：SQL生成、可视化建议、报告框架
注意：给表结构、说明数据量
模板：【数据库】【表结构】【需求】【约束】
```

**市场/运营**

```
擅长：文案、活动方案、用户画像
注意：定调性、给样例、说禁忌
模板：【品牌】【活动】【调性】【禁忌】
```

### 3.3 通用能力

**需求驱动**

```
从问题出发，不是从工具出发

分析框架：
1. 最终目标是什么？
2. 卡在哪里？
3. LLM能帮哪部分？
```

**思维方式**

```
结构化：结论→分论点→证据
第一性原理：回归本质
系统思维：看整体、理解关联
批判性：质疑假设、多角度验证
```

**统筹能力**

```
资源整合：
  LLM → 生成、解释
  搜索 → 最新信息
  文档 → 权威来源
  人   → 经验判断
```

## 四、问题诊断

### 4.1 诊断框架

```
效果不好时，按顺序检查：

Step 1: 理解问题
  症状：答非所问
  解决：加背景说明、明确任务

Step 2: 信息不足
  症状：输出太泛
  解决：补充上下文、数据、约束

Step 3: 格式问题
  症状：格式混乱
  解决：明确格式要求、给示例

Step 4: 深度不够
  症状：回答太浅
  解决：加角色设定、思维链

Step 5: 边界问题
  症状：编造事实
  解决：验证结果、换方法
```

### 4.2 编程类问题

| 问题 | 原因 | 解决 |
|------|------|------|
| 代码错误 | 描述不具体 | 明确技术栈、边界情况 |
| 理解偏差 | 有歧义 | 拆解需求、给上下文 |
| 优化方向错 | 目标不明 | 明确优化目标、量化指标 |
| 调试无效 | 信息不全 | 给完整代码、错误、环境 |

### 4.3 数学类问题

| 问题 | 原因 | 解决 |
|------|------|------|
| 概念不清 | 没定水平 | 指定受众、要求举例 |
| 跳步太多 | 没要求 | 要求一步步写 |
| 证明混乱 | 没结构 | 指定证明方法、要求结构 |
| 不会建模 | 没指导 | 要求建模步骤 |

### 4.4 写作类问题

| 问题 | 原因 | 解决 |
|------|------|------|
| 风格不符 | 没说风格 | 给样例、描述风格 |
| 内容太泛 | 没约束 | 要求具体、要求数据 |
| 结构混乱 | 没指定 | 指定结构、要求大纲 |

## 五、模板库

### 5.1 万能模板

```
【我要】[一句话目标]
【背景】[当前情况]
【约束】[限制条件]
【格式】[期望格式]
```

### 5.2 代码模板

```
【任务】写代码/调试/审查/重构
【技术栈】[语言+框架+版本]
【代码】
```
[粘贴]
```
【错误】[如有]
【期望】[具体要求]
```

### 5.3 学习模板

```
【主题】[...]
【基础】[当前水平]
【时间】[可用时间]
【目标】[用来做什么]

请：
1. 列核心内容（按优先排序）
2. 给第一周计划
3. 推荐资源
```

### 5.4 分析模板

```
【问题】[现象]
【背景】[环境/上下文]
【已尝试】[做过什么]
【期望】[想要什么结果]

请：
1. 分析可能原因（按概率排序）
2. 给排查步骤
3. 提供解决方案
```

### 5.5 迭代模板

```
继续上一任务。

【评价】
- 好的部分：...
- 需改进：...

【调整】
- 方向：...
- 补充：...

请基于反馈改进。
```

### 5.6 决策模板

```
【决策】在[A]和[B]之间选择
【目标】[...]
【约束】[...]

请用表格对比：
| 维度 | A | B |
|------|---|---|
| 优点 |   |   |
| 缺点 |   |   |
| 风险 |   |   |

给出建议+理由
```
